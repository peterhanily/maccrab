# MacCrab v1.7.6 — Storage init crash-loop hot-fix

**TL;DR:** v1.7.5 daemons could enter a crash-loop in storage init on
existing installs because of a `SchemaMigrator` bug where two co-resident
stores (EventStore + AlertStore sharing `events.db`) silently skipped
each other's migrations. v1.7.6 fixes the migrator and adds visible
errors + auto-recovery as defense in depth. **No data loss** — the
fix repairs existing databases in-place. **No reboot or extension
re-approval** required to install.

## What was broken

On v1.7.5, AlertStore prepare-statement init failed with
`table alerts has no column named llm_investigation_json`. The daemon
exited in ~127 ms, launchd respawned every 10 s, and the dashboard
reported "Detection engine appears silent".

The error was masked by `os.log`'s default `<private>` redaction —
console showed only `Failed to initialize storage: <private>` with
no actionable signal. SIP-protected machines can't enable
`private_data:on` log mode either, so the redaction was a hard
diagnostic dead end.

## What v1.7.6 changes

### The actual fix

`SchemaMigrator.run()` now always re-applies this store's migrations
idempotently, instead of skipping based on a counter that another
co-resident store already moved. `apply()`'s already-applied detector
covers both `duplicate column name` (for ADD COLUMN) and `already exists`
(for CREATE TABLE / CREATE INDEX), so re-runs are no-ops when the
schema is already in place.

### Diagnostic surface

- Storage-init errors logged with `.public` privacy — actual error
  text reaches the unified log without redaction.
- `sysext_started.json` written before any storage init, distinguishing
  "launched but crashed in init" from "never launched" in the dashboard.
- `last_crash.json` records init failures + recovery attempts so a
  future "Recover" button (deferred to v1.7.7) can read structured
  diagnostic context.

### Recovery escape hatches

- Daemon auto-recovers from storage-init exceptions by backing up
  `events.db{,-wal,-shm}` to `events.db.corrupt-<unix-ts>` and
  retrying init from scratch. On second consecutive failure, exits
  cleanly to halt the launchd respawn loop.
- `maccrabctl repair --fix-storage` does the same backup operation
  manually (with a `PRAGMA integrity_check` gate to skip destructive
  backup if the DB is actually healthy).

For the v1.7.5 bug specifically, neither escape hatch fires — the
SchemaMigrator fix runs first and the missing column gets added
in-place on the existing DB. Field-verified: 448 events + 182 alerts
preserved on a previously-crashing install.

## How to install

### If you're already on MacCrab via Sparkle auto-update

Open MacCrab.app. The update banner appears within a few minutes of
the appcast publish. Click "Install Update". Sparkle replaces the app
bundle, sysextd terminates the crash-looping v1.7.5 daemon, and the
new v1.7.6 daemon starts. Banner clears within ~30 s.

### If you're on Homebrew

```bash
brew upgrade --cask maccrab
```

### Fresh install

Download the DMG from
[github.com/peterhanily/maccrab/releases/v1.7.6](https://github.com/peterhanily/maccrab/releases/tag/v1.7.6),
drag MacCrab.app to Applications, launch, approve the system extension
in System Settings.

## What you don't need to do

- **No reboot.** The daemon process restart happens automatically
  when sysextd swaps the binary.
- **No extension re-approval.** The new sysext binary inherits the
  approval granted to the same team-id + bundle-id combo.
- **No DB migration / repair.** The SchemaMigrator fix runs on first
  launch and applies the missing column to your existing alerts
  table. All event and alert history is preserved.

## Known limitations

- If your install has 8+ zombie sysexts queued for uninstall on
  reboot from prior upgrades, sysextd may need a reboot to fully
  swap in the new binary. This is a separate macOS sysextd bug
  (covered by the v1.7.5 zombie-sysext banner) and would have
  required a reboot regardless of this fix. Most installs won't
  have this backlog.
