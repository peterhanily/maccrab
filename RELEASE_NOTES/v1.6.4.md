## MacCrab v1.6.4 — FP reduction in alerts and campaigns

Field-driven hotfix. User dogfooding after v1.6.2 surfaced four specific
false-positive patterns firing repeatedly on a developer workstation; all
four are now fixed at the structural level, not just the rule level.

### What was firing

1. **`MDM Enrollment Status Check via profiles`** — Google's Updater
   legitimately runs `profiles status -type enrollment` as part of
   deciding whether a Chrome / Drive update should defer to enterprise
   policy. MacCrab was flagging that as reconnaissance.

2. **`Coordinated Attack from single process` on `/usr/bin/csrutil`** —
   user running `csrutil disable` triggered two rules at once
   (`csrutil_status_check` tagged `discovery`, and
   `sip_check_before_tampering` tagged `defense_evasion`). The campaign
   detector saw "2 tactics from 1 process" and fired a coordinated-attack
   campaign.

3. **`Coordinated Attack from single process` on Sparkle's Autoupdate** —
   MacCrab's own Sparkle updater binary (running during a MacCrab
   upgrade) was being treated as an attacker. Legitimate updaters touch
   defense_evasion (code-signing checks) + persistence (plist writes) as
   a matter of course.

4. **`Kill Chain: Multi-Stage Attack` with 3 tactics** — threshold of 3
   tactics in 10 min was trivially hit by routine admin work (`find`
   for discovery, `curl` for exfil, `csrutil status` for defense
   evasion).

### Fixes

- **Coordinated-attack: require ≥2 distinct rule IDs** before cross-
  tactic correlation. A single rule carrying both `attack.discovery` and
  `attack.defense_evasion` tags no longer counts as "two tactics from one
  process" — that's one alert with rich tagging, not an attack pattern.
  Both PID-based and path-based branches of the detector gate on this.

- **Auto-updater allow-list** in the campaign detector. New
  `isKnownBenignProcess` covers Sparkle Autoupdate (any path containing
  `Sparkle.framework/` or `.sparkle-project.Sparkle/Installation/`),
  Google's Updater / SoftwareUpdate / Keystone, Microsoft AutoUpdate,
  macOS `softwareupdated`, Homebrew. Used by both the coordinated-
  attack and kill-chain detectors.

- **Kill-chain threshold raised 3 → 4.** Four distinct tactics is a
  materially stronger signal than three while still matching real
  multi-stage attack shapes (discovery → credential_access → persistence
  → exfiltration is the textbook four-step pattern).

- **`csrutil_status_check.yml` scoped to status queries only.** Now
  matches `csrutil status`, `csrutil netboot`, `csrutil
  authenticated-root status`, `csrutil --help`. `csrutil disable`
  variants fall through exclusively to
  `sip_check_before_tampering.yml`, so one invocation = one alert =
  one tactic.

- **`mdm_enrollment_check.yml` excludes auto-updater processes.** Filters
  on GoogleUpdater, GoogleSoftwareUpdate, Sparkle, Autoupdate, generic
  `updater` / `Updater` parents, softwareupdate, Brave Browser Helper,
  Microsoft AutoUpdate, Mozilla Maintenance Service, plus a
  `ParentImage|endswith: '/launcher'` exit for GoogleUpdater's subprocess
  chain.

### Regression lock

New `CampaignDetectorFPRegressionTests` suite (5 tests):
- Single-alert-two-tactics stays silent
- Two distinct alerts on same process still fires coordinated-attack (counter-test)
- Sparkle Autoupdate path allowlisted
- GoogleUpdater path allowlisted
- Three tactics across three alerts stays silent (new kill-chain threshold)

The counter-test is important — a future over-broad gate that turned
coordinated-attack entirely off would catch here.
