# MacCrab v1.12.3 — v1.12 rollup release

v1.12.3 is the consolidated v1.12 line — every feature, hardening,
and fix from v1.12.0 / v1.12.1 / v1.12.2 packaged together with
comprehensive release notes. **No new code vs v1.12.2** — same
binary, same SHA-class, just a clean canonical install target with
a single set of notes for everyone upgrading from v1.11.x or
earlier.

If you are on **v1.11.x or earlier**, this is the recommended
release to upgrade to. Sparkle in-place upgrade works directly:
open MacCrab.app, choose **Check for Updates…**, click **Install
Update**, relaunch when prompted.

### Manual upgrade for v1.12.0 / v1.12.1 installs

Your installed MacCrab.app carries broken nested Sparkle helpers
from a build-pipeline bug fixed in v1.12.2. Sparkle cannot install
any newer version — you will hit "An error occurred while running
the updater". One-time manual upgrade:

**Option A — Homebrew (recommended)**

```
brew update
brew upgrade --cask maccrab
```

**Option B — Manual DMG**

1. Download
   [MacCrab-v1.12.3.dmg](https://github.com/peterhanily/maccrab/releases/download/v1.12.3/MacCrab-v1.12.3.dmg).
2. Quit MacCrab from the menu bar (or right-click in Activity
   Monitor → Quit).
3. Open the DMG and drag `MacCrab.app` into `/Applications`,
   replacing the existing copy.
4. Launch the new `MacCrab.app`. Approve the SystemExtension
   prompt if it appears (usually doesn't on in-place reinstalls).

Once you're on v1.12.3, future Sparkle upgrades work normally.

### Detection — supply-chain coverage wave

- **Mini Shai-Hulud** — same family, narrower attack window.
- **Lightning PyPI** — credential-harvest variant on PyPI.
- **TanStack CVE-2026-45321** — npm package compromise via
  publish-token theft.

Rule corpus growth (vs v1.11.1):

| Tier                         | v1.11.1 | v1.12.3 | Δ    |
|------------------------------|---------|---------|------|
| Single-event Sigma           | 388     | 424     | +36  |
| Sequence (temporal)          | 33      | 39      | +6   |
| Graph (multi-entity)         | 0       | 6       | NEW  |
| **Total compiled rules**     | **421** | **469** | **+48** |

The canonical worm self-propagation signature (credential read +
GitHub publish-endpoint egress, both from the same `node` lineage

graph rule, with O(1) tactic-indexed dispatch.

### Detection — intent posterior

A Bayesian belief network now maintains a per-process-tree
posterior over attacker goals (benign / credentialHarvest /
exfiltration / persistence / destructive / reconnaissance /
lateralMovement / unknown). Wave-5 actors:

- **`IntentClassifier`** — LLM-backed structured intent over a
  `BehaviorBrief` (package name + installer lineage + credential
  reads + network egress + content anomaly flags + AI-agent
  attribution). Routes through `LLMService` (Ollama / Claude /
  OpenAI / Mistral / Gemini) with sanitization + caching + circuit
  breaker. Per the NDSS 2025 "Mind the Gap" paper, Llama 3.3 70B
  local hits F1 0.77 / GPT-4.1 cloud hits F1 0.99 on this task.
- **Heuristic fallback** — when no LLM is configured, a
  deterministic rule-based classifier alone catches the worm
  self-propagation shape (credentialRead + publish-endpoint egress
  = `lateralMovement` verdict). No cloud dependency required.
- **`PromptIntentBridge`** — correlates an AI agent's recent
  context-reads (CLAUDE.md, README, Cursor rules) against the
  actually-installed package name to classify installs as
  `.userInitiated` / `.autonomous` / `.slopsquat` /
  `.vagueDestructive` / `.injectionContext`.
- **`CounterfactualReasoner`, `StylometricFingerprinter`,
  `HoneyPromptManager`, `BayesianIntentEngine`,
  `NextTechniquePredictor`, `AttestationEnricher`,
  `PackageContentAnalyzer`, `PackageMetadataAnalyzer`,
  `TyposquatDatabase`** — supporting actors. `TyposquatDatabase`
  ships ~200 npm + ~200 PyPI top-package corpora with
  Damerau-Levenshtein + Unicode confusable folding.

### Performance — daemon cold start: 114 s → 118 ms

Two pre-ship audit cycles (round 9: 6 parallel domains; round 10:
5 parallel domains) found seven sources of synchronous main-thread
work at daemon boot, totalling ~114 seconds on cold disk. All
deferred behind `Task.detached` after the first heartbeat:

- 7 GB `tracegraph.db` quick_check (skipped via
  `SchemaMigrator.skipQuickCheck:`).
- `SelfDefense` self-SHA-256 (now lazy in `start()`).
- `BaselineEngine.load`.
- `ThreatIntelFeed`.
- `BundledThreatIntel`.
- `ESClientMonitor`.
- Six polled monitors (USB / clipboard / browser-extension /
  rootkit / EDR / TEMPEST).

`boot_phase` breadcrumbs are stamped at 14 milestones in
`DaemonSetup.swift` so future regressions surface immediately.

### Dashboard — in-app Sigma YAML editor

The Detection workspace now ships a read-only viewer **and** a
TextEditor save flow that pipes the edited YAML through the
bundled `compile_rules.py` and writes the resulting JSON to
`/Library/Application Support/MacCrab/user_rules/<uuid>.json`. The
daemon picks up the change via a `.reload_tick` mtime watcher; no
restart needed. Disable / re-enable rule actions write the same
overlay with `enabled=false`. First write fires a single
AppleScript admin prompt to create the override directory
`root:admin 0775`; subsequent edits don't prompt.

### Privacy + LLM hardening

- **Webhook + syslog default hostname** changed to `maccrab-host`
  (not the machine hostname). Override via
  `MACCRAB_WEBHOOK_HOSTNAME` / `MACCRAB_SYSLOG_HOSTNAME`.
- **MCP payloads** — `get_alerts`, `get_alert_detail`, and
  `scan_text` route through `LLMSanitizer.sanitize()` before
  returning to the agent, matching the redaction guarantees of the
  cloud-LLM backends.
- **OpenAI host allowlist** — `hasSuffix("api.openai.com")` (which
  let `evilapi.openai.com` through) replaced with a dot-anchored
  exact-host Set plus `.openai.azure.com` dot-suffix with a
  `count > suffix.count` guard.
- **Gemini** — model-name allowlist is now a `[a-z0-9._-]+` regex
  capped at 64 chars.
- **Ollama** — plaintext-remote detection now explicitly nil-checks
  the host.

### Self-defense — Sparkle/cask upgrade FP fix

The sysext's `SelfDefense` actor watches `/Library/Application
Support/MacCrab/compiled_rules` and the running binary for tamper
events. Before v1.12.1, every Sparkle/cask in-place upgrade tripped
**two false positives**:

1. `MacCrab Tamper Detection: File Deleted` on `compiled_rules` —
   because the dashboard's elevated `rm -rf + cp -R` rule-sync
   touched the watched directory.
2. `MacCrab Tamper Detection: Binary Modified` on `maccrabd` — because
   the .app replacement legitimately swapped the binary on disk.

Both fixed via two independent evidence channels:

- **Sentinel file** at `/Library/Application
  Support/MacCrab/.maccrab_self_update_in_progress` dropped by
  `RuleBundleInstaller.copyRulesWithElevation` at the start of its
  elevated session (root:admin 0644) and removed at the end.
  `SelfDefense` checks for it (mtime ≤ 90 s) before firing
  delete/rename/write alerts on data-dir paths.
- **Signer-validity check** via `codesign -dvv --verify --` on the
  hash-mismatched binary — if it still verifies under MacCrab's
  Developer ID (Team `79S425CW99`), re-baseline silently. A real
  tamper can't forge our signature without the private key.

Trust posture is unchanged for actual tampering: real `rm -rf` from
a non-MacCrab process without the sentinel, replacement with
unsigned / ad-hoc / other-team-signed binaries, `pkill maccrab`,
`launchctl bootout`, `systemextensionsctl uninstall`, debugger
attach, signal-based termination, and plist removal all still
surface as critical alerts.

### Build pipeline — Sparkle install fix (v1.12.2 → v1.12.3)

v1.12.0 RC28 added `--deep --entitlements <APP_ENT> --force` to the
outer `.app` codesign pass to seal `Resources/*` into the bundle
signature. That assumption was wrong — codesign's bundle-mode
signing seals `Resources/*` via `_CodeSignature/CodeResources`
regardless of `--deep`. The `--deep` flag is for recursing into
nested SIGNED code (frameworks, XPC services), and on the outer
sign it re-signed every nested Mach-O — including Sparkle's
helpers — propagating MacCrab's main-app entitlements onto each.

macOS refuses to launch a Sparkle XPC helper carrying
`com.apple.developer.system-extension.install`, which surfaced as
"An error occurred while running the updater" on every Sparkle
in-place upgrade against v1.12.0 / v1.12.1. **Fix shipped in
v1.12.2**: drop `--deep` from the outer sign. Sparkle's helpers
keep their own (entitlement-free) signatures, the outer .app keeps
its main-app entitlements, Resources still get sealed via
CodeResources. v1.12.3 carries the same fix.

### Other notable hardening

- `RuleBundleInstaller` manifest-content sync check (RC18) —
  RC11-RC17 silently failed to deliver fixes because the bundled
  `.bundle_version` matched the installed one; now also compares
  `manifest.json` byte-for-byte.
- AppleScript `do shell script` parser fix (RC12) — replaced
  `find -exec ... \;` (which AppleScript misparsed as escape,
  failing with `-2741`) with `chmod -R u=rwX,go=rX`.
- Disable-rule UUID lookup (RC30) — runtime fallback scan of
  `Resources/compiled_rules/` keyed by each JSON's internal `id`,
  since bundled JSONs are slug-named not UUID-named.
- 5 task_for_pid / mach_port rules got
  `filter_text_tools` / `filter_shells` / `filter_apple_paths` /
  `filter_dev_ai_tools` FP suppression for `/bin/ls`, `grep`,
  Claude Code, and similar.
- `try!` force-unwraps removed from EventStore / AlertStore
  recovery paths.
- Pipe-deadlock fix in the YAML editor save (subprocess could fill
  stderr's 64 KB pipe before `waitUntilExit()` returned).
- 2 new persistence rules (`launchagent_with_distant_future_trigger`,
  `maintainer_publish_hour_anomaly`) got proper filter blocks for
  Apple/brew/MDM writers and CI runner parents.

## Compatibility note for v1.12.0 / v1.12.1 archive consumers

The v1.12.0 and v1.12.1 GitHub releases remain available for audit
and forensics purposes but are marked **pre-release** and yanked
from the Sparkle appcast — please use v1.12.2 or v1.12.3 for any
new install. v1.12.0 and v1.12.1 have functioning detection but
broken nested-Sparkle entitlements, so they cannot auto-update to
anything newer.
