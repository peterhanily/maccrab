## MacCrab v1.7.5 — Heartbeat split + repair tooling

Architectural improvements driven by a real-world v1.7.3 incident
where the dashboard correctly reported "Detection engine appears
silent" but the user had no easy way to diagnose why. Three
additions: split heartbeat for liveness vs rich payload, a
`maccrabctl repair` self-diagnostic command, and a dashboard
zombie-sysext banner that points users at the right fix when
macOS sysextd state is the actual problem.

### Heartbeat split — liveness decoupled from rich payload

Pre-v1.7.5 the daemon wrote one `heartbeat.json` from inside an
async Task that included an EventStore query, a CollectorRegistry
snapshot, and four downstream snapshot writes. Any stall in any
of those = no heartbeat write = dashboard banner.

v1.7.5 adds a **synchronous** liveness timer that runs on the
dispatch queue and writes a minimal `heartbeat.json` containing
only `written_at_unix`, `uptime_seconds`, `sysext_has_fda`, plus
the event/alert counters. No actor hops. No queries. No snapshot
writes. Cannot deadlock because there's nothing async to deadlock
on.

The rich payload (per-event-category counts, collector health,
drop counter) now lives in a separate `heartbeat_rich.json` written
by the async Task. The dashboard's `HeartbeatSnapshot.isStale`
gating reads only the liveness file; rich fields are merged from
the rich file when present. Pre-v1.7.5 daemons that wrote
everything inline still decode correctly — backward-compatible.

### `maccrabctl repair` subcommand

```
maccrabctl repair               # diagnose + auto-fix safe issues
maccrabctl repair --dry-run     # diagnostics only, no actions
```

Six-phase output:
1. **Daemon process** — `pgrep -lf com.maccrab.agent`. Reports PID
   if alive; explains the "silent banner" if dead.
2. **Heartbeat liveness** — stat `heartbeat.json`, age vs the 120s
   threshold, color-coded.
3. **System extension state** — parses `systemextensionsctl list`,
   reports the activated version + count of zombies pending
   uninstall on reboot.
4. **Orphaned writeSnapshot temp files** — finds any `.tmp` files
   in the support dir, removes them (or shows them under
   `--dry-run`).
5. **Daemon reload** — sends `SIGHUP` if the daemon is alive (forces
   rule + config reload without restart). Skipped if dead.
6. **Recommendations** — clear next steps for issues that need
   operator action: reboot, re-approve extension, grant FDA.

### Zombie-sysext banner in the dashboard

`AppState.refreshZombieSysextCount()` runs `systemextensionsctl list`
on each refresh tick (~10 s) and counts MacCrab entries in
`[terminated waiting to uninstall on reboot]` state. When ≥3, a
banner appears at the top of MainView: "N prior MacCrab versions
queued for uninstall — reboot to clear them and ensure the active
version starts cleanly." Distinguishes "the active version is
truly hung" from "macOS sysextd is in a transitional state and a
reboot will fix it" — different messaging, different fix.

### Why the v1.7.4 hotfix wasn't enough

v1.7.4 cured the leak (per-writer in-flight guards) but the
heartbeat write was still gated on async work succeeding. If
ANY future change adds a slow path between the heartbeat and the
file write (a new actor query, a slow disk, a stuck dispatch queue),
the dashboard would silently flip to "engine silent" again. v1.7.5
moves the liveness write onto the dispatch thread itself, where
nothing async can interfere. This isn't paranoid — the v1.7.0
through v1.7.4 history shows we hit this class of bug repeatedly.

### Tests

926 in 188 suites pass (same as v1.7.4). v1.7.5 changes are mostly
file-format split + new CLI command + new AppState property; the
existing test suite covers the underlying snapshot writers.

### Notes for operators

- **First action when "engine silent" appears**: run
  `maccrabctl repair`. It tells you immediately whether the issue
  is "process dead, reboot needed" or "process alive, something
  upstream is stuck."
- The repair command is safe to run with the daemon running —
  it only sends SIGHUP (config reload) and removes orphan
  temp files. Nothing destructive.
- The zombie-sysext banner appears proactively — operators don't
  need to know to look for it.
- Pre-v1.7.5 dashboards continue to work against a v1.7.5 daemon
  (they read the inline fields from heartbeat.json that aren't
  present any more, and the rich fields stay nil). They just
  don't get the rich payload. Upgrade the dashboard to match.

### Roadmap

- v1.7.6 if needed for further hotfixes; otherwise back to feature
  work (carry-over from the v1.7.x backlog: per-call LLM telemetry,
  daemon resource usage in heartbeat, more MCP spawn shapes).
- Long-term: `maccrabctl repair` could grow more auto-fix actions
  (e.g., trigger sysextd to retry activation, clear corrupted
  preferences). v1.7.5 ships a minimal-footprint version that
  only touches files MacCrab itself owns.
