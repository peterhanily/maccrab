# MacCrab v1.11.1 — First-launch beachball hotfix

v1.11.1 is a focused hotfix on top of v1.11.0. After v1.11.0 shipped,
field-testing surfaced a pre-first-frame beachball on dashboard
launch — multi-second on first install / Sparkle update, sub-second
but still visible on warm launches. v1.11.1 fixes it by deferring
three classes of launch-time work off the main thread.

Sparkle in-place upgrade from v1.11.0 lands the fix without any user
action other than relaunching MacCrab.app when prompted.

## What was slow

Three things ran on the main thread BEFORE SwiftUI rendered the
first frame:

1. **`RuleBundleInstaller.syncIfNeeded()`** in `MacCrabApp.init()`.
   On first install / Sparkle update the sync verifies the bundled
   rule manifest (SHA-256 each rule file) and then copies all 427
   compiled JSON files to `/Library/Application Support/MacCrab/
   compiled_rules/`. Cold-cache cost: 200 ms – 2 s depending on disk.
   Pre-fix the dashboard window didn't paint until this was done.
2. **`V2LiveDataProvider()` four sequential SQLite opens** —
   alerts.db, events.db, campaigns.db, tracegraph.db. All opened
   serially on the @MainActor V2LiveDataProvider class. Each open
   runs schema migrations (the v1.11.0 v4 ALTER on alerts.db
   especially), WAL setup, pragma application. Cold-cache cost:
   50 – 300 ms total.
3. **`AppState.loadSuppressPatterns()` + `loadSuppressedIDs()`** —
   two small UI-state file reads on @MainActor at AppState init
   time. Cost is small (~10-30 ms) but additive.

Combined worst case: a multi-second beachball on first install with
the Sparkle Update Ready notification in the foreground.

## The fix

1. **`RuleBundleInstaller.syncIfNeeded()` deferred** to a
   `Task.detached(priority: .utility)` from the WindowGroup's
   `.onAppear`. The window paints first; the sync runs in the
   background; the daemon SIGHUP at sync completion notifies the
   sysext. The dashboard's mtime-gated `AppState.loadRules` refresh
   (5 s tick) picks up the new corpus on its next tick.
2. **`V2LiveDataProvider()` SQLite opens parallelized + detached.**
   All four opens now run in parallel via `async let` +
   `Task.detached`, and the actor types (Sendable) are returned to
   the @MainActor class. Cold open drops from sum to ~max + actor
   hop. Schema-v4 ALTER on alerts.db still runs but on a background
   thread.
3. **`AppState.loadSuppress*` deferred** to a
   `Task { @MainActor in ... }` so AppState's init returns
   immediately. The Published `suppressionPatterns` /
   `suppressedIDs` default to empty for the brief window before
   the Task completes; benign — by the time the user can interact
   (>50 ms after first frame), the data has populated.

## Verifying after upgrade

After Sparkle relaunches MacCrab.app, first launch should feel
instant. The dashboard window should paint in well under 200 ms
even on cold cache. Use Activity Monitor's "View → Sample Process"
on `MacCrab` during a fresh launch to confirm — pre-fix you'd see
the main thread stuck inside `RuleBundleInstaller.verifyManifest`
for hundreds of milliseconds; post-fix the main thread hits
`runloop` waiting for AppKit messages immediately after init.

## Compatibility

- **No reboot or extension re-approval** required. The system
  extension bundle ID is unchanged; sysext activation re-uses the
  existing approval.
- **No data loss.** No schema changes — alerts.db still at v4 from
  v1.11.0; events.db / campaigns.db / tracegraph.db unchanged.
- **macOS 13 minimum** (unchanged).

## Tests

1404 / 1404 unit tests pass. The hotfix is a pure deferral — no
behavior changes — so existing tests cover it. Future v1.11.x will
add an integration test exercising the launch path with profiled
main-thread time-to-first-frame.
