## MacCrab v1.6.13 — Size-cap hardening

v1.6.12 wired the size-cap enforcer end-to-end but its failure modes
weren't paranoid enough to ship at scale. v1.6.13 hardens each one:

### Seven hardenings

1. **Pre-flight disk-space check.** VACUUM needs scratch space
   ~= current DB size. Before running it, the enforcer queries the
   volume's `volumeAvailableCapacityForImportantUsage` and requires
   ≥ 1.3× the current DB size. If the volume is tight, the prune
   still happens (pages freed internally) but VACUUM is skipped
   with an explicit warning. The next hourly tick tries again once
   disk frees up.

2. **Single VACUUM per sweep.** v1.6.12 called VACUUM inside the
   prune loop — up to 8 full-file rewrites on a big DB. v1.6.13
   prunes everything first, then VACUUMs once at the end.

3. **50% row-deletion cap per sweep.** A broken overage estimate
   in v1.6.12 could in principle delete way more rows than needed.
   v1.6.13 caps each sweep at 50% of total rows. If the DB is still
   over cap after that, the next hourly tick does another 50%.
   Converges to target across 3–5 hours on a heavily-over-cap DB
   instead of wiping in one pass.

4. **WAL checkpoint pair.** Previous v1.6.12 VACUUM in WAL mode
   could leave pages in the `-wal` file, so the main `.db` file
   (what the Settings UI measures as "Current size") wouldn't
   shrink visibly. v1.6.13 runs `wal_checkpoint(PASSIVE)` before
   VACUUM and again after — PASSIVE never blocks writers,
   escalating to RESTART (briefly parks new writers, doesn't
   require zero readers) only if PASSIVE couldn't drain the WAL.

5. **Reentrancy guard.** Hourly timer + any on-demand entry point
   (prune-now button, CLI) can collide. New `EventStore.
   beginSizeCapPrune()` returns false when another sweep is
   already active — second caller no-ops and logs "another sweep
   already active, skipping". Enforcer uses `defer` to release,
   so the guard opens even on error or panic.

6. **15-minute first-sweep delay.** Raised from 10 min to 15 min
   so the first startup window lets collectors + inventory scans
   + baseline hydration settle before the enforcer competes for
   IO. Hourly cadence unchanged.

7. **Startup confirmation log.** On daemon boot the enforcer
   emits a single line:

   ```
   Size-cap timer armed: cap=500 MB, target=400 MB, currently 19406 MB (OVER CAP — first sweep in 15 min, hourly thereafter)
   ```

   Operators can grep this to verify v1.6.13 is actually loaded
   and what state the DB is in.

### Observability: one grep, complete picture

Every sweep emits structured logs:

```
Size-cap enforcer armed: DB 19406 MB exceeds cap 500 MB; target 400 MB.
Size-cap phase 1: pruned 3500000 rows (estimated 3500000, cap 3500000); logical size now 19406 MB
Size-cap phase 2: skipping VACUUM — need 25228 MB free, have 8200 MB. File size unchanged; will retry next tick.
Size-cap sweep complete: 19406 MB → 19406 MB (rows pruned: 3500000, vacuum: skipped)
```

The above example is the low-disk path: rows pruned internally,
file can't shrink this tick, will try again when disk frees. The
happy path shows `vacuum: success` and a shrunken final number.

### What's unchanged

- Detection engine, rules, AI Suite, event loop — identical to v1.6.12
- Alerts, campaigns, suppressions — unaffected
- Only the events DB is pruned (the same retention behavior as before)

### Tests

**776 tests pass (up from 769).** +7 new tests covering the
hardening:

- Reentrancy guard returns false while another sweep holds it
- `walCheckpoint()` safe on empty store
- `walCheckpoint()` drains WAL after inserts and preserves row count
- VACUUM followed by checkpoint leaves store queryable + writable
- Guard + defer release cycle works across iterations
- Concurrent acquire during pruneOldest honors guard
- `pruneOldest` continues to work when guard is held (independent API)

### Upgrading from v1.6.12

No-op upgrade. Daemon picks up the new timer at next restart /
sysext reactivation. On-disk layout unchanged. No data migration.
