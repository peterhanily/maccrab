# MacCrab v1.10.0 — Workspace Dashboard + Visual TraceGraph

The dashboard-rewrite release. v1.10.0 replaces the v1 SwiftUI
dashboard with a workspace-based design (Overview, Alerts,
Investigation, Intelligence, Protection, System), ships a real visual
TraceGraph, and bundles `maccrabctl` and `maccrab-mcp` inside
`MacCrab.app` so Sparkle in-place updates keep the terminal CLIs
current.

It also folds in a 280+ finding pre-ship audit-fix pass spanning
security, performance, scalability, localization, accessibility, and
daemon correctness — driven by structured fix-waves rather than a
manual triage list.

## What's new

### V2 dashboard

Six workspaces, each with its own tab strip:

- **Overview** — KPIs (open alerts 24h, active campaigns by severity,
  events/sec, 8-bucket events sparkline), severity-bucketed alert
  histogram, top-N rule firings.
- **Alerts** — multi-select triage, bulk-suppress, severity filter,
  rule fingerprint clustering, full LLM investigation render with
  D3FEND chip strip, "What to do" hint, analyst pill row, MITRE
  tactic + technique chips, "Show in Agent Traces" cross-link.
- **Investigation** — TraceGraph (visual), Agent Traces, AI
  Analysis. The TraceGraph view draws the trace anchor at centre
  with a flame icon, members radiating on a ring (two concentric
  rings if >12 members), edges drawn from the anchor, hover-to-
  highlight, and a Graph/List toggle for accessibility.
- **Intelligence** — Threat Intel feed refresh, custom feeds +
  IOCs entry-point card with reveal-folder, LLM provider key
  management with reveal-folder + Settings deep-link, package
  freshness inspector, AI analysis history.
- **Protection** — Prevention layer status, AI Guard, browser
  extensions, MCP server inventory.
- **System** — Permissions (TCC), ES Health, Documentation index.

Suppression management is now first-class: a Suppressions viewer
shows every active entry, lift-suppression is one click, and
campaigns suppress all their contributing alerts atomically.

### Visual TraceGraph

The TraceGraph workspace now renders an actual hub-and-spoke graph
instead of a CLI-hint placeholder. The trace anchor sits at centre
with a flame icon, member entities radiate around it on concentric
rings (the layout splits into two rings past 12 members to keep
nodes legible), edges are drawn from the anchor to every non-anchor,
and hover dims everything but the highlighted node. A Graph/List
toggle in the header swaps to a screen-reader-friendly list
representation of the same data.

Each entity carries an icon by type — process, file, network,
ai_agent, persistence, tcc — making the kill-chain shape readable at
a glance.

### Causal trace store

`tracegraph.db` is new on first launch of v1.10. Stores entities,
edges, and trace memberships with the same column-level AES-GCM
encryption, chmod-0o660 file mode, and auto-vacuum=incremental as
`events.db` / `alerts.db` / `traces.db`. Existing v1.9 stores are
unaffected.

### MCP trace tools

Five new tools exposed by `maccrab-mcp`:

- `get_traces` — list recent causal traces from TraceGraph
- `get_trace_detail` — full trace with anchor + members + hash chain
- `hunt_trace` — substring search across traces
- `verify_bundle` — verify a `.maccrabtrace` bundle (schema, Merkle, signature)
- `trace_from_event` — pivot from an event ID to its containing trace

That brings the MCP tool count to 17 (was 12 in v1.9.0).

### Bundled CLI

`maccrabctl` and `maccrab-mcp` now ship inside
`MacCrab.app/Contents/Resources/bin/`. Both `Casks/maccrab.rb`
postflight and `scripts/install.sh` symlink the brew-bin entries
into the in-app paths so a Sparkle in-place update of MacCrab.app
automatically refreshes the terminal CLI.

This fixes the pre-1.10 path-shadow bug: the cask's `binary` stanza
created a symlink at `$HOMEBREW_PREFIX/bin/maccrabctl` pointing into
the caskroom's version-pinned copy, so after every Sparkle update
the terminal CLI kept resolving to the previous version's binary —
and any subcommand added since (e.g. `intel refresh`, `trace ...`,
`unsuppress --id`) failed with `Unknown command`.

`install.sh` also detects Apple Silicon and auto-replaces stale
brew-cask symlinks at `/opt/homebrew/bin/{maccrabctl,maccrab-mcp}`
on manual install.

## Upgrading

### From v1.9.x

`brew upgrade maccrab` or accept the Sparkle in-app prompt — both
paths preserve `/Library/Application Support/MacCrab/` (events.db,
alerts.db, campaigns.db, traces.db, suppressions.json, daemon
config, LLM keys). `tracegraph.db` is created on first launch and
empty until the daemon anchors a trace.

The cask uninstall stanza now deactivates the system extension
before removing the .app (`early_script` runs
`systemextensionsctl uninstall`). This eliminates the orphaned
"pending" sysext entry that pre-v1.10 cask uninstalls left in
sysextd's ledger.

### From v1.8.x or earlier

Same as v1.9.x — Sparkle / brew handles the upgrade. The schema
migrations run idempotently in version order on first launch. The
v1.3.0 sysext model has been in place since April 2026, so no
LaunchDaemon cleanup is required for any v1.8.x → v1.10 path.

## Breaking changes

None for end users. The dashboard layout has changed substantially —
a few familiar v1 panels moved (Events view is now under
Investigation; Threat Intel and Package Freshness are now under
Intelligence) — but no settings, data files, or CLI-invocation
contracts were broken.

## Known limitations

- **Phantom Alert fields**: the v2 dashboard surfaces panels for
  `analyst.*`, `d3fendTechniques`, and `remediationHint`, but
  `AlertStore` doesn't yet persist them — the panels gracefully hide
  via `if let / !isEmpty` guards so users see no broken pills, but
  the data isn't yet available. A v1.11 schema migration will
  introduce a `metadata_json` column to populate them.
- **isBrewInstalled detection**: brew users on Apple Silicon will
  receive Sparkle auto-updates because the cask install resolves
  `Bundle.main.bundleURL` to `/Applications/MacCrab.app` (a real
  directory, not a symlink), bypassing the brew-detection check
  that would have suppressed Sparkle in favour of `brew upgrade`.
  This is desirable for the v1.9 → v1.10 transition but will be
  fixed in v1.11.
