## MacCrab v1.6.14 — Settings slider actually reaches the daemon

v1.6.12 wired the size-cap enforcer. v1.6.13 hardened it against
failure modes. v1.6.14 closes the last mile: the dashboard's slider
now actually reaches the sysext, and two silent-decode bugs in
`daemon_config.json` parsing that had been voiding operator configs
for many releases are fixed.

### Bugs this release surfaces

**1. The Settings slider has been inert for as long as it's existed.**
`@AppStorage("maxDatabaseSizeMB")` and `@AppStorage("retentionDays")`
only write to the app's own `UserDefaults`. Nothing in the codebase
copied those values to `daemon_config.json`, which is what the sysext
reads. Operators who moved the slider got no behavior change — the
sysext kept the 500 MB / 30 d defaults. v1.6.12 fixed the daemon
side; v1.6.14 fixes the UI side.

**2. `daemon_config.json` parsing was silently dropping every
operator-supplied config.** Two compounding hazards in
`DaemonConfig.load()`:

- `JSONDecoder.keyDecodingStrategy = .convertFromSnakeCase` turns
  `max_database_size_mb` into `maxDatabaseSizeMb` (lowercase `b`).
  The Swift property is `maxDatabaseSizeMB` (capital `B`). The decode
  fails with `keyNotFound`.
- Swift's auto-synthesized `Decodable` ignores stored-property
  defaults — every non-Optional field must be present in the JSON, or
  decode fails. Any partial config (almost all of them) was rejected.

Either hazard caused `load()`'s `try?` to swallow the error and
return `DaemonConfig()`. Anyone who followed the CLAUDE.md example
of `"max_database_size_mb": 500` got all-defaults on every field —
not just storage, but thresholds, poll intervals, outputs, LLM
provider. The daemon has been running on compile-time defaults on
every machine that had a `daemon_config.json` since the feature
shipped.

**3. Size-cap ignored the WAL.** Enforcer measured only the main
`.db` file. A 480 MB main file + 40 MB WAL + 2 MB SHM presented as
"under 500 MB cap" while actually consuming 522 MB on disk.

**4. Orphan user-domain DBs linger after reinstall.** Pre-sysext
dev daemons wrote to `~/Library/Application Support/MacCrab/events.db`.
After a sysext install that DB becomes orphaned (sysext writes to
`/Library/...`). The dashboard's most-recent-mtime picker could
still select the orphan.

### What's fixed

- **Dashboard → daemon wiring.** Settings slider `onChange` now
  writes `~/Library/Application Support/MacCrab/user_overrides.json`
  and sends `pkill -HUP com.maccrab.agent`. The daemon overlays
  this file on top of its own `daemon_config.json` during
  `DaemonConfig.load()`, bounded to the two storage keys and gated
  by home-directory uid ownership (so a rogue process can't write
  overrides as a different user).
- **SIGHUP reloads storage config.** `SignalHandlers.install()`
  now re-reads `daemon_config.json` + user overrides on SIGHUP,
  updates the live `state.maxDatabaseSizeMB` / `state.retentionDays`,
  and kicks an immediate size-cap sweep if either changed. Both
  timer closures (prune + size-cap) read those values live each
  tick instead of capturing them at setup — no restart needed.
- **Defense-in-depth decoder.** `DaemonConfig.decode()` now:
  (a) overlays the user dict on top of a freshly-encoded
  `DaemonConfig()` defaults dict, so partial configs always
  decode; (b) pre-rewrites known snake_case keys with
  trailing-uppercase hazards (`max_database_size_mb`,
  `maxDatabaseSizeMB`) to the exact property camelCase before
  feeding the decoder.
- **WAL-aware size check.** `measureDatabaseFootprintMB()` sums
  `db + db-wal + db-shm`. Operators setting tight caps now get
  consistent numbers between `du` and the daemon's enforcer.
- **Orphan-DB reaper.** On root-sysext startup, stale (>24 h)
  `/Users/*/Library/Application Support/MacCrab/events.db*` files
  are renamed to `events.db.orphan-<YYYYMMDD-HHMMSS>*` instead of
  deleted — forensic evidence preserved, dashboard stops picking
  the orphan.

### Tests

**783 tests pass (up from 776).** Seven new tests across two suites:

- `DaemonConfigOverridesTests` (4) — defaults when no file,
  snake_case keys decode, camelCase keys decode, partial config
  leaves untouched fields at defaults.
- `DatabaseFootprintTests` (3) — missing files = 0, db-only path,
  db+wal+shm summed correctly.

### Upgrading

Install v1.6.14 → open MacCrab → Settings → Storage Limit → set the
cap you want → SIGHUP fires automatically. Verify with:

```
log show --predicate 'subsystem == "com.maccrab.agent"' --last 5m \
  | grep -i "storage config\|size-cap"
```

You should see a `Storage config reloaded: maxDatabaseSizeMB
OLD->NEW` line and a sweep invocation if you dropped below current
usage. Any orphan user-domain DB from pre-sysext installs is already
quarantined at sysext startup — no manual cleanup required.

### Meta-lesson

Three generations (v1.6.12, v1.6.13, v1.6.14) of trying to fix "the
size cap doesn't work" each landed a real fix and each exposed the
next silent-failure layer beneath it. The wiki article
[[maxdatabasesizemb-wired-bug]] is updated with the v1.6.14
follow-on and a concrete rule going forward: when adding any new
`daemon_config.json` key, write a test that provides ONLY that key
in JSON and asserts the daemon behavior changes. The v1.6.12
post-mortem listed this as its lesson #2. Still unlearned until now.
