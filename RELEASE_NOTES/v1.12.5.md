# MacCrab v1.12.5 — Self-FP cleanup wave + Threat Intel fix

v1.12.5 is the canonical v1.12 release. It bundles every feature
and hardening from v1.12.0 → v1.12.4 plus twelve targeted fixes
for self-FP and UX-papercut issues surfaced by field testing on
v1.12.4. Three of those (the Threat Intel directory permissions,
the tamper rule firing on MacCrab's own helpers, and the AI-tool
exfiltration rule firing on unrelated `curl` calls) had been
latent since v1.11.x or earlier — v1.12.5 closes them.

If you are on **v1.11.x or earlier**, this is the recommended
release to upgrade to. Sparkle in-place upgrade works directly:
open MacCrab.app, choose **Check for Updates…**, click **Install
Update**, relaunch when prompted.

### Manual upgrade for v1.12.0 / v1.12.1 installs

Your installed MacCrab.app carries broken nested Sparkle helpers
from a build-pipeline bug fixed in v1.12.2. Sparkle cannot install
any newer version. One-time manual upgrade:

```
brew update
brew upgrade --cask maccrab
```

Or download
[MacCrab-v1.12.5.dmg](https://github.com/peterhanily/maccrab/releases/download/v1.12.5/MacCrab-v1.12.5.dmg),
quit MacCrab, drag the new copy into `/Applications`. Future
Sparkle upgrades work normally.

### macOS 26 users currently on v1.12.2 / v1.12.3

If you're stuck in an Intelligence-tab crash loop:

```
defaults delete com.maccrab.app v2.dashboard.workspace
```

Then `brew upgrade --cask maccrab` or use Sparkle "Check for Updates…".

## What v1.12.5 specifically fixes (over v1.12.4)

Twelve fixes — eight from the v1.12.4 field-test report, four
deeper issues from the post-report polish audit.

### Self-defense (sysext-side) — Sparkle/cask upgrade FPs

1. **`Binary Modified` tamper alert during in-place upgrade**.
   `SelfDefense.integrityCheck()` was hashing and signature-
   verifying `CommandLine.arguments[0]` — under sysextd that's the
   bundle-relative loader path the kernel handed in, which doesn't
   resolve to the on-disk file after Sparkle/cask replaces the
   .app. Fixed by using `self.binaryPath` (proc_pidpath-resolved
   absolute path, captured once at init) everywhere the binary's
   hash or signature is read.

2. **DispatchSource `.write` handler on the binary path had no
   gate.** During an in-place upgrade the DispatchSource fires
   `.write` on the binary too, and the old handler emitted
   `.configModified .high` with no sentinel/signer check. Now
   honors `isSelfUpdateInProgress` AND `isSignedByMacCrabTeam`,
   matching the `integrityCheck()` gate.

3. **Periodic `integrityCheck()` rules-hash branch had no sentinel
   gate.** The DispatchSource rules-write handler honored
   `isSelfUpdateInProgress` since v1.12.1, but the 15-second
   periodic check didn't — so a Sparkle rule push that arrived
   past the 60 s startup grace window would re-fire
   `.rulesModified .high` every 15 s. Fixed by applying the same
   gate.

4. **`integrityCheck()` rules-dir lookup**. Was fishing for the
   rules path via `monitoredPaths.first(where: { $0.description.
   contains("rules") })`; if more than one entry's description
   contains "rules", the wrong path is picked. Now uses
   `self.rulesDir` directly.

5. **`checkForImpersonation` never matched the release sysext.**
   Probe was `pgrep -x maccrabd`. The release sysext executable
   is named `com.maccrab.agent` — the probe never found it, so
   the impersonation protection was silently disabled in
   production. Now matches either name with `pgrep -f`.

### Self-defense (rules-side) — `Attempted Tamper of MacCrab Components`

6. **Rule fires on `/bin/rm`, `/usr/bin/pkill`, `/bin/launchctl`
   spawned by MacCrab itself** at startup. The `filter_maccrab_self`
   block was AND-style (Image AND ParentImage both had to match
   MacCrab) — it didn't catch the common shape where MacCrab
   spawns a system helper (Image=`/bin/rm`, Parent=MacCrab).
   Added a Parent-only `filter_maccrab_parent` matching any
   process whose parent is a MacCrab binary
   (`MacCrab.app/Contents/`, `com.maccrab.agent`, `MacCrabAgent`,
   `maccrabd`, `maccrabctl`).

7. **Same rule extended to brew/cask/Sparkle upgrade lineage**.
   `filter_installer` had `/installer`, `/system_installd`,
   `/Sparkle`, `/sparkle`. Now also covers `/Updater`,
   `/Autoupdate`. Added `filter_upgrade_installer` matching
   ParentImage `/brew`, `/osascript`, `/sysextd`, `/launchctl`,
   `/Updater`, `/Autoupdate`. Added `filter_brew_lineage` matching
   CommandLine `/Homebrew/`, `/Cellar/maccrab/`, `Caskroom/maccrab/`.
   Every legitimate cask uninstall step + the dashboard's
   elevated rule sync + sysext relocation now filter cleanly.

### Threat Intel — feeds 0, IOCs 0 (latent since v1.11.0 RC2)

8. **Threat intel directory permissions blocked the dashboard's
   read.** `/Library/Application Support/MacCrab/threat_intel/`
   was created `0o700` by `ThreatIntelFeed.init()` as a v1.11.0
   RC2 "tightening" audit fix. The cache *file* inside was
   `0o644` (world-readable), but you need traverse (`x`) on the
   directory to open files inside — `0o700` blocked the
   user-context dashboard's read. Symptom on field machines:
   Threat Intel panel stuck at feeds=0 / iocs=0; "Refresh" ran
   but `ThreatIntelFeed.cachedIOCs(at:)` silently failed at the
   directory level. **Reverted to `0o755`**; cache file stays
   `0o644`. The original audit's "privacy/timing leak" concern
   was paper-thin for a single-user workstation. On upgrade,
   `ThreatIntelFeed.init` calls `setAttributes` to chmod the
   existing dir to `0o755` so the fix applies to pre-existing
   installs without operator intervention.

9. **`maccrabctl intel refresh` exit 1**. Compounding bugs: the
   `pkill -HUP com.maccrab.agent` invocation was missing `-f`
   (matches against Darwin's 15-char `p_comm`, never sees the
   bundle identifier), and SIGHUP is reserved for the rule-reload
   channel — SIGUSR1 is the intel-refresh signal. Plus
   user-context `pkill` against the uid-0 sysext returns EPERM
   regardless. Now uses `-USR1 -f` and on EPERM prints an
   informational message + exits 0 instead of dying with a fake
   error.

### AI Guard — `AI Tool Child Process Uploads Data to Non-Standard Destination`

10. **Rule fired on any `curl -X POST` at startup, including
    unrelated system tooling.** Title says "AI Tool Child
    Process" but the condition didn't enforce it — pre-fix any
    process running `curl -X POST` to an unknown destination
    tripped it. Added `selection_ai_parent` requiring ParentImage
    to be a known AI tool (claude / cursor / copilot / codex /
    aider / continue / cody / zed / windsurf / goose) or their
    editor-app helpers. Rule now means what its title promises.

### Dashboard — Intelligence workspace

11. **Integrations panel showed
    `ModifiedContent<Text, _ForegroundStyleModifier<Color>>(...)`
    debug dump**. Used `Text("foo \(Text("bar").bold())")`
    interpolation; SwiftUI doesn't render Text-in-Text via string
    interpolation, it calls `description` on the modified struct
    and splices the dump. Switched to markdown `**bold**` which
    SwiftUI's `LocalizedStringKey`-init Text renders natively.

12. **Integrations panel empty despite Objective-See tools
    installed**. `SecurityToolIntegrations.writeSnapshot()` was
    writing `integrations_snapshot.json` correctly, but the
    dashboard's `_loadIntegrations` only read three config files
    (`daemon_config.json`, `notifications.json`,
    `alert_notifications.json`) — it never read the snapshot.
    Added a `SecurityToolIntegrations.readSnapshot(at:)` call
    that surfaces every detected tool as a `kind: "detected-tool"`
    integration entry. Expanded Objective-See coverage to include
    RansomWhere, ReiKey, DoNotDisturb, TaskExplorer, Netiquette,
    WhatsYourSign, ProcessMonitor, FileMonitor.

### Typosquat scoring — `pip` false positive

13. **`pip` flagged as a typosquat of `pipx` with score 80**.
    The exact-match-is-popular check at the top of
    `TyposquatDatabase.score(...)` only handled the *homoglyph*
    case (Cyrillic 'р' encoding). For plain ASCII `pip` — which
    is itself in the top-PyPI corpus — the function fell through
    to the distance loop and matched `pipx` at distance 1. Added
    the ASCII membership check *before* the homoglyph branch;
    locked in by `pipIsNotTyposquatOfPipx` regression test
    covering `pip` / `pipx` / `next`.

v1.12.1, v1.12.2, v1.12.3, and v1.12.4.

### Detection — supply-chain coverage wave

v1.12 ships detection coverage for the September 2025 Shai-Hulud
worm class and the April–May 2026 follow-on incidents
(Mini Shai-Hulud, Lightning PyPI, TanStack CVE-2026-45321). Rule
corpus growth (vs v1.11.1):

|------------------------------|---------|---------|------|
| Single-event Sigma           | 388     | 424     | +36  |
| Sequence (temporal)          | 33      | 39      | +6   |

The canonical worm self-propagation signature (credential read +

### Detection — intent posterior

Bayesian belief network per process tree + LLM-backed
`IntentClassifier` over a `BehaviorBrief` (package + lineage +
credential reads + egress + anomaly flags). Routes through
`LLMService` (Ollama / Claude / OpenAI / Mistral / Gemini) with
sanitization, caching, circuit breaker. Heuristic fallback alone
catches the worm self-propagation shape — no cloud dependency
required.

### Performance — daemon cold start: 114 s → 118 ms

Seven sources of synchronous main-thread work deferred behind
`Task.detached` after the first heartbeat: 7 GB `tracegraph.db`
quick_check, `SelfDefense` self-SHA-256, `BaselineEngine.load`,
`ThreatIntelFeed`, `BundledThreatIntel`, `ESClientMonitor`, six
polled monitors. 14 `boot_phase` breadcrumbs for regression
visibility.

### Dashboard — in-app Sigma YAML editor

Read-only viewer + TextEditor save flow piped through the bundled
`compile_rules.py`. Writes to `user_rules/<uuid>.json`; daemon
picks up via `.reload_tick` mtime watcher. Disable / re-enable
rule actions write the same overlay with `enabled=false`.

### MCP — 8 new tools (25 total)

`check_typosquat_score`, `scan_package_content`,
`analyze_package_metadata`, `verify_package_attestation`,
`classify_package_intent`, `predict_next_technique`,

### Privacy + LLM hardening

- Webhook + syslog default hostname → `maccrab-host`
- MCP `get_alerts` / `get_alert_detail` / `scan_text` route
  through `LLMSanitizer.sanitize()`
- OpenAI host allowlist: dot-anchored exact-host Set +
  `.openai.azure.com` dot-suffix with length guard
- Gemini model-name regex allowlist; Ollama nil-host check

### Self-defense — sentinel + signer-validity gates (from v1.12.1)

`RuleBundleInstaller.copyRulesWithElevation` drops
`/Library/Application Support/MacCrab/.maccrab_self_update_in_progress`
at the start of its elevated session (root:admin 0644), removes
at the end. `SelfDefense` checks the sentinel before firing
delete/rename/write alerts on data-dir paths.
`isSignedByMacCrabTeam(at:)` shells to `codesign -dvv --verify --`
and verifies `TeamIdentifier=79S425CW99` before firing on binary
hash mismatch. Trust posture unchanged for actual tampering.

### Build pipeline — Sparkle install fix (from v1.12.2)

Dropped `--deep` from the outer `.app` codesign pass. Was
propagating MacCrab's main-app entitlements onto Sparkle's
`Installer.xpc`, making macOS refuse to launch the updater.
`Resources/*` still seal via `_CodeSignature/CodeResources`.

### macOS 26 Tahoe support (from v1.12.4)

`TyposquatDatabase.loadBundledCorpus(name:)` no longer reaches
`Bundle.module` (which `fatalError`s on macOS 26 because of a
stripped SwiftPM resource-bundle `Info.plist`). Probes the
canonical SPM search paths directly with `Data(contentsOf:)`.
`build-release.sh` also overwrites the SPM-stub `Info.plist` with
a complete CFBundle plist (defense in depth for any future
`Bundle.module` consumer).
