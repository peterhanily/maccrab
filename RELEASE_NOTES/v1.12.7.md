# MacCrab v1.12.7 — dashboard responsiveness fix

v1.12.7 is a single-purpose patch release on top of v1.12.6.
SwiftUI refresh-binding code only — no schema migrations, no
new entitlements, no new external network paths, no rule
changes.

If you're on **v1.12.6**, Sparkle in-place upgrade works directly:
open MacCrab.app → **Check for Updates…** → **Install Update** →
relaunch when prompted.

## What was broken

Two related field-reported bugs on production hosts with a busy
events.db / alerts.db:

1. **Auto-refresh staleness.** The dashboard's Alerts, Campaigns,
   and Metrics panels stopped refreshing until the user closed
   and reopened the menubar window. New alerts/events from after
   the dashboard opened never showed up in the live tables. Close
   and reopen (no quit) always restored fresh data, then 5–10
   seconds later it would go stale again.

2. **Mutation lag.** When the user clicked Suppress on an alert,
   the row sat unchanged for ~5 seconds before the suppression
   visibly applied. The 5s lag was particularly noticeable when
   the user expected the alert to immediately move from the Open
   tab to History.

## What's fixed

### Wave 9P — partial-write refresh pattern in 5 workspaces

Same root cause as Wave 9G (v1.12.6 RC2) in V2IntelligenceWorkspace,
but in 5 more workspaces. Each `.task(id:)` body gated all `@State`
writes behind one trailing `MainActor.run` after multiple
sequential SQLite reads:

```swift
let a = await provider.alerts(...)       // up to 200 rows
let c = await provider.campaigns(...)    // up to 50 rows
let k = await provider.kpis()            // 5 inner queries
let h = await provider.alertHistogram()  // ≤ 10K row scan
await MainActor.run {
    self.alerts = a; self.campaigns = c
    self.kpis = k;   self.histogramBuckets = h
}
```

On a host with a large DB, the combined load could exceed the 5s
auto-refresh tick. When `state.refreshTick` incremented, SwiftUI
cancelled the body before the trailing `MainActor.run` fired —
every reload silently discarded its result. Closing the menubar
window destroyed the view; reopening created a fresh state with
`refreshTick = 0` and an uncontested 5s window for the first
load, which usually completed.

Fix: write each piece of `@State` to its own `MainActor.run` as
soon as the individual await resolves. Worst case post-fix: only
the SLOWEST single query in a workspace might still miss a tick;
faster queries before it always land. Workspaces patched:

- `V2OverviewWorkspace` (alerts → campaigns → kpis → alertHistogram)
- `V2AlertsWorkspace.reload()` (alerts → campaigns → suppressions)
- `V2SystemWorkspace` (heartbeat-fast lands first; permissions can lag)
- `V2DetectionWorkspace` (rules → extensions → mcpServers → haystack)
- `V2InvestigationWorkspace.reload()` (traces → recentAlerts)

`V2EventsWorkspace` and `V2PreventionWorkspace` were audited and
confirmed not vulnerable (different refresh model / single-await
minimal structure with no partial-split possible).

### Wave 9Q+9R — optimistic mutations with pending-overlay reconciliation

The dashboard is read-only since v1.12.6 Wave 9A; mutations go
through a file-IPC inbox channel that the daemon polls every 5
seconds (hardcoded in `DaemonTimers.swift:1169`). The daemon's
inbox poll cadence plus the dashboard's reload cadence together
meant every mutation had a ~5–10 second visible lag.

Wave 9Q's first attempt flipped local `@State` on click — but
the immediate `state.refreshTick += 1` it added then fired a
reload that read the still-pre-mutation DB value back over the
optimistic flip. User saw: instant flip → bounce-back → 5s wait
→ real change. Worst of both worlds.

Wave 9R adds a pending-mutation overlay. Five `Set<String>` track
in-flight mutations:

- `pendingSuppressedAlertIds`
- `pendingUnsuppressedAlertIds`
- `pendingDeletedAlertIds`
- `pendingSuppressedCampaignIds`
- `pendingLiftedSuppressionKeys`

Each mutation function:

1. Inserts the relevant ID into the pending set.
2. Flips local `@State` (e.g. `alerts[idx].suppressed = true` or
   `alerts.removeAll { $0.id == id }`).
3. Dispatches the provider call.
4. On failure: removes from pending set and rolls back the flip.

`reload()` then MERGES the pending overlay on top of the DB read
— so even when the next refresh tick reads `alerts.db` before the
daemon has applied the inbox file, the optimistic flip survives.
`reload()` also PRUNES pending entries whose DB value has caught
up (e.g. DB now returns the alert with `suppressed=true`, so the
local pending registration is no longer needed and is dropped).

Net user-visible behavior:

| Action | Pre-9Q | Wave 9Q | Wave 9R (shipping) |
|---|---|---|---|
| Click Suppress | 5 s wait → row moves | Instant flip → 1–5 s bounce-back → real change | Instant flip, stays flipped |
| Click Delete | 5 s wait → row vanishes | Same bounce-back | Instant removal, stays removed |
| Click Unsuppress | 5 s wait | Same bounce-back | Instant restore |
| Click Suppress Campaign | 5 s wait | Same bounce-back | Instant removal |
| Click Lift Suppression | 5 s wait | Same bounce-back | Instant removal |

The DB-side mutation still takes the same ~5–10 seconds (file-IPC
inbox + daemon apply); the user just no longer sees the gap.

Mutations patched: `suppress`, `suppressCampaign`, `bulkSuppress`,
`bulkSuppressCampaigns`, `unsuppressAlert`, `deleteAlert`,
`liftSuppression`.

V2MockAlert.suppressed changed from `let` to `var` (required to
flip the flag in-place for the optimistic toggle).

## Known limitations

- **Pending set entries never expire.** If a mutation succeeds at
  the inbox-queue step but the daemon never applies it (daemon
  crashed mid-flight, inbox file got corrupted), the optimistic
  flip persists in the local overlay indefinitely. The next
  dashboard close/reopen clears all per-workspace `@State`,
  including the pending sets, so the worst-case "stuck" overlay
  is bounded by the next window close-reopen cycle.
- **Bulk-suppress partial-failure leak.** `bulkSuppress` adds every
  selected alert ID to `pendingSuppressedAlertIds` up-front, then
  dispatches. If the provider's per-ID `queueInboxMutation` fails
  for some IDs (e.g. inbox dir not writable or disk full mid-batch),
  the failed IDs are never queued for the daemon — but they stay
  in the pending overlay, so they LOOK suppressed in the UI
  forever. Recovery: close/reopen dashboard to clear `@State`.
  Bounded by how rare the partial-inbox-write failure is in
  practice (very rare on a healthy system).
- **Cross-workspace consistency lag.** Suppressing an alert in
  V2AlertsWorkspace flips the overlay there, but V2OverviewWorkspace's
  Recent Activity card reads via its own `reload()` which doesn't
  consult the V2Alerts pending sets. So an alert suppressed on
  the Alerts tab may briefly still appear on the Overview tab
  until the daemon catches up to both. Worst-case delay ≤ the
  daemon's inbox poll cadence (5 s — hardcoded in DaemonTimers).
