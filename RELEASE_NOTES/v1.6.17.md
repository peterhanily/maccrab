## MacCrab v1.6.17 — Threat Intel: 50–200× more IOCs, per-IOC context, working browser

v1.6.16 surfaced the IOC values but the Browse tabs were just bare
strings, the cache was loading ~30–500 entries from "_recent" feed
endpoints, and the picker UX masked which category had data. v1.6.17
fixes all three plus the missing real-time-refresh path.

### What changed

- **Feed sources upgraded from "_recent" / "_recommended" to full CSV
  exports.** The previous endpoints were lightweight 24–48 h
  windows — typically 30–500 entries per feed. The full CSVs deliver
  the actual active IOC sets:

  | Feed              | Old endpoint                                 | New endpoint                               | Order of magnitude |
  |-------------------|----------------------------------------------|---------------------------------------------|---|
  | Feodo Tracker     | `ipblocklist_recommended.txt` (~30–300)      | `ipblocklist.csv` (full active set, ~1 k–5 k) | 10–100× |
  | URLhaus           | `text_recent/` (~500–2 k URLs)               | `csv_online/` (currently active, ~3 k–8 k) | 5–10× |
  | MalwareBazaar     | `txt/sha256/recent/` (~100 hashes/day)       | `csv/recent/` (full recent CSV with metadata) | 1–5× larger AND carries family + tags |

- **Every IOC now carries metadata** instead of being a bare string:
  - `source` — which feed contributed it ("MalwareBazaar", "Feodo",
    "URLhaus", "Custom")
  - `firstSeen` — when the feed first observed the IOC in the wild
  - `malwareFamily` — Lumma, AsyncRAT, Cobalt Strike, etc.
  - `tags` — free-form labels from the feed (campaign IDs, droppers,
    payload kinds)
  - `fileType` — for hashes only ("exe", "macho", "dll", …)
  - `lastSeenInFeed` — when MacCrab most recently saw the IOC in a
    refresh; drives age-based eviction

  The `ThreatIntelFeed.IOCRecord` struct replaces the previous
  `Set<String>`. All four lookup methods (`isHashMalicious`,
  `isIPMalicious`, `isDomainMalicious`, `isURLMalicious`) preserve
  their existing API, plus four new accessor methods — `recordForHash`,
  `recordForIP`, `recordForDomain`, `recordForURL` — return the full
  metadata for use in alert descriptions.

- **Browse IOCs tab rebuilt around context.** Each row now shows:
  - Color-coded source chip (red MalwareBazaar, orange Feodo, blue
    URLhaus, green Custom)
  - Malware family pill if known
  - File-type pill for hashes
  - Tag pills (first 3, with a +N counter for the rest)
  - First-seen date
  - Hashes display first-12 + last-12 hex with an ellipsis so they
    fit a row without truncating to noise
  - Search now matches IOC value, family, tag, source, and file type —
    not just the bare string
  - Newest-first sort so the most-recently-active IOCs appear at the top
  - Per-category counts shown in the picker labels themselves (the
    v1.6.16 picker visually responded but with no per-category
    counts you couldn't tell which tabs had data — fixed)

- **Per-feed health row.** Three colored badges (green/orange/red) +
  hover-tooltips — visible signal when a specific upstream feed is
  failing (HTTP 503, network blocked, etc.) instead of the analyst
  guessing why counts haven't moved. Per-feed `lastUpdate` and
  `lastError` (with timestamp + reason string) live in the cache file.

- **Refresh Now button.** New control next to the "Last sync" badge
  sends `SIGUSR1` to the daemon (sysext + `maccrabd` fallback),
  which triggers a one-shot feed refresh. Operators don't have to
  wait the 4 h cadence after editing custom IOCs or after a feed
  outage.

- **Per-category caps + age-based eviction.** Bounds the cache so a
  runaway feed (or a hypothetical malicious mirror) can't drive
  unbounded growth:

  | Category | Hard cap | Default age cutoff |
  |----------|----------|--------------------|
  | Hashes   | 200 000  | 30 days |
  | IPs      | 25 000   | 30 days |
  | Domains  | 100 000  | 30 days |
  | URLs     | 75 000   | 30 days |

  Records that haven't appeared in any feed refresh for 30 days are
  evicted (the feed advertises IOCs only while they're active, so
  stale entries are sinkholed / cleaned-up indicators). Beyond the
  cap, oldest-by-`lastSeenInFeed` is dropped first. **Custom imports
  are pinned** — operator-supplied IOCs persist across refreshes and
  age cycles regardless.

### Cache schema

The on-disk `feed_cache.json` schema changed. v1.6.17 daemons rewrite
the cache within hours of upgrade — no migration needed but the
dashboard will show "no cache yet" briefly on first launch after
upgrade until the daemon completes its first feed refresh.

### Known scope limits

- **Per-feed timestamps may be missing on first cache write** if any
  feed fails on first sync (the cache file is still written but the
  per-feed map only records successful syncs). The dashboard renders
  this as "never" / gray badge.
- The "No threat-intel cache yet" empty state may flicker briefly
  on dashboard launch immediately after upgrading — gone after the
  first 30 s poll.
- **Other panels not yet audited.** The v1.6.17 work focused on
  Threat Intel. A panel-by-panel review of Browser Extensions,
  Prevention, Package Freshness, Integrations, AI Guard, ES Health,
  Permissions (TCC), Docs, Rules, Events, Campaigns is queued for
  v1.6.18 — same pattern: surface the data the daemon actually has,
  add context to bare lists, expose feed health.

### Tests

**801 tests pass.** Two existing
`ThreatIntelFeedCachedStatsTests` cases were updated to drive a real
`ThreatIntelFeed` actor (so they exercise the new schema end-to-end
through the encoder rather than constructing fixture JSON by hand).

### Upgrading

Install v1.6.17. Open Threat Intelligence → Browse IOCs and click
"Refresh Now" to populate immediately, or wait for the next 4 h
auto-refresh cycle. Existing custom imports + API keys + suppression
state preserved unchanged.
