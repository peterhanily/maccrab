# MacCrab v1.7.7 — Memory hot-fix (autorelease pool drain)

**TL;DR:** v1.7.6 daemons could accumulate 1+ GB of Swift heap over an
hour at sustained event rates, because two collectors (`EsloggerCollector`,
`UnifiedLogCollector`) parse JSON per event in long-running async Tasks
without an `autoreleasepool`. Foundation autoreleased objects from
`JSONSerialization.jsonObject(with:)` had nowhere to drain. v1.7.7 wraps
each iteration in `autoreleasepool { }` — one-line-per-loop fix, zero
behaviour change, bounded heap from the first event.

## Field reproduction

A v1.7.6 daemon at 197 events/sec sustained climbed from 50 MB → 1.52 GB
RSS in ~90 minutes. Heap dump:

```
NSConcreteData (Bytes Storage):  42,267 × 16 KB =  692 MB
CFString:                     2,340,055 × 127 B =  297 MB
NSDictionary:                 2,339,169 × 64 B  =  150 MB
NSError:                      2,339,131 × 48 B  =  112 MB
_NSJSONReader:                2,339,131 × 16 B  =   37 MB
                                                 ─────────
                                                 ~1.29 GB
```

The exact 1:1:1 count (NSDictionary ≈ NSError ≈ _NSJSONReader, all
~2.34M) is the signature of `JSONSerialization.jsonObject(with:)` —
it allocates one of each per call, autoreleased.

The 692 MB of NSConcreteData are the input read buffers (per-line Data
slices from the eslogger/log streams).

## Root cause

Swift async Tasks don't carry an implicit `@autoreleasepool`. Foundation
APIs that internally autorelease (`JSONSerialization`, many CoreFoundation
bridge calls) accumulate their returns in the calling Task's pool. For
short-lived Tasks this is fine — the pool drains when the Task completes.
For collector hot loops that run for the lifetime of the daemon, it's a
slow leak proportional to event rate.

`EsloggerCollector.readLoop` calls `JSONSerialization.jsonObject(with:)`
per event in a `while true { ... }` loop. `UnifiedLogCollector` does the
same in its streaming consumer. Both leaked at proportional rates.

## The fix

Wrap each per-iteration body in `autoreleasepool { ... }`:

```swift
while let newlineIndex = residual.firstIndex(of: newline) {
    autoreleasepool {
        // ... existing line processing, JSONSerialization, normalise ...
    }
}
```

`continuation.yield` (AsyncStream) is synchronous — it doesn't suspend —
so the pool unwinds cleanly per event. Cost: nanoseconds per call. Benefit:
heap stays bounded regardless of run duration.

## Why this didn't bite earlier

The leak existed pre-v1.7.0. It was masked by the v1.7.3 → v1.7.5 daemon
crash-loops (the daemon never lived long enough to leak meaningfully).
v1.7.6 fixed the crash-loop. Field installs running v1.7.6 finally stayed
alive long enough to expose the latent autorelease accumulation. So this
isn't a v1.7.6 regression — it's a v1.7.6-revealed pre-existing bug.

## Diagnostic discipline added

The diagnostic chain that found this:
1. `ps -o rss` showed the climb (272 MB → 1.52 GB)
2. `vmmap` and the dashboard's Activity Monitor revealed Private Memory
   = 1.31 GB (excluding the 30 MB shared/mmap), proving it wasn't file-cache
3. `sudo heap` ranked the live allocations and the 1:1:1 NSDictionary/
   NSError/_NSJSONReader pattern named the source

A v1.7.7 audit pass for `JSONSerialization` (or any autoreleasing Foundation
API) used inside `while`/`for await`/`Task` bodies will land in
`scripts/pre-release-audit.sh` to prevent regressions of this class.

## How to install

### Sparkle auto-update

Open MacCrab.app. Update banner appears within minutes of appcast publish.
Click "Install Update". sysextd hot-swaps the binary; new daemon starts
with bounded heap.

### Homebrew

```bash
brew upgrade --cask maccrab
```

### Fresh install

[github.com/peterhanily/maccrab/releases/v1.7.7](https://github.com/peterhanily/maccrab/releases/tag/v1.7.7)

## What you don't need to do

- **No reboot.** Daemon process restart is automatic when sysextd swaps the binary.
- **No extension re-approval.** Same team-id + bundle-id.
- **No DB migration.** Storage layer unchanged from v1.7.6.

## Expected steady-state numbers after upgrading

For your machine's event rate (~197 events/sec, ~107 file events/sec),
expect daemon RSS to settle at:

- **~80–150 MB Private Memory** (Swift heap + actor state)
- **~30–60 MB Shared** (mmap of events.db window + frameworks)
- **Total RSS: ~120–250 MB**, oscillating with file-cache pressure

Compare this to the v1.7.6 trajectory of unbounded growth. If your install
holds RSS below 300 MB after running for a day, the fix is doing its job.
