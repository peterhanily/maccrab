# MacCrab v1.17.1 — notification + reliability fixes

A fast-follow on v1.17.0 that fixes how notifications work (GitHub
issue #2) and two issues operators reported.

## Fixed

- **Notifications now come from MacCrab — and stop when you uninstall.**
  Previously, alert banners were posted through macOS's built-in
  "System Events" helper, so there was no "MacCrab" entry in System
  Settings > Notifications to control them, and a leftover background
  component could keep firing banners after an uninstall. MacCrab now
  delivers its own notifications: they're attributed to **MacCrab**,
  get a normal on/off entry in System Settings > Notifications, and go
  away the moment the app is removed. If you've denied notification
  permission, the Notifications settings tab now tells you and links
  straight to the right System Settings pane.
- **"Reload rules"** no longer says *"Daemon not running"* when the
  detection engine is in fact running — the dashboard now signals the
  engine reliably.
- **Rule search** shows matching results immediately instead of a brief
  empty list while the rule index loads.

## Note on how notifications are delivered

Banners are now shown by the MacCrab menu-bar app while it's running
(it normally runs continuously). Detections are always recorded and
visible in the dashboard regardless, and you can review past alerts
there at any time.

## Upgrade

- **Homebrew:** `brew upgrade --cask maccrab`
- **Sparkle:** accept the in-app update prompt and relaunch when asked.
- **Manual:** download the DMG from the release page and replace
  MacCrab.app.

Universal (Apple silicon + Intel), Developer ID signed and notarized.
Minimum macOS 13.

Thanks to the operators who reported these — including the persistent-
notifications-after-uninstall report (issue #2).
