## MacCrab v1.5.4 — Fresh-install tamper alert fix + hardening

Hotfix addressing the false `rules_modified` tamper alert that fired on every
clean install, plus two security hardening changes prompted by a code review.

### Fixed

- **No more bogus "tamper detection: rules modified" alert on fresh install.**
  On first launch the sysext's SelfDefense component starts watching
  `/Library/Application Support/MacCrab/compiled_rules/` for writes — then
  the app's `RuleBundleInstaller` copies rules from the app bundle into that
  exact directory. The write event was being interpreted as an active attack
  on detection content, producing a critical tamper alert within seconds of
  opening the app.

  The fix: SelfDefense now observes a 60-second startup grace window. Writes
  to the rules directory during that window rebaseline the stored hash
  silently; writes after the window keep the original behavior (hash compare
  → `rules_modified` alert on any actual change). This also covers
  DaemonSetup's `sequences/` subdir creation at boot and Sparkle-delivered
  rule updates where the copy happens immediately after relaunch.

- **TCC.db probe rejects symlinks.** The v1.5.3 Full Disk Access query
  opened `/Library/Application Support/com.apple.TCC/TCC.db` with
  `sqlite3_open_v2`, which follows symlinks. An attacker with write access
  to that directory could swap the file for a symlink pointing at an
  attacker-controlled DB and steer the FDA probe. `AppState.querySysextFDAInDB`
  now `lstat`s the path first and bails if it's a symbolic link — matching
  the existing pattern used by EventStore and AlertStore.

- **FDA client match tightened.** The v1.5.3 fix used
  `LIKE 'com.maccrab.agent%'` to catch any `.systemextension`-suffixed
  variant of the sysext identifier. That pattern would also match any future
  `com.maccrab.agent.*` bundle ID. Replaced with an explicit set:
  `IN ('com.maccrab.agent', 'com.maccrab.agent.systemextension')`.

- `CLAUDE.md` tactic directory count corrected 18 → 17 (dev-only).
