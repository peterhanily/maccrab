# MacCrab v1.10.1 — Dashboard suppression hotfix

v1.10.1 is a hotfix release. It exists because suppressing alerts /
deleting alerts / suppressing campaigns from the dashboard silently
failed on every machine where MacCrab.app was running as the logged-in
user against a root-owned alerts.db — which is every notarized release
install since the v1.3 System Extension transition. The dashboard surfaced
a "Database is read-only" toast suggesting a workaround that didn't
exist (`maccrabctl alerts suppress <id>` is not a real subcommand).

Upgrade path: Sparkle in-place update from v1.10.0 lands the fix without
any user action other than relaunching MacCrab.app when prompted.

## What was broken

The dashboard runs as the user; the System Extension owns
`/Library/Application Support/MacCrab/alerts.db` as `root:wheel 0600`.
The five mutation paths in `V2LiveDataProvider` all opened that
database directly:

- `suppressAlert(id:)` — Alerts workspace per-row suppress
- `unsuppressAlert(id:)` — Suppressions viewer "Lift" button
- `deleteAlert(id:)` — History "Delete" action
- `suppressAlerts(ids:)` — Alerts workspace bulk-suppress
- `suppressCampaign(id:)` — Campaign card "Suppress all contributors"

Every one of them issued an SQLite `UPDATE` / `DELETE` against the
root-owned file and got back `SQLITE_READONLY`. The error toast told the
user to use `maccrabctl alerts suppress <id>`, which is not a real
maccrabctl subcommand (the existing `maccrabctl suppress` is rule-path
allowlisting via `suppressions.json`, not alert-row mutation), or to
"run the dashboard with elevated privileges" — LaunchServices ignores
`sudo open MacCrab.app`, so that path is also dead.

## The fix

Re-use the file-IPC scaffold that v1.10.0 added for the "Reduce
events.db now" flush button. The System Extension creates
`/Library/Application Support/MacCrab/inbox/` at boot with mode 1777
(world-write + sticky bit) and runs a poller that drains the directory.

v1.10.1 teaches the poller to recognize four new request-file prefixes:

| Prefix | Action |
|---|---|
| `suppress-alert-<id>.json` | `AlertStore.suppress(alertId:)` |
| `unsuppress-alert-<id>.json` | `AlertStore.unsuppress(alertId:)` |
| `delete-alert-<id>.json` | `AlertStore.delete(alertId:)` |
| `suppress-campaign-<id>.json` | Self + fan-out to all `campaignId == id` alerts + `CampaignStore.setSuppressed(id:, true)` |

Each file is a JSON `{"id": "<uuid>", "queuedAt": "<iso>", "source": "MacCrabApp"}`.
The owning UID is captured via `stat()` and logged for audit.

The poll interval drops from 30 s to 5 s — the original cadence was
sized for flush requests (infrequent, slow to run); alert mutations are
interactive and need to feel snappy.

The dashboard's mutation paths now:

1. Try the direct DB write first (works in dev where the daemon runs as
   the user against `~/Library/...`).
2. On `SQLITE_READONLY`, drop a JSON file into the inbox and return
   `true`. Local UI flips optimistically; the next 5 s alerts refresh
   tick observes the actual row state.
3. The error toast — only seen if the inbox dir itself is missing /
   unwritable, i.e. a corrupted install — now correctly tells the user
   how to recover instead of pointing at a phantom maccrabctl
   subcommand.

The campaign-suppress fan-out (loop through every alert with the
matching `campaignId` and suppress each) used to run client-side. The
daemon now does it under one IPC request, so a campaign with 80
contributing alerts is one inbox file instead of 80.

## Verifying after upgrade

After Sparkle relaunches MacCrab.app:

1. Open the Alerts workspace.
2. Right-click any open alert → Suppress.
3. The alert should grey out within 5 seconds. (Pre-fix: it would
   flash an error toast and stay visible.)

If the alert does not flip:

```bash
# Confirm the inbox dir exists with mode 1777
ls -ld /Library/Application\ Support/MacCrab/inbox

# Confirm the daemon is alive
sudo systemextensionsctl list com.apple.system_extension.endpoint_security

# Tail the daemon log for inbox processing
log stream --predicate 'subsystem == "com.maccrab.daemon"' \
  | grep '\[inbox\]'
```

The log should show `[inbox] suppress-alert id=… uid=… ok` shortly
after each click.
