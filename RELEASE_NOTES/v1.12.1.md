# MacCrab v1.12.1 ‚ÄĒ Self-defense FP hotfix

v1.12.1 is a same-day hotfix on top of v1.12.0. After v1.12.0 shipped,
field testing surfaced two false-positive tamper alerts that fire on
every v1.11.1 ‚Üí v1.12.0 Sparkle / cask in-place upgrade. v1.12.1
silences both without weakening the self-defense posture.

Sparkle in-place upgrade from v1.12.0 lands the fix without any user
action other than relaunching MacCrab.app when prompted.

## The two FPs

### `MacCrab Tamper Detection: File Deleted` on `/Library/Application Support/MacCrab/compiled_rules`

The dashboard's `RuleBundleInstaller.copyRulesWithElevation` runs
`osascript ‚Ä¶ rm -rf <compiled_rules> && cp -R ‚Ä¶` via an admin prompt
to refresh the installed rule corpus when the bundled
`.bundle_version` / manifest content is newer than the installed copy.
The sysext's `SelfDefense` actor watches that directory via a
`DispatchSource` file-system source and fires a critical "File
Deleted" tamper alert on every Sparkle / cask upgrade ‚ÄĒ by design,
since "EDR-disable-then-fire-payload" is a real attacker pattern.
The bug is that the existing critical-path detection couldn't tell
"MacCrab is updating itself" apart from "attacker is deleting
MacCrab".

### `MacCrab Tamper Detection: Binary Modified` on `maccrabd`

Sparkle's in-place .app replacement legitimately swaps the binary on
disk while the old version is still running in memory. The sysext's
periodic 15 s `integrityCheck()` sees the on-disk SHA-256 ‚Č† its
startup baseline and fires a critical `binary_modified` alert.
Same diagnosis: legitimate self-update was indistinguishable from
real tampering.

## What changed

### Sentinel-file mechanism (FP #1)

`RuleBundleInstaller.copyRulesWithElevation` now drops a sentinel
file at `/Library/Application Support/MacCrab/.maccrab_self_update_in_progress`
at the start of its elevated session (created `root:admin 0644` by
the same `osascript` admin command) and removes it at the end.

`SelfDefense` exposes a `nonisolated static` helper
`isSelfUpdateInProgress()` that returns true when the sentinel
exists AND its mtime is within the last 90 s. The file-event
handler checks this before firing delete / rename / write alerts on
paths inside the data dir:

- Sentinel fresh ‚Üí suppress the alert, log at `.notice`, schedule a
  delayed re-baseline (`Task.sleep(5 s)` + recompute the rules-dir
  hash) once the new tree has been written.
- Sentinel stale / absent ‚Üí fire the alert as before (real tamper).

The periodic `integrityCheck()` `fileDeleted` branch also consults
the sentinel, so the 15 s re-fire loop doesn't re-emit while the
elevated `cp -R` is still mid-flight.

The 90 s TTL bounds the suppression window if the sentinel ever
gets orphaned by a mid-flight failure. The cleanup step in
`copyRulesWithElevation` uses `; rm -f` (not `&&`), so the sentinel
clears even when an earlier shell step fails ‚ÄĒ defence-in-depth
against partial-success states.

### Signer-validity check (FP #2)

Before alerting on a binary hash mismatch in `integrityCheck()`,
`SelfDefense` now calls `isSignedByMacCrabTeam(at:)`, which shells
to `codesign -dvv --verify --` and checks the output contains
`TeamIdentifier=79S425CW99` (MacCrab's Developer ID team).

- Valid signature ‚Üí re-baseline silently, log at `.notice`. A real
  tamper can't forge our signature without the private key, so the
  gate is safe.
- Invalid / missing / mismatched signature ‚Üí fire the alert as
  before.

The `DispatchSource` `.delete` event on the binary path was already
non-critical and silently dropped (Homebrew upgrade pattern), so no
change needed there.

## Trust posture

Self-defense remains as strict as before for the cases it's meant
to catch:

- `rm -rf` of any monitored path from any process not holding the
  sentinel ‚Üí critical alert.
- Binary replaced with an unsigned, ad-hoc-signed, or other-team-
  signed binary ‚Üí critical alert.
- `pkill maccrab`, `launchctl bootout`, `systemextensionsctl
  uninstall` ‚ÄĒ all unchanged, all still surfacing as critical via
  the Sigma rule `maccrab_tamper_attempt.yml`.
- Debugger attach, signal-based termination, plist removal ‚ÄĒ all
  unchanged.

The suppression is keyed to TWO independent evidence points
(sentinel file + valid signature). Bypassing both requires a
sentinel write to a root-owned dir AND a forged code signature ‚ÄĒ
the same controls every commercial macOS EDR relies on.

## What you should see

After upgrading from v1.12.0 to v1.12.1:

- The v1.12.0 daemon may fire one final "binary_modified" or
  "file_deleted" alert during the swap itself (the v1.12.0 daemon
  is what's running while v1.12.1 replaces it on disk).
- Once v1.12.1 is the active version, future Sparkle / cask
  upgrades to v1.12.2+ should pass silently.

## Code changes (precise list)

- `Sources/MacCrabApp/RuleBundleInstaller.swift`:
  - `copyRulesWithElevation` shell command now opens with `touch
    '<sentinel>' && chown root:admin '<sentinel>' && chmod 0644
    '<sentinel>' && ‚Ä¶` and closes with `; rm -f '<sentinel>'`.

- `Sources/MacCrabCore/Detection/SelfDefense.swift`:
  - Added `static let selfUpdateSentinelPath`, `selfUpdateSentinelTTL`.
  - Added `static func isSelfUpdateInProgress()` (nonisolated).
  - Added `static func isSignedByMacCrabTeam(at:)` (nonisolated).
  - Added `handleSelfUpdateDelete(path:desc:)` actor method.
  - Added `rebaselineAfterSelfUpdate(path:)` actor method.
  - Wired sentinel check into the `.delete` and `.rename` branches
    of the `DispatchSource` event handler.
  - Wired signer check into the `binaryModified` branch of
    `integrityCheck()`.
  - Wired sentinel check into the `fileDeleted` loop of
    `integrityCheck()`.

- Version markers bumped in `MacCrabVersion.fallback`,
  `Casks/maccrab.rb`, `homebrew/maccrab.rb`, README badge,
  CHANGELOG entry.

No new rules, no schema changes, no behavior changes outside the
self-defense suppression path.
