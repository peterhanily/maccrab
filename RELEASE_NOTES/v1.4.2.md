## MacCrab v1.4.2 — Rules ship with the app; big noise drop

Every improvement to detection rules or the compiler between v1.3.11
and v1.4.1 never reached users who updated via Sparkle. The Homebrew
cask's postflight copies rules into place on install and `brew
upgrade`, but Sparkle replaces only the `.app` bundle — the installed
`compiled_rules/` directory stayed frozen at whatever version first
installed it. v1.4.1 field data proved it: rules like
`wifi_attack_tool` were still firing with the pre-v1.3.11 compiler
bug, `invisible_unicode_in_source` was still enabled after being
deprecated in v1.4.0, and four other rule changes hadn't taken effect.

### Fixed

- **Compiled rules now ship inside the app bundle and sync on
  launch.** `MacCrab.app/Contents/Resources/compiled_rules/` carries
  a `.bundle_version` marker. On every launch the app compares it
  against the installed copy at `/Library/Application Support/
  MacCrab/compiled_rules/`; when the bundled copy is newer, it
  replaces the installed tree and SIGHUPs the detection engine.
  Sparkle updates now ship rules. No more cask-reinstall workaround.

- **Cross-process correlator filters out shell-utility chains.**
  `brew install` legitimately fires thousands of events from
  `bash` + `ruby` + `curl` + `git` + `dirname` + `readlink` +
  `env` + `locale` all touching shared paths in
  `/opt/homebrew/Cellar/` and `/private/tmp/brew-*`. The v1.4.1
  user's `brew reinstall` produced 3,000+ chain alerts in 30
  seconds. New gate drops chains where ≥80% of participants are
  small shell helpers AND ≥4 distinct utilities are involved AND
  no `execute` action appeared — a classic curl→bash download-
  and-run attack (only 2 utilities, includes execute) still
  fires.

- **Correlator ignores terminal device files.** `/dev/tty*`,
  `/dev/pts/`, `/dev/ttys*` added to `ignoredPathPrefixes`. A
  62-hit repeat FP where "sudo+zsh touched /dev/ttys000" (the
  shell writing your password prompt to your terminal) now
  suppressed at ingress.

- **Correlator ignores Homebrew workflow paths.** Added
  `/private/tmp/homebrew-`, `/private/tmp/brew-`,
  `/opt/homebrew/var/`, `/opt/homebrew/Cellar/`,
  `/usr/local/Homebrew/`, `/usr/local/Cellar/`.

- **Campaign detector excludes USB hot-plug + crypto-token-kit
  alerts from kill-chain tactic counts.** Plugging in a YubiKey
  near an open terminal shouldn't look like "5 tactics, 14
  alerts Multi-Stage Attack" — but it did in field data.
  `maccrab.usb.*` and
  `maccrab.deep.crypto_token_extension` no longer contribute to
  kill-chain tactic thresholds.

- **Campaign detector excludes Apple-system-daemon processes
  from coordinated-attack emission.** Processes under
  `/usr/libexec/`, `/System/Library/`, or
  `/System/Applications/Utilities/` (xpcproxy, mobileassetd,
  usernoted, rtcreportingd, nsurlsessiond, …) legitimately span
  multiple MITRE tactics as part of macOS bookkeeping. Field data
  had 15+ bogus "Coordinated Attack from single process"
  campaigns over 24h on one user's machine — all Apple daemons.

- **`Credential File Access Followed by Network Upload`
  lowered from critical to high.** At critical it bypassed
  NoiseFilter Gate 3 (trusted-browser-helper suppression), so
  every Chrome/Edge/Firefox helper legitimately reading its own
  Cookies DB and syncing to the browser's cloud fired at
  critical. At high, Gate 3 drops the match on trusted browsers
  — a non-browser process doing credential-read→upload still
  fires at high. User had 22 critical FPs on Chrome Helper alone.

### Upgrading

Backwards-compatible. No schema changes. 600 tests pass.

**What changes for existing users on launch:**
The rule sync runs at app start. On most installs, the first v1.4.2
launch notices the installed rule set is older than the bundled one
and replaces it — your `/Library/Application Support/MacCrab/
compiled_rules/` directory is refreshed in place, and the detection
engine picks up the new rules within seconds via SIGHUP.

**Clearing historical alerts:**
If your DB has accumulated noisy alerts from the old rules (as
v1.4.1 field data showed), use Settings → Detection Engine →
Retention → "Clear alerts older than 7 days" to wipe them and start
with a clean signal baseline.
