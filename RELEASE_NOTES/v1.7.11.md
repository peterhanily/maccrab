# MacCrab v1.7.11 — Dashboard memory hot-fix

Patch-only. Dashboard target. Daemon unchanged from v1.7.10.

## What was broken

Field-reproduced: parking the dashboard on the **Events** tab grew
retained memory ~1.5 GB/day. Heap dump signature: 1+ million
`NSLayoutConstraint` + `NSISRestrictedToZeroMarkerVariable` —
AppKit Auto Layout solver state retained across NSTableView rebinds
that the SwiftUI poll cycle was triggering hundreds of times per
second.

Different process and different bug class from the v1.7.6 → v1.7.10
daemon-side leaks. This one is **retained** memory (not autoreleased
Foundation), in the **SwiftUI app target**, driven by NSTableView
constraint inflation.

## What v1.7.11 changes

### `EventStream` memoizes its filtered list

The view's body previously rebuilt the filtered+sorted events list
on every body re-evaluation, AND read it twice per body call (once
for the count badge, once for the Table data). Each unrelated
`@Published` mutation in AppState (heartbeat, agentLineage, MCP
baselines, etc.) triggered a body re-eval → double rebuild → fresh
Table rebind → Auto Layout constraint allocation that NSTableView
doesn't release.

v1.7.11 caches the filtered+sorted result in `@State` and recomputes
only when an actual input changes (events list, filter, sort).
Body reads the cache directly. SwiftUI sees a stable array reference
between unrelated AppState mutations and stops re-binding the Table.

### `AppState.refresh()` skips no-op work

Three high-frequency refresh functions (`refreshHeartbeat`,
`refreshStorageHealth`, `refreshRuleTamper`) now short-circuit when
their backing JSON file hasn't been re-written since the last
successful poll. Mirrors the existing pattern in `refreshAgentLineage`
/ `refreshMCPBaselines` / `refreshTCCSnapshot`.

Plus equality checks on `isConnected`, `appHasFDA`, `sysextHasFDA`,
and `fullDiskAccessGranted` so unchanged values don't fire
`@Published` and trigger SwiftUI body re-eval cascades.

### Audit codification

`scripts/pre-release-audit.sh` Pass 9 now also scans
`Sources/MacCrabApp/`. Won't catch this specific bug class
(constraint retention isn't an autoreleasepool-shape problem) but
forces authors of future dashboard hot-path code to think about
Foundation pool drainage when adding `JSONSerialization` /
`Data(contentsOf:)` / similar inside polling loops.

## How to install

### Manual / verification

[github.com/peterhanily/maccrab/releases/v1.7.11](https://github.com/peterhanily/maccrab/releases/tag/v1.7.11)

### Sparkle auto-update

Held intentionally pending field verification on the affected test
machine. Once a 2-3 hour soak with the dashboard parked on Events
confirms `NSLayoutConstraint` count stays under ~25K, the appcast
is published.

### Homebrew

Same hold. After verification: `brew upgrade --cask maccrab`.

## Expected steady-state numbers

For a dashboard parked on the Events tab:
- `NSLayoutConstraint` count: **stable around 5-15K**, not climbing
- App RSS: **120-250 MB**, stable indefinitely
- 24-hour soak: < 50 MB growth

## Compatibility

UX-only patches in `MacCrabApp` target. Daemon code unchanged.
No reboot, no extension re-approval, no data migration.

Tests: 929/929 passing. Pre-release audit: 10/10 passes green.
