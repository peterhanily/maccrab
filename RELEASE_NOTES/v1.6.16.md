## MacCrab v1.6.16 — Threat Intel browser + match list

The Threat Intelligence panel previously showed "Malicious Hashes:
3 142, IPs: 187, Domains: 41 822, URLs: 50 000" with no way to see
*what* those values actually were, where they came from, or what the
IOC list had matched on. v1.6.16 makes all three visible.

### What's new

- **Browse IOCs tab is now a real searchable browser.** The previous
  stub showed a stats summary plus a couple of CLI hints. Now: a
  category picker (Hashes / IPs / Domains / URLs), a substring search
  box, source attribution per category (MalwareBazaar / Feodo Tracker /
  URLhaus / custom imports), and a virtualized scroll list capped at
  the first 200 results so a 100K-hash set doesn't melt the UI.
  Search narrows the cap; analysts can pivot to `maccrabctl` for
  deeper analysis.

- **Recent Matches panel.** New section under Browse IOCs lists the
  alerts where threat-intel rules actually fired —
  `maccrab.threat-intel.hash-match` (CDHash matched a known-bad SHA-256)
  and `maccrab.dns.threat-intel-match` (DNS query to a known-bad
  domain). Each row shows the rule title, kind chip (hash / domain),
  process name, alert description (which contains the matched value),
  and the timestamp. Empty state explicitly distinguishes "list isn't
  catching anything" from "you're seeing a clean machine," with a
  pivot hint to filter the Alerts tab by `maccrab.threat-intel`.

- **`ThreatIntelFeed.cachedIOCs(at:)`** static accessor exposes the
  full on-disk IOC set without instantiating the actor. Same shape as
  `cachedStats(at:)` — single decoder call, returns a `IOCSet` value.

- **Empty-state distinguishes "no cache" from "zero IOCs."** When the
  daemon hasn't yet written `feed_cache.json` (fresh install, daemon
  paused, feed update failed) the browser shows a clock-and-arrow
  banner explaining the 4-hour refresh cadence and pointing to the
  Feeds tab for manual update — distinct from "loaded zero IOCs."

### Known scope limits

- **No per-IOC source attribution yet.** All three feeds merge into
  one `Set<String>` per category in the daemon's cache schema. The
  source caption per category is the best we can show without a
  schema migration. Tracked as a v1.7 enhancement (per-IOC
  provenance + first-seen timestamp).
- **IP-on-network-connection matches feed BehaviorScoring, not the
  alert store** — only DNS-domain and CDHash matches produce alerts
  today. So those won't appear in the "Recent Matches" panel even
  when an IP-list match contributes to a behavior-score detonation.
  Matches the existing pre-v1.6.16 behavior; promoting these to
  alerts is a behavior change with FP impact, deferred.

### Tests

**801 tests pass (up from 799).** Two new tests in
`ThreatIntelFeedCachedStatsTests`:

- `cachedIOCs returns the full IOC set, not just counts`
- `cachedIOCs returns nil when cache file is missing`

### Upgrading

Install v1.6.16. Open Threat Intelligence → Browse IOCs to use the
new browser. Recent Matches populates from existing AlertStore data,
no migration. Existing API keys, custom imports, feed cache, and
configuration are preserved unchanged.

### Why this shipped fast on top of v1.6.15

v1.6.15 wired `threatIntelStats` end-to-end (the counts had been
displayed-but-unwritten — the same bug shape as the maxDatabaseSizeMB
saga). Once the counts were live, the immediate analyst question was
"OK, but *what* hashes? *which* domains? what's been *caught*?" —
v1.6.16 answers that. Same daemon-side cache file, same atomic-swap
pattern, just the missing accessor + UI.
