## MacCrab v1.7.2 — Deferred-queue cleanup pass

The 8-item carry-over queue from v1.7.0 + v1.7.1, all in one release.
Pre-ship deep-dive review found 2 HIGH + 3 MEDIUM issues — all fixed
before push. 918 tests, 7/7 audit passes, ship-ready.

### Added — Collector liveness registry on heartbeat

New `CollectorRegistry` actor in `MacCrabAgentKit`. Tracks per-
collector last-tick timestamps, event counts, error counts, last-error
strings, and computed health (event-driven collectors get the benefit
of the doubt; polling collectors go unhealthy after 5× expected
interval without a tick). 16 collectors pre-registered at daemon
startup with their characteristic poll cadence + event-driven
classification.

Heartbeat schema bumped v3 → v4 with two new fields:
`collector_health: [{name, lastTickUnix, eventCount, errorCount,
lastError, expectedIntervalSeconds, healthy}]` and aggregate
`events_dropped: UInt64`. Backward-compatible: v1.7.0 / v1.7.1
dashboards reading a v4 heartbeat see the legacy fields unchanged
and silently ignore the new ones.

`MonitorTasks` event-stream consumers now call `recordTick(name:)`
once per emitted event — 12 single-line insertions across the
existing `for await event in state.<collector>.events` loops.

`ESHealthView` now renders the daemon-reported collector list
(replacing the previous hardcoded list) with per-collector health,
event count, error count, last-tick relative time, and last-error
inline.

### Changed — AI Analysis panel search

`AIAnalysisView` now has a `searchText` field that filters across
both Investigation Summaries and Defense Recommendations by
ruleTitle, description, processName, or processPath.

### Changed — Prevention panel search + per-mechanism drill

`PreventionView` adds a `searchText` field over recent prevention
activity, plus a per-mechanism drill sheet. Tap any row in "Recent
Prevention Activity" → opens a sheet showing every alert attributed
to that mechanism (DNS sinkhole, supply chain gate, persistence
guard, sandbox analysis, AI containment, network blocker, TCC
revocation, or Other). Mechanism inference uses rule-title token
matching with an explicit `Other` bucket for unknowns.

### Changed — Aider / Codex MCP spawn-shape matchers

`MCPAttributor.looksLikePackageToken` now recognizes `aider_mcp_*`
(Python module flag form), `@openai/codex` (NPM scope), and
`openai-codex-mcp-*` (Codex CLI MCP server form). Category
extraction supports the new prefixes. Carry from v1.7.0's deferred
field-coverage list. Two new tests cover each shape.

### Changed — Package Freshness panel search

`PackageFreshnessView` filters the local scan results by package
name, registry, or risk-level keyword. Existing per-package detail
sheet preserved.

### Changed — Integrations panel search + per-tool drill

`IntegrationsView` adds search over installed-tool name, version,
and capabilities. Tapping a tool card opens a detail sheet showing
the tool's path, log path, version, running status, and full
capabilities list.

### Added — Rule engine P50 / P95 / P99 execution percentiles

`RuleEngine.RuleStats` gains an `execSamplesNs: [UInt64]` reservoir
sized at 256 samples per rule (~860 KB worst case across 420 rules).
Sampling uses Vitter Algorithm R: under reservoirSize append; once
full replace at uniform-random index in `[0, reservoirSize)` with
probability `reservoirSize / evaluationCount`. Computed properties
`p50ExecNs`, `p95ExecNs`, `p99ExecNs` derive from the reservoir;
`RuleRow` shows p95 in the row footer when ≥50 samples exist
(threshold raised from 20 in pre-ship review).

### Added — EventStore schema migration v2: indexed MCP attribution

Migration v2 promotes MCP attribution from `raw_json` only (v1.7.0)
to top-level indexed columns: `mcp_server_name`, `mcp_server_category`,
`ai_tool_session_id`, plus a composite index
`idx_events_mcp_server` on `(timestamp, mcp_server_name)`. Insert
SQL bumped from 21 → 24 bind columns; missing attributions bind
NULL. Migration is wrapped in BEGIN/COMMIT by `SchemaMigrator`;
duplicate-column errors on retry are treated as already-applied.

### Pre-ship review fixes (HIGH)

- **`CollectorRegistry` lazy-registration default.** Pre-fix:
  unknown collectors got `eventDriven: true, expected: 60s` — a
  forgotten `register()` call would silently masquerade as
  event-driven and never fail health. Fix: default
  `eventDriven: false, expected: 300s` + `logger.warning` so the
  missing registration surfaces at runtime.
- **Heartbeat decoder silent drops.** Pre-fix: malformed
  `collector_health` entries from a buggy daemon were silently
  dropped via `compactMap`, leaving the dashboard short on
  collectors with no operator signal. Fix: count drops + log
  warning via `os.log` if any entry fails to decode.

### Pre-ship review fixes (MEDIUM)

- **`PreventionView.inferMechanism` fallthrough.** Pre-fix: unknown
  alert titles fell through to `.aiContainment` mis-bucketing the
  drill. Fix: explicit `.other` enum case + `aiContainment` is now
  matched only when title contains AI-Guard / credential / fence
  tokens.
- **RuleRow p95 threshold.** Pre-fix: showed p95 at 20 samples (5%
  tail = 1 outlier — too noisy). Fix: 50-sample minimum.
- **`ESHealthView` empty state contrast.** Pre-fix: "Awaiting first
  heartbeat…" rendered with `.tertiary` — fails WCAG AA on dark
  mode. Fix: `.secondary`.

### Notes for operators

- The schema v2 migration runs once on first v1.7.2 launch.
  EXISTING events in the DB get NULL values for the new columns —
  expected and correct. New events from v1.7.2+ get the columns
  populated automatically when MCP attribution fires.
- Heartbeat schema v4 is backward-compatible: v1.7.0 / v1.7.1
  dashboards continue to render older heartbeat fields and ignore
  the new ones. Mixed-version fleets work without coordination.
- `events_dropped` stays at 0 on healthy machines — it counts
  internal queue overflow, AsyncStream backpressure, and parse
  errors. Watch it for anything unexplained.
