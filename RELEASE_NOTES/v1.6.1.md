## MacCrab v1.6.1 — Noise reduction from field data

Hotfix for the alert-noise pattern observed on dogfood hosts: ~19 identical
"Injected Library: lldb-rpc-server loaded MaxClawdroomTVOS.debug.dylib"
alerts every 48 h on a quiet Xcode-using machine, plus occasional
"Multiple Processes Contacting Same Destination" firings on GitHub Desktop
and /usr/bin/curl, plus C2-beacon "alert storm" campaigns triggered by
Xcode's embedded git.

### What was wrong

Three structural bugs:

1. **Forensic-loop alerts bypassed the AlertDeduplicator entirely.** The
   5-minute forensic timer in `DaemonTimers.swift` inserted alerts directly
   into `AlertStore` without calling `deduplicator.shouldSuppress`. A long-
   lived process repeatedly flagged by `CrashReportMiner`,
   `PowerAnomalyDetector`, or `LibraryInventory` therefore produced a
   fresh alert every scan cycle.

2. **LibraryInventory had no Xcode / debug-workflow allow-list.** Xcode's
   `lldb-rpc-server` legitimately loads the user's compiled `.debug.dylib`
   for every active debug session. Because the library is unsigned (it's
   the user's local build output), the flagger fired critical alerts on
   routine iOS/tvOS development.

3. **CrossProcessCorrelator's network-convergence path didn't allow-list
   dev workflows** and its emission in `EventLoop.swift` also bypassed
   the deduplicator. GitHub Desktop Helper + git subprocess + openssl all
   contacting github.com counts as "convergence," but it's actually just
   `git push` happening.

### What v1.6.1 changes

- **All three forensic emitters route through `deduplicator.shouldSuppress` /
  `recordAlert`.** Identical alerts within the 1-hour suppression window
  are now dropped silently.

- **LibraryInventory process allow-list** covers `lldb-rpc-server`, `lldb`,
  `debugserver`, `Instruments`, `RemoteTestRunner`, `xctest`, `XCTRunner`.
  Path-prefix skip for `/Applications/Xcode.app/`. Build-artifact pattern
  filter for `.debug.dylib`, `/DerivedData/`, `/Build/Products/`,
  `/.build/debug/`, `/.build/release/`, `/target/debug/`, `/target/release/`.
  Plus per-process-per-library dedup inside the actor so the same loaded
  dylib can't re-alert across scan cycles.

- **`c2_beacon_pattern.yml`**: added `/usr/bin/`, Xcode-Developer, and
  CommandLineTools paths to `filter_system`. Severity dropped medium → low
  until a proper timing-variance sequence rule replaces the Sigma stub.
  Description updated to acknowledge the Sigma rule can't do timing
  analysis — a truthful framing is more useful than a misleading rule name.

- **CrossProcessCorrelator** gained `allEventsAreDevWorkflow` — new gate
  that skips convergence when every participant is a known dev-workflow
  binary (git + openssl + GitHub Desktop Helper contacting github.com no
  longer fires). Plus the emission path in `EventLoop.swift` now goes
  through the deduplicator keyed on destination host.

- **Six high-FP-risk Sigma rules** got standard `filter_terminal` +
  `filter_apple_parent` blocks: `curl_to_raw_ip`, `python_http_server`,
  `netcat_listener`, `ngrok_or_tunnel`, `sensitive_file_search`,
  `ssh_key_file_read`. The SSH key rule also gained IDE + backup-tool
  filters (Xcode, Cursor, VS Code, JetBrains, Time Machine, Backblaze).

### Expected impact

On the dogfood host that produced the ~19 Xcode alerts / 48 h: that number
should drop to **1**. The first alert for a given (process, library) pair
still fires so new loads are visible; subsequent identical alerts within
the 1-hour suppression window are silent. The internal per-pair dedup keeps
it silent for the rest of the daemon's lifetime.
