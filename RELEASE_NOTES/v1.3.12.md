## MacCrab v1.3.12 — Suppressions survive upgrades

Critical hotfix for a long-standing bug that became noticeable after the
recent release cadence. Your dismissed alerts and suppression patterns
now persist across every upgrade path.

---

### Fixed

- **Suppressions no longer reset after an app upgrade.** Dashboard
  state (per-alert dismissals and rule/process suppression patterns)
  was being saved to a directory that can switch between
  `~/Library/Application Support/MacCrab/` and
  `/Library/Application Support/MacCrab/` depending on which
  database was touched most recently. After an upgrade the sysext
  writes fresh events to the system-wide database, that directory
  flips to the root-owned location, and the dashboard's writes to
  it silently fail (the dashboard runs as your user, not root).
  Next launch finds an empty state file in the new location and
  every previously-dismissed alert reappears.

  Fix: anchor all dashboard UI state to `~/Library/Application
  Support/MacCrab/`, where the dashboard always has write access.
  On first launch after updating, the old saved state is
  automatically read out of the legacy location and re-saved to the
  stable one, so you don't lose anything that's currently persisted.

### Also

- Established a `RELEASE_NOTES/v{VERSION}.md` convention so release
  notes surfaced in Sparkle and on the GitHub releases page are
  written for users, not pulled directly from the developer-facing
  CHANGELOG.

---

**Compatibility:** backwards-compatible. Existing suppressions are
migrated on first launch.
