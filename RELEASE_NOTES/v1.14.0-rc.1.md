# v1.14.0-rc.1 — MCFP R0 memo + R1 static-fingerprinter reference

**Status: internal release candidate.** Not published to Sparkle appcast or Homebrew cask. Source tag only.

First v1.14 RC. Closes both R0 (the dyld+SIP+library-validation paradox) and R1 (static reference implementation) of plan §6 MCFP. Static-only form ships; dyld + runtime components deferred to v1.15 R2 evaluation per the R0 decision.

## What landed

**`docs/mcfp-research/R0.md`.** R0 memo per plan §6.4 documenting the five options for resolving the dyld+SIP+library-validation paradox + the decision to ship option (d) — drop the dyld component for v1.14 — and the explicit re-evaluation triggers (R1 stability fail / R2 imposter evasion / sysext gains NOTIFY_MMAP for another reason).

**`docs/mcfp.md` v1 specification.** The canonical fingerprint form:

```
mcfp1/static/<arch>/<lc>/<cs>/<ent>
```

Each component is short on purpose so consumers can match against pieces (`mcfp1/static/cs/<hex>` to assert "binary signed with this exact codesign posture") rather than committing to the full string.

**`MCFPStatic.fingerprint(path:)`.** Pure function returning a `MCFPStaticResult`. Uses:

- Direct Mach-O magic-number reads for `arch` (no subprocess).
- `otool -L` for `lc` (LC_LOAD_DYLIB names joined by newline, SHA-256 12-hex prefix).
- `CodeSigningCache` (existing MacCrabCore actor) + `codesign -d -r-` for `cs` (`<team_id>|<flags>|sha256|<designated_requirement>` joined, SHA-256 12-hex prefix).
- `codesign -d --entitlements :-` parsed via PropertyListSerialization for `ent` (sorted keys joined by newline, SHA-256 12-hex prefix). Values intentionally dropped — see spec rationale.

**`maccrabctl fingerprint <pid-or-path> [--json]`.** CLI surface per plan §6.4 R1 acceptance. Verified end-to-end:

```
$ maccrabctl fingerprint /usr/bin/true
mcfp1/static/universal/238807fd6f9f/e0c0c66ca8f5/e3b0c44298fc
```

PID resolution via `proc_pidpath` so the operator can fingerprint a running process directly.

## R1 stability evidence

`MCFPStaticTests` includes the explicit same-binary stability assertion (plan §6.4 R1 kill criterion): two consecutive `fingerprint(path: "/usr/bin/true")` calls produce byte-identical results. The cross-reboot stability target (≥95%) is collected via a nightly corpus job to `docs/mcfp-research/corpus.jsonl` rather than the unit suite.

## ArtifactStore parallel-init lock

Side-effect of v1.14 work: under heavier parallel-test load (more SQLCipher connections being opened concurrently), we observed sporadic `SQLITE_MISUSE` on `prepare` after a fresh `sqlite3_open_v2` + `PRAGMA key` window. Added a process-wide `NSLock` serializing just the open + key + initial schema-migration window. Once a connection is fully opened, `SQLITE_OPEN_FULLMUTEX` handles per-connection thread safety as before. The lock is invisible to call sites; it only affects the speed of constructing new ArtifactStores in parallel (and the cost is dominated by the SQLite open itself anyway).

- `MCFPStaticTests` — 9 cases covering canonical-format shape, arch enumeration, component lengths, same-binary stability across two runs, different-binary divergence, missing-file error, hex12 helper determinism + length, empty-string hash matching the well-known SHA-256 prefix.
