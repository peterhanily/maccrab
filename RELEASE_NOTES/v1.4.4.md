## MacCrab v1.4.4 — Fix "database is locked" banner

Same-day hotfix for v1.4.3. The new fail-loud storage-error banner was
working correctly — but the underlying SQLite configuration was too
fragile. A routine WAL checkpoint briefly holds the DB write lock;
without a `busy_timeout` set, the next insert returned
`SQLITE_BUSY` immediately, surfaced as a "Detection data is not being
saved" red banner to the user. One transient lock → one scary banner.

### Fixed

- **`PRAGMA busy_timeout = 5000`** added to `EventStore`,
  `AlertStore`, and `CampaignStore`. SQLite will now retry a
  busy-lock for up to 5 seconds before returning `SQLITE_BUSY`,
  which is more than enough to wait out any WAL checkpoint. This
  is the standard fix for this error class — before v1.4.4 the
  stores were using the default `busy_timeout = 0`.
- **Storage-error banner threshold tuned.** v1.4.3's banner fired
  on a single transient error. v1.4.4 requires **≥5 write
  failures AND the most recent within 2 minutes** before showing
  the red banner. Single transients still get logged (visible via
  `sudo log show`) but don't nag. When the root cause is actually
  persistent — full disk, permissions, DB corruption — the
  banner still fires, because those will trip the threshold
  quickly.

---

Backwards-compatible. No schema changes. If you were seeing the
"Detection data is not being saved" banner on v1.4.3, it will stop
appearing on v1.4.4 unless there's a real persistent issue.
