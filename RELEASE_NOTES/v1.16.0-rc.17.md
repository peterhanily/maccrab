# v1.16.0-rc.17 — Per-invocation plugin inputs threaded

Hardening chapter starts. The `inputs` field declared on every plugin manifest since v1.13a-1.3 finally connects through to the plugin runtime.

## What landed

**`CaseContext.inputs: PluginInvocationInputs`** — new field on the per-invocation context. PluginRunner.runCollector / runEnricher / runAnalyzer all populate it from the `inputs:` parameter they already accept.

**Three plugins now honor their manifest inputs:**

| Plugin | Input | Effect |
|---|---|---|
| `macho-analyzer` | `--path=/full/path` | Analyzes the supplied binary; falls back to dogfood default set when omitted. |
| `tcc-lite` | `--includeDeniedGrants=true\|false` | Filters denied access rows. Default true. |
| `launchd-lite` | `--includeSystemBaseline=true\|false` | Walks `/System/Library/LaunchAgents` + `LaunchDaemons` when true (the Apple baseline). Default false (volume + signing posture make it mostly noise). |

**`maccrabctl plugin run` accepts generic `--<key>=<value>` and `--<key> <value>` flags.** Each is parsed into `InputValue` (bool / integer / string heuristic) and threaded through to the plugin via `caseContext.inputs.values[key]`.

End-to-end verification:
```
$ maccrabctl plugin run com.maccrab.forensics.macho-analyzer \
    --case <id> --path=/bin/ls
   Artifacts committed:  1
   Notes:
     - Mach-O analyzer: 1 binaries analyzed (operator-supplied path).
```
