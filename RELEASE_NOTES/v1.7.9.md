# MacCrab v1.7.9 — Memory hot-fix (round 2) + observability + 3 new rules

The biggest patch release in the v1.7 line, but every change is
patch-scope: no architecture changes, no new training data, no
redesigns. Bundle: a memory leak that v1.7.7 didn't fully fix, two
new audit passes that codify the lessons, three new detection rules,
heartbeat counter fix, brew ↔ Sparkle drift prevention, and the
final cleanup of two cosmetic UI items reported in v1.7.8.

## What was broken

A v1.7.8 install on a busy machine climbed to 2.07 GB private heap
over 9 hours. Heap dump showed **135,689 × 16 KB NSConcreteData
buffers** dominating — same Foundation-autorelease leak class as
v1.7.7 but at the **per-CHUNK** read level rather than per-LINE.
v1.7.7's fix wrapped the inner per-line body but missed the outer
loop's `fileHandle.availableData` Data accumulation.

## Fixes shipped

### Memory: outer-loop autoreleasepool wrap

`EsloggerCollector.readLoop`, `UnifiedLogCollector` (per-chunk),
`KdebugCollector` (full wrap), and `FileHasher.computeSHA256`
(per-chunk file hashing) now drain Foundation autoreleased objects
every iteration. Inner pools retained as belt-and-suspenders for
peak memory on chunks containing many lines.

`FileHasher` is the likely dominant source on field machines: every
file event that triggers IOC hash matching reads the file in 64 KB
chunks via `handle.read(upToCount:)`, returning autoreleased Data
per chunk. Without pool drainage that's GBs of accumulated buffers
per day on a busy file-event machine.

### Audit codification (new Pass 9 + Pass 10)

`scripts/pre-release-audit.sh`:

- **Pass 9** — autoreleasing Foundation calls inside `while`/`for
  await` bodies without a surrounding `autoreleasepool` block fail
  the audit. Catches `JSONSerialization`, `NSRegularExpression`,
  `DateFormatter`, `ISO8601DateFormatter`, `availableData`,
  `readDataOfLength`, `read(upToCount:)`, `Data(contentsOf:)`. Found
  2 sites the v1.7.7 fix had missed.

- **Pass 10** — when ≥ 2 stores share a `.db` file, each store's
  migrations must be runnable independent of the global
  `user_version` counter. Codifies the v1.7.6 SchemaMigrator fix.

### Heartbeat per-collector counters

`event_count: 0` for every primary collector while
`events_processed: 1M+` was telemetry that operators couldn't trust.
Pre-fix only secondary collectors (with their own MonitorTask
`for await` loop) called `recordTick`. Fixed by attributing each
event in `EventLoop` to a representative primary collector by
`event_category`.

### Three new detection rules

- **`persistence/xpc_service_replacement.yml`** — writes to
  `/Library/LaunchDaemons/*.plist` or `/Library/LaunchAgents/*.plist`
  by unsigned/ad-hoc-signed processes. Severity: high.
- **`defense_evasion/network_extension_unsigned.yml`** — unsigned
  `NEPacketTunnelProvider`/`NEDNSProxyProvider`/`NEAppProxyProvider`/
  `NEFilterDataProvider` installs. These can intercept every
  packet on the device. Severity: critical.
- **`credential_access/crypto_wallet_data_access.yml`** — reads of
  native crypto wallet directories (Electrum, Exodus, Atomic, etc.)
  AND browser-extension wallet storage (MetaMask, Phantom, Coinbase,
  Trust, Binance) by anything other than the wallet app or browser.
  Targets Atomic Stealer (AMOS), Banshee, similar macOS infostealers.
  Severity: critical.

The original c2_beacon_pattern timing-variance rewrite was descoped:
Sigma's sequence engine doesn't support "step A repeated N times in
a window" (only "step A then step B"). The crypto-wallet rule fills
a more pressing coverage gap.

### `maccrabctl repair --fix-storage` schema sanity

Adds `PRAGMA table_info(alerts)` column-presence check to
fingerprint the v1.7.5 → v1.7.6 bug shape (alerts table missing
`llm_investigation_json`). `PRAGMA integrity_check` passes for that
state because the file IS valid SQLite — it's just schema-stale.
The new check recommends installing v1.7.6+ instead of a destructive
backup that would lose alert history.

### `ESCollector` defensive autoreleasepool

Per-event ES kernel callback wrapped defensively. Pass 9 doesn't
flag it (no `while let` shape) but the discipline applies.

### brew ↔ Sparkle drift detection

When `Bundle.main.bundleURL.path` contains `/Caskroom/`:
- Sparkle's `automaticallyChecksForUpdates = false`
- Settings → AI Backend shows a caption: "Installed via Homebrew.
  Background auto-update is off; upgrade with `brew upgrade --cask
  maccrab`."

Stops the v1.6.13 → v1.7.5 channel-drift incident from recurring.

### Metrics export schema 2

`/var/tmp/maccrab.metrics.json`:
- Added `resident_memory_mb` via `mach_task_basic_info` (no sudo)
- Added `events_dropped_total` and `events_per_sec_lifetime`
- Schema 1 → 2

The continuous RSS visibility specifically motivated by the 3-iteration
leak chase v1.7.6 → v1.7.7 → v1.7.9: catch the next leak shape at
100 MB instead of 1 GB.

### UX cleanup (carries forward from staged v1.7.8 hold)

- **Zombie-sysext banner removed entirely.** Showed on every
  dashboard launch with no productive dashboard-side action available.
  Same diagnostic still surfaces via `maccrabctl repair`, which is
  the right surface (it can both diagnose AND walk through next steps).
- **Sidebar layout reworked.** `.navigationSplitViewStyle(.prominentDetail)`
  + `frame(minWidth: 1100)` enforces a generous window minimum so
  narrow-window overlay behaviour can't trigger. Mail.app / Calendar.app
  pattern.

## How to install

### Manual / verification

[github.com/peterhanily/maccrab/releases/v1.7.9](https://github.com/peterhanily/maccrab/releases/tag/v1.7.9)

### Sparkle auto-update (NOT yet active for v1.7.9)

The Sparkle appcast publish is held intentionally pending field
verification on a second test machine. Once that confirms the
memory fix at scale, the appcast is published and existing
v1.7.8 users get the update banner on next launch.

### Homebrew (NOT yet active for v1.7.9)

Same hold reason — tap formula not yet bumped. After verification,
`brew upgrade --cask maccrab` will resolve to v1.7.9.

## Compatibility

No data migration. No reboot. No extension re-approval. Same install
path as v1.7.8 with the new sysext binary picking up on next launch.

## Expected steady-state numbers

For ~107 file events/sec sustained:
- Daemon RSS: **120–250 MB**, oscillating with file-cache pressure
  (NOT growing monotonically — that's the leak fingerprint)
- Private memory: **~80–150 MB** (Swift heap + actor state)
- Shared memory: **~30–60 MB** (events.db mmap window + frameworks)
- `/var/tmp/maccrab.metrics.json` `resident_memory_mb` should track
  the same RSS curve over time, surfacing any future leak before
  it reaches 1 GB

Tests: 929 passing in 189 suites. Pre-release audit: 10/10 passes green.
