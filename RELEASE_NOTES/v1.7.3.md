## MacCrab v1.7.3 — Memory regression hotfix

The v1.6.22 perf reduction (2.76 GB → 50 MB resident) regressed back
to 2.31 GB on a v1.7.2 test host. This release restores the cap.

### What was leaking

A v1.7.x audit identified three causes:

1. **Detached snapshot Tasks in the heartbeat (v1.7.1).** The
   heartbeat-every-30s body was wrapped in `Task { ... }` to allow
   `await` across actor isolation, then spawned 4 nested
   fire-and-forget `Task { ... }` calls for snapshot writers
   (lineage, MCP baseline, rule telemetry, TCC). When any snapshot
   write stalled — slow disk, contention, an actor busy from a
   prior call — the next 30 s tick spawned 4 more. Tasks
   accumulated, each captured `state: DaemonState` strongly,
   memory grew unbounded.

2. **`CollectorRegistry.entries` dictionary uncapped (v1.7.2).**
   `recordTick(name:)` lazy-registers unknown names with no upper
   bound. Any code path emitting variant names (PIDs, paths,
   timestamps embedded in collector names) would grow the
   dictionary indefinitely.

3. **`MCPAttributor.cache` non-deterministic eviction (v1.7.0).**
   `cache.keys.first` removed an arbitrary entry on overflow —
   not LRU. Frequently-accessed entries got evicted; stale
   negative-cache entries persisted; re-walk pressure grew.

### Fixes

- **Heartbeat overlap guard.** New `HeartbeatInFlight` class wraps
  an NSLock + Bool. `tryAcquire()` returns false if a previous tick
  is still running, dropping the new tick. The 4 snapshot writes
  are now serialised inside the outer Task via plain `await`. Hard
  cap: 1 outstanding heartbeat Task at any time. Stalled snapshots
  back-pressure cleanly with one `WARN` log per dropped tick.

- **CollectorRegistry cap (default 64).** `init(maxEntries:)`
  parameter floored at 16 to prevent operator misconfiguration.
  When `recordTick` lazy-registers and cap is reached, evict
  oldest-by-lastTick (preferring never-ticked entries). One `WARN`
  log per eviction surfaces the misregistration to operators.

- **MCPAttributor LRU eviction.** New `accessSeq: [pid_t: UInt64]`
  parallel map bumped on every cache hit/miss. Eviction picks the
  entry with the lowest seq. Same pattern as the v1.6.21
  RuleEngine.regexAccessSeq.

### Audit codification — Pass 8

`pre-release-audit.sh` now scans every `private var <field>: [...]`
or `private var <field>: Set<...>` declaration in actor source
files in `MacCrabCore` + `MacCrabAgentKit`. Each must have either:

- explicit cap-and-evict logic in the same file
  (`removeValue(forKey:)`, `removeFirst`, `removeAll`, `count >=`),
- an inline `// bounded:` comment, or
- an entry in the audit's `BOUNDED_FIELD_ALLOWLIST` with rationale.

A v1.7.3 baseline allowlist documents 21 pre-existing fields whose
bounds are external (USB devices, installed plugins, configured
servers, hardware count). New unbounded actor maps trigger the
audit and fail the release pipeline. Catches the v1.7.2 regression
class going forward.

### Tests

922 in 187 suites pass (was 918). +4 net (`V173HotfixTests.swift`):
- CollectorRegistry cap evicts oldest-tick entry
- CollectorRegistry prefers never-ticked entries for eviction
- CollectorRegistry cap floor (maxEntries < 16 is clamped)
- MCPAttributor LRU eviction returns nil on re-walk after eviction

### Notes for operators

- Memory should return to v1.6.22 baseline (≤100 MB resident on a
  fresh-DB system) immediately after upgrade.
- `events_dropped` heartbeat counter and any "Skipping heartbeat
  tick" log lines are diagnostic signals — investigate if they
  spike (indicates disk I/O backpressure or a stuck actor).
- The audit baseline allowlist (21 entries) is documented in
  `scripts/pre-release-audit.sh`. Fields tagged "follow-up in
  v1.7.4" should get explicit caps in the next release.

### Roadmap

- **v1.7.4 audit follow-ups:** explicit caps for `quarantineAlerted`,
  `revocationHistory`, `alertedSleepProcesses`, `cachedResults`,
  `profiles` — each currently allowlisted as "operator-controlled
  bounds" but would benefit from explicit caps for defence-in-depth.
- Field reports on the new heartbeat behaviour. If "skipping
  heartbeat tick" logs ever appear, the snapshot serialisation may
  need profiling.
