## MacCrab v1.4.7 — triggering-event detail + three more FPs

Field feedback after v1.4.6 landed surfaced two new FP classes
and one recurring UX gap: alerts showed the rule title but not
the actual command that fired it.

### Added

- **Triggering Event in the alert detail panel.** Selecting an
  alert now fetches the originating `Event` by id and renders a
  new "Triggering Event" GroupBox with command line, PID, signer
  (plus team id), parent process, ancestor chain, file path /
  source path (for file events), destination hostname:port
  (for network events), and TCC service + allowed (for TCC
  events). Alerts without a backing Event (USB / clipboard /
  tamper) gracefully omit the box.

### Fixed

- **Tamper detection no longer fires on Apple's own sysext
  activation.** `SelfDefense` was treating `attrib` events on
  the binary path at HIGH severity. Every install, upgrade, and
  sysextd reactivation stamps xattrs on the sysext executable
  (`/Library/SystemExtensions/<uuid>/com.maccrab.agent.*`) and
  fired "Config Modified" self-alerts. `attrib` events on
  non-critical monitored paths (the binary, events.db, config
  files) are now silently ignored — only the LaunchDaemon plist
  and compiled-rules directory still escalate on attrib
  changes.

- **`plist_written_to_library.yml` no longer fires on browser
  helpers.** Google Chrome running from `/var/folders/…` (its
  own per-user temp) writing to `~/Library/Google/Chrome/…`
  matched the "plist in Library from temp" pattern. Fix:
  broadened `filter_apple_signed` to `apple | appStore | devId`
  so Developer-ID-signed browsers pass, and added
  `filter_browser_helper` covering Chrome, Chromium, Firefox,
  Safari, Edge, Arc, Brave, Opera, Vivaldi.

Existing alerts in the database from pre-1.4.7 runs will stay
visible — these fixes gate new firings only. Retention pruning
(default 30 days) will age out the stale entries, or use the
Alerts → Suppress control to hide them immediately.
