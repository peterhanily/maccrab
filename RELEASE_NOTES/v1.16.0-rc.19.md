# v1.16.0-rc.19 — ConsentManager wired into PluginRunner

The policy layer shipped at v1.15.0-rc.2 finally fires on every Collector invocation.

## What landed

**`PluginRunner` constructor accepts a `ConsentManager`.** Default is `LoggingConsentManager()` — same defaults as before. Production paths (maccrab-mcp on `mcpFromAgent`, scheduled-run daemon when it lands) construct with their own consent manager.

**`PluginRunner.runCollector` now takes `mode: ConsentMode = .interactive` and `agentName: String? = nil`** parameters. Before invoking the plugin, calls `consent.decide(_:)` and:
- `.granted` / `.autoApproved` → proceeds.
- `.denied(reason)` → throws `PluginRunnerError.runtimeError` with the reason embedded.

The reason for MCP-from-agent denials includes the operator-runnable `maccrabctl case allow-ai --content <id>` command (plan §10.8). The MCP layer can render the §10.8 structured-error response directly from it.

**Highest-emitted privacy class computed from manifest outputs.** For plugins like `imessage-bodies` / `mail-bodies` / `safari-deep` / `applescript-runtime` (content+ class), MCP-from-agent calls without `ai_content_allowed=1` deny correctly.

## What's deferred

- `PluginRunner.runEnricher` / `runAnalyzer` consent gating — same shape can land additively; this RC focuses on the Collector path (the volume case).
- UNUserNotificationCenter-backed production ConsentManager — needs `.app` bundle context (MacCrabApp or future `maccrab-forensicsd.app`).
