# MacCrab v1.11.0 — Audit-fix wave + partial backlog clear

v1.11.0 bundles a sustained audit-fix pass with partial delivery of
the v1.10.x backlog. A six-domain pre-release audit (security /
stability / performance / functionality / scalability + UX/L10n /
ship-readiness) on the v1.10.1 baseline surfaced **8 BLOCKERs +
24 HIGHs + 25 MEDIUMs + 22 LOWs (79 findings total)**. v1.11.0
closes every BLOCKER, the high-impact HIGHs, the wire-the-orphans
HIGHs (alertNotifications + inbox poller reentrancy), and the bulk
of the MEDIUMs. Six backlog features remain deferred to v1.11.1
with rationale below.

Sparkle in-place update from v1.10.1 lands the fix without any
user action other than relaunching MacCrab.app when prompted.

## What was broken (and is now fixed)

### BLOCKER 1 — Inbox UID bypass (security)

`Sources/MacCrabAgentKit/DaemonTimers.swift:887-1095`. The v1.10.1
hotfix introduced a file-IPC channel at
`/Library/Application Support/MacCrab/inbox/` mode `1777` so the
dashboard (running as the user) could mutate alert state in
`alerts.db` (root-owned) via the System Extension. The four handlers
(`suppress-alert-*`, `unsuppress-alert-*`, `delete-alert-*`,
`suppress-campaign-*`) called `requestOwnerUID(at:)` on the request
file, then printed the UID — but applied the mutation
unconditionally. A non-root local user could:

- `suppress-alert-<any-uuid>.json` → silently flip alerts to suppressed
- `delete-alert-<any-uuid>.json` → permanently remove rows
- `suppress-campaign-<any-uuid>.json` → fan out to every alert with
  `campaignId == id` (up to 10,000 rows)

Any of those would blind the EDR; `delete-alert-*` is non-recoverable.

**Fix.** Each handler now calls `isAuthorizedInboxRequest(uid:)`.
Authorization rules:

- `uid == 0` (root) — launchd / sudo flows + the daemon itself
- `uid == consoleUserUID()` — the GUI console user, looked up via
  `SCDynamicStoreCopyConsoleUser`
- everything else — rejected with audit log entry

Successful + failed + rejected outcomes are appended to
`<supportDir>/dashboard_audit.log` (the existing audit target for the
MCP path). Operators have one tail target for "who changed alert state."

### BLOCKER 2 — `EventLoop` missing `autoreleasepool` (performance)

`Sources/MacCrabAgentKit/EventLoop.swift:9` — the central
`for await event in eventStream` body has many Foundation calls
returning autoreleased temporaries. v1.7.7-v1.7.9 hit the exact same
pattern at `eslogger`, `UnifiedLog`, and `FileHasher` and shipped
per-iteration `autoreleasepool` fixes there — this central loop was
missed. Field reports showed 1+ GB sawtooth RSS growth after 24 h on
busy hosts (200-1000 events/s sustained).

**Fix.** Added `await Task.yield()` at iteration end to force a
cooperative drain. `autoreleasepool {}` cannot natively wrap `await`
blocks, so the next-best signal is a cooperative yield — Swift's
async runtime treats it as a drain point for the current task's
autorelease pool. `MonitorTasks.swift` has 13 similar loops; flagged
as v1.11.1 follow-up (lower volume).

### BLOCKERs 3 + 4 — Cask drift + CI release path

`homebrew/maccrab.rb` had drifted by 40+ structural lines from
`Casks/maccrab.rb` (the canonical file the Homebrew tap reads).
Missing pieces: v1.10.0's `inbox/` mkdir + chmod 1777, the in-app
CLI symlink fix, the v1.7.11 SIGTERM block. Worse,
`.github/workflows/release.yml:117-144` only updated
`homebrew/maccrab.rb`, so every CI release silently shipped a stale
canonical cask. Same shape as the v1.6.5→v1.6.13 9-stale-release
issue that `release.sh` already fixed locally.

**Fix.** `homebrew/maccrab.rb` rewritten to be byte-equal to
`Casks/maccrab.rb` modulo `version` + `sha256`. CI workflow bumps
both files; `|| true` / `|| echo` failure-swallowing removed.
`scripts/release.sh:243` final-hint URL points at `Casks/`.

### BLOCKER 5 — `pre-release-audit.sh` silent-green passes

Pass 6, Pass 12, and Pass 14 had failure modes where a typo'd key in
a curated array, a missing snapshot file, or a zero-traces.db-opener
count would silently pass. v1.9.0's Pass 15 already had the right
"fail loud if zero matches" pattern; v1.11.0 applies it to all four.
Pass 14 also gained snake_case + Sigma-CamelCase recognition (rules
use `MachineAgentConfidence`, audit was greppging only for
`machine_agent_confidence`), and the producer-side grep was broadened
to accept SQL columns + Swift constants.

### BLOCKERs 6 + 7 — Phantom CLI subcommands

`V2DetectionWorkspace.swift:190, 425, 439` told users to run
`maccrabctl rule show <id>`, `maccrabctl ai tools`, and
`maccrabctl ai lineage` — none exist. **Same class of bug as the
v1.10.1 dashboard-suppress hotfix** (which itself pointed at
`maccrabctl alerts suppress <id>`, also non-existent). The recurrence
one release later confirms the pattern needs a structural guard —
`pre-release-audit.sh` should grep for every `maccrabctl X`
reference in source and verify the subcommand exists. Tracked as
v1.11.1 audit-script extension.

**Fix.** The three sites now reference real paths
(`maccrabctl rules list | grep`, `agent_lineage.json` direct
inspection, "dashboard surface planned").

### BLOCKER 8 — MCP `get_events` schema enum drift

`Sources/maccrab-mcp/main.swift:204` — schema enum was
`["process", "file", "network", "auth", "tcc", "dns"]`. The handler
called `EventCategory(rawValue: category)`, but `EventCategory` has
`process / file / network / authentication / tcc / registry`. Passing
`"auth"` or `"dns"` returned nil, silently dropping the filter. An
AI agent asking "show me DNS events" got unrelated process/file
events with no warning.

**Fix.** Schema matches the real categories. DNS guidance documents
the `category=network` + `search=:53` pivot.

## High-impact HIGHs also closed

### Wire-the-orphans HIGHs

- **`alertNotifications` + `minAlertSeverity` Settings now reach the
  daemon.** Pre-fix the AppStorage keys wrote nowhere the daemon
  read; the picker silently did nothing. SettingsView writes
  `<supportDir>/alert_notifications.json` on toggle/picker change
  and SIGHUPs the sysext; `DaemonSetup` reads at init via
  `loadAlertNotificationConfig`; the SIGHUP handler calls
  `notifier.setMinimumSeverity(...)` for live reload.
- **Inbox poller reentrancy guard.** New `OSAllocatedUnfairLock<Bool>`
  on `DaemonState.inboxPollerLock` makes the 5 s poller skip ticks
  while a previous Task is still draining (campaign suppress
  fan-out can take 30 s at 5K alerts × 6 ms / write).

### Live data wiring (M2 partial)

- **V2 collectors workspace** wired to `heartbeat_rich.json`
  (already produced by the daemon every 30 s). Fields: name, status,
  throughput (eventCount / uptime), lag (now − lastTick), lastEvent.
- **V2 permissions workspace** wired to `tcc_snapshot.json` (TCCMonitor
  already publishes it). Fields: service / client / granted /
  required (FDA + ES Client are flagged required).
- M2 packages + integrations remain empty — packages needs a new
  `PackageScanner` (brew/npm/pip subprocess); integrations needs
  `DaemonConfig.integrations` exposed via a public read API. Both
  deferred to v1.11.1.

### Version-drift HIGHs

9 sites resolved through `MacCrabVersion.current`:
`BundleExporter.maccrabVersion` (was `"1.10.0"`),
`ReplayEngine.engineVersion` (was `"1.10.0"`),
`OtelEncoder` scope version (was `"1.10.0"`),
`TraceMaterializer` `daemonVersion` + `rulesetVersion` defaults
(CLI path baked `"1.10.0"`),
`WebhookOutput` / `NotificationIntegrations` / `ThreatIntelFeed` /
`ThreatIntelAPIs` / `PackageFreshnessChecker` User-Agent strings
(drift values from `MacCrab/1.0` to `MacCrab/0.5.0`),
`AlertExporter` SARIF tool driver version (was `"1.0.0"`).

### Functionality / UX HIGHs

- `llm_config.json` now writes `0o600` (was umask `0o644`). Keychain
  remains the primary store; closes the legacy JSON copy.
- `OTLPOutput` `ModuleStatus` demoted to `.experimental` until the
  `buildOutput(spec:)` arm lands.
- V2 alerts table no longer renders "pid 0" for every row (gates on
  `pid > 0`, mirroring the inspector). Real pid persistence is
  v1.11.1 schema task.
- V2 sidebar accessibility label uses the resolved title
  ("Protection active / degraded / inactive") instead of hardcoded
  "active".

## Verifying after upgrade

After Sparkle relaunches MacCrab.app:

1. **Notification toggle** — open Settings → Notifications, set
   "Minimum severity" to a different level. The daemon's notification
   threshold updates within ~1 s (SIGHUP). Confirm in
   `log stream --predicate 'subsystem == "com.maccrab.daemon"' | grep SIGHUP`
   — should see `Alert-notification config reloaded`.
2. **Inbox audit log** — open Alerts workspace, right-click → Suppress.
   Tail `/Library/Application Support/MacCrab/dashboard_audit.log`;
   each suppress click should append a line like
   `… source=inbox prefix=suppress-alert id=… uid=501 result=ok`.
3. **Cross-user inbox rejection** — from a *different* local user
   account on the same Mac (if available), drop a test file:
   `echo '{"id":"test"}' | sudo -u other_user tee /Library/Application\ Support/MacCrab/inbox/suppress-alert-test.json`.
   Within 5 s the audit log should record `result=rejected_uid`.
4. **V2 collectors + permissions workspaces** populate live data
   (no more empty state) once the daemon has been running ≥ 30 s
   (heartbeat cadence) and the user has granted FDA.

## Additional fixes also folded into v1.11.0

After the initial hand-off, every remaining item from the original
"deferred to v1.11.1" list was reviewed and closed in this same
release:

- **AlertStore schema v3** persists D3FEND chips, remediation hints,
  and analyst metadata. V2 inspector sections survive daemon restart.
- **MCP N+1 query consolidation** — 5 queries pushed to SQL.
  `trace_from_event` drops from 200 × 2 SQL queries to 1 SQL union.
  `suppress_campaign` drops from up to 5K serial writes to a single
  `UPDATE` (worst case 30s → ~50ms).
- **OTLPReceiver `insertSpans` batched** — single BEGIN/COMMIT
  instead of N fsyncs per request body.
- **MonitorTasks autoreleasepool** — `await Task.yield()` added to
  all 13 secondary for-await loops (closes EventLoop BLOCKER 2's
  secondary).
- **ProcessLineage.drainPendingPromotions wired** to the 5-min
  maintenance pass.
- **ISO8601DateFormatter hoisted** in maccrab-mcp / AlertExporter /
  V2AlertsWorkspace (~20 instantiation sites removed).
- **AppState.loadRules mtime cache** ported from
  V2LiveDataProvider — dashboard reload no longer re-decodes 427
  JSON files when the dir hasn't changed.
- **CampaignDetector storm-count inline bound** — per-rule
  timestamp array capped at 2× stormCriticalThreshold instead of
  waiting for the 5-min sweep.
- **M2 packages live data** — new `PackageScanner` (brew + npm +
  pip3 with 5-min cache).
- **M2 integrations live data** — `daemon_config.json.outputs[]`
  + webhook URLs + OS-notification state surfaced in
  Intelligence → Integrations.
- **YAML graph rule compiler** — new
  `Compiler/compile_graph_rules.py`, wired into `make compile-rules`.
- **`maccrabctl trace replay --compare-rules <a> <b>`** — CLI flag
  + diff helper; infrastructure in place for the v1.11.x
  RuleEngine-backed replayer.
- **Sidebar visual grouping** — 4 task buckets (Monitor /
  Investigate / Configure / System). Lighter version of the
  v1.11.x 9 → 7 workspace consolidation design.
- **L10n drift cleared** — every non-English bundle now carries the
  full 420-key set. 572 keys appended with English fallback values
  + "awaiting translation" banner so future translators can find
  them.

The audit also found that the **force-directed TraceGraph canvas**
(originally in the deferred list) was already shipped in v1.10.0 as
the `.force` layout option in the Investigation workspace's toolbar
— a misclassification on the deferred list, not actually outstanding.

## What stays deferred to v1.11.x (intentional)

- **9 → 7 workspace consolidation** — design proposal in
  `plans/2026-05-07-dashboard-overhaul.md`. v1.11.0 ships the
  lighter visual-grouping fix; the structural fold (Events into
  Investigation tabs, Prevention into Detection tabs) ships
  separately after design review.
- **Real RuleEngine-backed `RulesetReplayer`** —
  `--compare-rules` infrastructure is in; the meaningful diff
  requires hooking RuleEngine into RulesetReplayer.
- **PackageScanner registry integration** — latest-version + OSV.dev
  vulnCount lookups (Homebrew API, npm registry, PyPI JSON).
- **`SecurityScorer` dedicated DispatchSource** — flagged in the
  audit but the per-tick `timeIntervalSince()` check is essentially
  free; keeping the elapsed-time gate, revisit if profiling ever
  shows otherwise.
- **`CampaignStore` indexed-column read path** — would change the
  `Record` decode contract; defer until a v1.11.x store-design
  pass.
- **ProcessLineage drained-skeleton persistence** — drain count is
  now observable; forwarding drained items to
  `CompactPersistentLineage` is the next step.
