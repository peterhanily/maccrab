## MacCrab v1.6.18 — Custom-import validators + FP fix on own daemon + version drift caught

A small follow-up to v1.6.17 closing three loose ends:

### 1. False positive on own daemon (the install-time alert)

v1.6.17 introduced a "Refresh Now" button on the Threat Intel browser
that sends `SIGUSR1` to the daemon (sysext + `maccrabd` fallback) via
`pkill -USR1`. On install, this triggered the
`security_tool_killed` rule which had `maccrabd` in its target list
and didn't filter non-fatal signals — so MacCrab alerted on its own
IPC.

**Fix:** updated two rules to handle this correctly going forward:

- `Rules/defense_evasion/security_tool_killed.yml`: removed `maccrabd`
  from `selection_target` (it's our own component, not a third-party
  tool worth flagging when killed). Added a `filter_nonfatal_signal`
  selection that excludes `-HUP`, `-USR1`, `-USR2`, `SIGHUP`,
  `SIGUSR1`, `SIGUSR2` — these are control signals, not termination.
  Added a `filter_maccrab_self` selection that excludes any pkill
  whose command line names our own daemons (`com.maccrab.agent`,
  `maccrabd`).
- `Rules/sequences/defense_evasion_kill_persist.yml`: same shape —
  removed `maccrabd` from the target list, added the non-fatal-signal
  filter to step 1.

Other security tools (Falcon, SentinelOne, Santa, LuLu, BlockBlock,
OverSight, etc.) remain in both rules' target lists — anyone trying
to kill THOSE still triggers as designed.

### 2. Custom-import IOC validation

Pre-v1.6.18 `addCustomIOCs(...)` and `loadCustomFile(path:type:)`
accepted any string. A user pasting "hello world" or "TODO" into the
Threat Intel → Import tab silently inserted those as IOCs. The MISP
sync, STIX bundle import, and bundled IOC paths shared the same
problem.

**Fix:** five new public validators on `ThreatIntelFeed` —
`validateHash`, `validateIP`, `validateDomain`, `validateURL` plus
the import return type `ImportResult { accepted, rejected }`.
- Hashes must be 32, 40, or 64 lowercase hex characters (MD5, SHA-1,
  SHA-256).
- IPs must parse as numeric IPv4 (octets 0-255 each) or as IPv6 via
  `getaddrinfo` with `AI_NUMERICHOST` (so a hostname can never
  silently land in the IP set).
- Domains require ≥2 labels with valid label characters, no scheme,
  no path/query, total ≤253 chars. Numeric-TLD rejected (catches
  bare IPv4 mis-categorized as domain).
- URLs must parse and have an `http`/`https`/`ftp` scheme + host.

Both `addCustomIOCs(...)` and `loadCustomFile(path:type:)` now return
an `ImportResult` reporting `accepted` count and the first 25
rejected entries. The dashboard's "Import from Text" flow validates
locally first and shows a user-visible status: "Imported 47 of 50.
Rejected 3 malformed: TODO, hello, ..." instead of silently writing
garbage to disk.

### 3. CFBundleShortVersionString drift caught + locked in

A cross-check showed `Xcode/project.yml` had
`CFBundleShortVersionString: "1.6.4"` stuck since the v1.6.4 release
13 versions ago. The `prerelease-check.sh` only validated
`CFBundleVersion`, so the short-version string drifted unnoticed for
months. Shipping binaries via `swift build` + the build-release
script use the plists directly (which were correct), so no shipped
binary was actually wrong, but anyone regenerating the Xcode project
from `project.yml` would have built with the wrong short version.

**Fix:**
- Both `CFBundleShortVersionString` entries in `Xcode/project.yml`
  bumped to `1.6.18` (and the v1.6.17 catch-up bump landed at the
  same time).
- `scripts/prerelease-check.sh` now validates
  `CFBundleShortVersionString` alongside `CFBundleVersion` — drift
  becomes a hard error at the next release.

### Tests

**807 tests pass (up from 801).** Six new in `ThreatIntelValidatorTests`:

- `validateHash accepts SHA-256, SHA-1, MD5 hex strings`
- `validateHash rejects garbage and wrong-length strings`
- `validateIP accepts valid IPv4 and IPv6, rejects garbage`
- `validateDomain requires a real domain shape`
- `validateURL requires scheme + host`
- `addCustomIOCs reports rejected lines instead of inserting them`

Two existing `cachedStats`/`cachedIOCs` tests updated to use
real-shape SHA-256 fixtures (64 hex chars) so they pass the new
validator.

### Upgrading

Install v1.6.18. Existing custom imports that were valid stay valid;
nothing is migrated or removed. The next time the daemon reloads
(SIGHUP or restart), it re-reads the `custom_*.txt` files in the
support dir through the new validator — any garbage that pre-v1.6.18
let through gets dropped. Operators see "Loaded N custom hashes from
… (rejected K malformed)" in the daemon log.

If you saw the install-time "Security Tool Process Terminated"
alert during v1.6.17 install, you can suppress that one alert in
the dashboard — v1.6.18 won't re-fire it.
