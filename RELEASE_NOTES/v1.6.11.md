## MacCrab v1.6.11 — Pre-publication hygiene

A small release that closes out three findings from a pre-publication
security sweep. No detection-engine changes, no behavioural changes
for end users — just the release pipeline and the CI posture being
honest about what they do.

### 1. `MacCrab-v*.pkg` dropped

Every release from at least v1.5.0 onward shipped a 1.9 KB `.pkg`
alongside the real 30 MB DMG. The PKG was a valid signed xar archive
with broken innards — `scripts/build-pkg.sh` generated a distribution
XML that referenced `#maccrab.pkg` but `productbuild` actually emits
`maccrab-component-$$.pkg`. The result: anyone who downloaded
`MacCrab-v*.pkg` from the GitHub releases page got an installer that
opened cleanly, showed the welcome screen, and installed nothing.

No one has reported this in 9+ releases — the PKG link sees very
little traffic — but shipping a broken artifact under a name that
implies "installer" is not acceptable for a security product. The
fix is to remove PKG from the release flow entirely:

- `scripts/release.sh` no longer calls `build-pkg.sh`
- `.github/workflows/release.yml` no longer has a Build .pkg
  installer step, and drops `.build/MacCrab-v*.pkg` from the artifact
  upload list
- `scripts/build-pkg.sh` removed from local tree (was never tracked)
- DMG + Homebrew cask are the supported install paths. Both were
  always the recommended route; this just makes it the only route.

### 2. PR workflow permissions hardening

Before today `.github/workflows/build.yml` and `rules.yml` ran on
`pull_request: branches: [main]` with no explicit `permissions:`
block, which means they inherited the default `contents: write` +
`packages: write` scope from the repo. Those jobs only read the
source tree, so there's no functional reason for write access, and
accepting unsolicited fork PRs with write-scope runners is a
well-known attack surface.

Added `permissions: contents: read` at the workflow level to both
files. `release.yml` keeps its tighter scoped `contents: write` on
the release-only tag-push trigger, which is what it actually needs.

### 3. Release scripts checked in, sanitised

The three release scripts — `release.sh`, `build-release.sh`,
`notarize.sh` — were gitignored and lived only on the maintainer's
laptop. CI `release.yml` referenced them but any fresh clone /
contributor fork of the release workflow would fail on missing
scripts.

The scripts never contained real secrets (credentials have always
been sourced from `~/.maccrab-release-env` which stays out of the
repo by design). The only personal reference was an example-usage
string in `release.sh` showing the maintainer's Developer ID format
— replaced with a generic `Your Name (TEAMID)` placeholder before
committing.

`scripts/build-pkg.sh` stays gitignored so a stale local copy can't
silently reintroduce the broken PKG path after this release.

### What didn't change

- Detection engine, rules, AI Suite, dashboard, EventLoop — identical to v1.6.10
- Test count remains 764
- Homebrew cask continues to point at the DMG asset

### Pre-publication context

This release lands alongside the public announcement of MacCrab.
The pre-publication sweep (git history search for credentials,
current-tree scan for API key shapes, fingerprinting-path audit,
release-artifact inspection, CI permission review) came back
clean across 10 categories with these three actionable items.
