## MacCrab v1.6.20 — Response-action safety hardening

A focused follow-up to v1.6.19. The same audit that produced
v1.6.19's safety guards (SafePIDValidator + DNSSinkhole allowlist +
SupplyChainGate ancestry) was extended to the **response action
surface** — the per-rule auto-actions that the Response Actions tab
configures and that `ResponseEngine.execute` runs when an alert
fires. Five issues found, all fixed.

### Quarantine: opt-in by default + system-path safe

- **Quarantine is and remains opt-in.** Built-in defaults are still
  only `notify` + `log` — no rule auto-quarantines unless the
  operator explicitly adds the action via the Response Actions tab.
- **`SafeQuarantinePathValidator` (new).** Refuses to quarantine
  files in:
  - System code: `/System/`, `/Library/Apple/`,
    `/Library/Frameworks/`, `/Library/PrivilegedHelperTools/`,
    `/Library/SystemExtensions/`, `/Library/LaunchDaemons/`,
    `/Library/LaunchAgents/`, `/usr/`, `/sbin/`, `/bin/`
  - Runtime state: `/private/var/db/`, `/private/var/folders/`,
    `/etc/`, `/Library/Application Support/MacCrab/`
  - Per-user data: Mail (`~/Library/Mail/`), Calendar / Contacts /
    Reminders, Keychain, Safari, iCloud (`~/Library/Mobile
    Documents/`), Photos library, Time Machine snapshots
  - The user's own MacCrab support dir
- **Symlink-safe.** Resolves symlinks via `resolvingSymlinksInPath`
  before the prefix check so an attacker who plants `/tmp/foo →
  /System/Library/X` can't trick auto-quarantine into moving the
  system binary.
- **Confirmation gate** (see below): destructive actions default to
  requiring an operator click in the dashboard before firing.
- Wired into both `ResponseEngine.quarantineFile` (auto-quarantine)
  and `ManualResponse.quarantineFile` (operator-initiated). Even a
  dashboard click can't move Mail's envelope-index.

### Network blocks: protected-IP allowlist

`SafeBlockableIP` (new) refuses to PF-block:

- Public DNS: 1.1.1.1 / 1.0.0.1, 8.8.8.8 / 8.8.4.4, 9.9.9.9,
  208.67.222.222, plus IPv6 equivalents
- Apple's range: 17.0.0.0/8 (iCloud, OCSP, App Store, software update)
- Loopback: 127.0.0.0/8, ::1
- Link-local: 169.254.0.0/16
- Multicast: 224.0.0.0/4 (Bonjour, AirPlay)
- Carrier-grade NAT: 100.64.0.0/10
- The current default gateway (resolved via `route -n get default`)

A poisoned threat-intel feed naming any of these as a "block this
destination" target now gets logged and dropped instead of taking
down DNS resolution. Wired into both `ResponseEngine.
blockNetworkDestination` and `ManualResponse.blockDestination`.

### `requireConfirmation` is now actually checked

Pre-v1.6.20 the flag was on `ResponseActionConfig`, decoded from
`actions.json`, and **completely ignored** by `ResponseEngine.
execute`. An operator who set `requireConfirmation: true` for a kill
action — expecting a "click to fire" gate — got instant kill instead.

Now: actions with `requireConfirmation = true` are logged as pending
and not auto-executed. The flag is exposed as a checkbox in the
per-rule editor; destructive actions (kill / quarantine /
blockNetwork) default to `requireConfirmation = true` when added
from the Add Rule Action sheet, with a warning-icon visual marker.
This is wire-the-orphans pattern instance #7 — caught by the audit
script's new Pass 1b extension below.

### `actions.json` user-home overlay + SIGHUP reload

Pre-v1.6.20 the dashboard tried to write `actions.json` to the
system path `/Library/Application Support/MacCrab/actions.json` —
which the user app can't write to without root. `try?` swallowed the
EPERM and the "Saved." banner lied. Same pattern as the v1.6.19
webhook fix.

Now: the dashboard always writes to `~/Library/Application
Support/MacCrab/actions.json`. The daemon's `ResponseEngine.
loadConfig` walks `/Users/*` (with file-ownership validation) and
prefers the most recently modified copy. SIGHUP reloads the config
in-place — no daemon restart required. The save banner now correctly
says "Saved. Daemon reloaded."

### `ManualResponse.killProcess` defense-in-depth

The user-initiated kill path now uses `SafePIDValidator` too. The
app runs unprivileged so kill() would EPERM on system PIDs anyway,
but explicit refusal gives a better error message ("PID X is on the
critical-system protect list") and protects against a future
elevated-privilege flow.

### Pass 1b: Codable Config field consumer audit

`pre-release-audit.sh` extended with a new pass that catches the
`requireConfirmation`-shaped bug class for the future. Walks public
fields on `ResponseActionConfig` and `NotificationIntegrations.
Config`, counts `.fieldName` references across `Sources/` minus
trivial `self.fieldName = fieldName` init self-assignments, and
errors when the count is zero (decoded but never read). Wired into
`release.sh` Step 0b. Adding a new decoded-config field without a
runtime consumer now fails the release pipeline before push.

### Tests

892 in 179 suites pass (was 860). +32 net:
SafeQuarantinePathValidator (17), SafeBlockableIP (15).

### Upgrading

Upgrade in place via Sparkle or `brew upgrade --cask maccrab`. No
config-file migration needed. Operators who configured
`requireConfirmation` pre-v1.6.20 expecting a confirmation gate:
those gates are now functional — re-open Response Actions and verify
the checkbox is set the way you expect for each per-rule action.
