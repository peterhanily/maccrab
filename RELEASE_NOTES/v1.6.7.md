## MacCrab v1.6.7 — Security hardening + AI Suite wiring

Follow-up to v1.6.6. Three distinct threads ship together:

### 1. Credential-audit hardening

An independent audit pass against the v1.6.6 code flagged four
credential-protection gaps. All four are now closed:

- **`LLMSanitizer` token-shape coverage.** Previously the sanitizer
  redacted usernames, hostnames, private IPv4, and emails. It relied
  on `CommandSanitizer`'s flag regex (`--api-key=…`) to catch raw
  keys — which doesn't help when a free-form prompt inlines an
  event description containing a literal `sk-ant-…`. Now the
  sanitizer redacts every vendor's key shape directly, regardless
  of surrounding context: Anthropic (`sk-ant-`), OpenAI (`sk-` /
  `sk-proj-`), Google (`AIza…`, 35 char), AWS (`AKIA`/`ASIA`/
  `AGPA`/etc.), GitHub (`ghp_`/`gho_`/`ghu_`/`ghs_`/`ghr_`/
  `github_pat_`), Slack (`xox[aboprs]-`), and generic `Bearer`
  tokens.

- **IPv6 private-range redaction.** Link-local (`fe80::/10`) and
  unique-local (`fc00::/7`, `fd…`) addresses now redact to
  `[PRIVATE_IPV6]`. Public IPv6 (`2001:db8::`) passes through.

- **Mac `ComputerName` redaction.** Strings like `Peters-MacBook-Pro`
  or `Corp-Ops-iMac` redact to `[COMPUTER_NAME]`. Heuristic is
  scoped to Mac product keywords so ordinary hyphenated prose
  (`multi-layer rules`) isn't touched.

- **`LLMConfig` now masks keys on description/debugDescription.** The
  struct already excluded API keys from `Codable` encoding (good),
  but any accidental `print(config)` or `String(describing: config)`
  would dump them via Mirror-based reflection. `description` and
  `debugDescription` now render keys as `<len=N,first=X,last=Y>` —
  enough for "is my key configured" troubleshooting without the
  key bytes themselves leaving the log.

- **`OllamaBackend` guards against plaintext remote Bearer.** If the
  user configures a remote Ollama endpoint over plain `http://` and
  sets an API key, the backend refuses the request and logs a
  diagnostic. Loopback (localhost/127.*/[::1]) is unaffected.

**22 new tests cover this pass.** Test suite now 741/741.

### 2. AI Suite: EventLoop lineage wiring

The v1.6.6 `AgentLineageService` shipped as an in-memory actor with
no event pipeline wired — you had to call `record` yourself. v1.6.7
hooks it into `EventLoop` so sessions, file I/O, network
connections, and process spawns under an AI-tool ancestor
automatically populate the timeline. Start an AI coding tool, open
the lineage service, and you'll see its activity in real time.

Six DaemonState fields added for the six v1.6.6 services so every
EventLoop, polling timer, and MCP handler can reach them through
the standard `state.*` plumbing. `AlertClusterService`,
`MCPBaselineService`, and `AgentLineageService` are wired with
defaults; `TriageService`, `LLMConsensusService`, and
`AgenticInvestigator` remain optional until their LLM dependencies
are configured (DaemonSetup will populate them when v1.6.8 wires
the Settings toggles).

### 3. New MCP tool: `cluster_alerts`

The first of the v1.6.6 AI Suite services to get an MCP surface.
AI agents running in Claude Code (or any MCP client) can now call

```
cluster_alerts { hours: 24, min_severity: "medium" }
```

to collapse the last N hours of alerts into fingerprinted clusters.
Each cluster carries size, max severity, MITRE tactics union,
first/last seen timestamps, and a stable id so expand/collapse
state is easy to track client-side. The full suite of six services
gets MCP exposure in v1.6.8 — cluster_alerts ships first because
it's stateless over the existing alert DB and brings immediate
triage leverage.
