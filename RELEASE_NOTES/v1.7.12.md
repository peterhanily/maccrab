# MacCrab v1.7.12 — LaunchAgent cleanup hot-fix

Resolves [issue #1](https://github.com/peterhanily/maccrab/issues/1)
reported by [@jjdresselhaus](https://github.com/jjdresselhaus).
**Thank you for the field reproduction.**

## What was broken

After `brew uninstall --cask maccrab`, the launch-at-login
`LaunchAgent` persisted under `~/Library/LaunchAgents/`. macOS's
launchd kept trying to start a now-missing binary on every
subsequent login — invisible to most users but confirmable via
`ls ~/Library/LaunchAgents/ | grep maccrab` after uninstall.

Root cause: two-sided. The cask's `uninstall` stanza only handled
system-level LaunchDaemons (the Endpoint Security sysext + the
legacy `maccrabd` plist), not the user-context LaunchAgent that
`SMAppService.mainApp.register()` creates when the user enables
launch-at-login. **AND** `SMAppService.unregister()` itself doesn't
always delete the underlying `.plist` on some macOS versions, even
though it removes the registration from the system database.

## What v1.7.12 changes

### App side — defensive `.plist` sweep after `SMAppService.unregister()`

`Sources/MacCrabApp/LaunchAtLogin.swift` now sweeps both possible
SMAppService write paths after every `.unregister()` call:

- `~/Library/LaunchAgents/com.maccrab.app.plist` — legacy path
- `~/Library/LaunchAgents/79S425CW99.com.maccrab.app.plist` —
  modern, team-id-prefixed (what most macOS 13+ systems write)

Three triggers for the sweep:

1. User toggles "Launch at Login" off in Settings → Startup
2. App startup `reconcile()` self-heal — if the preference is
   disabled but a stale `.plist` exists, clean it
3. The toggle's "already disabled, skipping" early return — still
   sweeps in case the API status reads disabled but a stale file
   remains on disk from a prior macOS version

### Cask side — already shipped April 30

The cask's `uninstall` stanza was patched on April 30 (cask-only
update, no DMG rebuild) to add user-context LaunchAgent handling,
SIGTERM fallback when the quit AppleEvent doesn't terminate the
menubar app, and explicit deletion of both `.plist` paths.

The two changes close the bug class from both directions: app-side
cleanup when the user disables launch-at-login through Settings,
plus cask-side cleanup when the user runs `brew uninstall` while
still having launch-at-login enabled.

## How to install

### Sparkle auto-update (active)

Open MacCrab.app. The update banner appears within minutes of
appcast publish. Click "Install Update".

### Homebrew

```bash
brew upgrade --cask maccrab
```

### Fresh install

[github.com/peterhanily/maccrab/releases/v1.7.12](https://github.com/peterhanily/maccrab/releases/tag/v1.7.12)

## What you don't need to do

- **No reboot.** Daemon code is unchanged from v1.7.11.
- **No extension re-approval.** Same team-id + bundle-id.
- **No manual cleanup.** Existing installs that previously had
  launch-at-login enabled get the stale `.plist` swept on first
  launch of v1.7.12 via the new startup self-heal.

## Verification

After upgrading, you can confirm cleanly:

```bash
# Stale LaunchAgent files (should always return empty)
ls -la ~/Library/LaunchAgents/ | grep -i maccrab

# Active LaunchAgent registrations (should return empty if you have
# launch-at-login disabled, or only the active registration if enabled)
launchctl list | grep -i maccrab
```
