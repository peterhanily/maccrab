## MacCrab v1.6.15 — Wire the orphans, build the timeline

A four-bundle audit-driven release. Closes the same "intent-vs-reality"
gap that bit v1.6.9 / v1.6.12 / v1.6.14: every shipped feature now has
daemon code consuming or producing it. Adds the headline visibility
feature that makes the "MacCrab sees what your agents actually did"
positioning concrete instead of aspirational.

### Bundle A — Wire the orphans

A deep-dive audit found four declared-but-unconsumed services that had
the exact shape of the v1.6.12 maxDatabaseSizeMB bug — surfaced in UI,
no daemon code reading them.

- **`TriageService`, `LLMConsensusService`, `AgenticInvestigator`** were
  declared in `DaemonState` but had zero callers anywhere in the code
  base. The wiki's "v1.6.10 architectural follow-up" had been pending
  five releases. Moved to `AppState` — outbound HTTPS with vendor API
  keys does not belong at ES-entitlement root privilege when the
  dashboard already owns the LLM config. `TriageService` is now wired
  to `AlertDetailView` as a one-click "Get AI Triage Recommendation"
  action that returns `suppress` / `keep` / `escalate` / `inconclusive`
  with a one-sentence rationale.
- **`threatIntelStats`** was a published `@MainActor` property that no
  code ever wrote to. `ThreatIntelView` rendered "Malicious Hashes: 0,
  IPs: 0, Domains: 0, URLs: 0" forever, the exact bug shape as the
  v1.6.12 size-cap slider. Wired to a static
  `ThreatIntelFeed.cachedStats(at:)` that reads the daemon-written
  feed cache.
- **`alertClusterService` + `mcpBaselineService`** were instantiated
  on `DaemonState` but never invoked. `ClusterSheet` already creates
  its own per-render copy; `MCPBaselineService` doesn't yet have its
  producer half wired (the "MCP-server-name attribution from process
  ancestry" piece). Removed both from `DaemonState` rather than ship
  fictional services. `MCPBaselineService` will return when its
  producer lands.

### Bundle B — Cache eviction perf

- **`LLMCache`** eviction was O(n log n) — sorted the entire entries
  dict on every overflow. The header doc claimed O(1) doubly-linked
  list. Rewrote to an O(n) min-scan over the existing `accessSeq`
  monotonic counter. Header now matches reality.
- **`RuleEngine` + `SequenceEngine` regex caches** used the exact
  anti-pattern that commit `de5ed04` introduced `accessSeq` to
  replace: `lastIndex(of:) + remove(at:) + append` was O(n) on every
  cache hit, at the 2048-entry cap and 420 rules under burst this is
  the dominant rule-eval cost. Both engines now use a sequence-number
  sidecar dict — O(1) hit promotion, O(n) min-scan only on overflow.

### Bundle C — Stop double-scanning

- **`IntegrationsView`** re-ran `SecurityToolIntegrations.detectInstalledTools`
  on every view open even though the daemon had already done the
  scan at startup. Daemon now writes
  `/Library/Application Support/MacCrab/integrations_snapshot.json`
  at startup and refreshes hourly inside the existing hourly timer.
  Dashboard reads the snapshot, falls back to a local scan when the
  file is missing (covers brand-new install + dev-without-daemon).
- **`BrowserExtensionsView`** verified to be doing the right thing
  already — sysext can't reliably read `~/Library/Application Support/...`
  paths under root context, so the dashboard's user-side scan is the
  authoritative producer. Left intentionally unchanged.

### Bundle D — Agent activity timeline

The headline visibility feature: substantiates the wiki's
`AgentLineageService` positioning ("MacCrab sees what your agents
actually did") that has been unsubstantiated since the data started
being captured in v1.6.7.

- **`AgentEvent` + `AgentSessionSnapshot`** are now `Codable`. New
  `LineageSnapshot` wrapper. Daemon writes
  `/Library/Application Support/MacCrab/agent_lineage.json` on every
  30 s heartbeat tick (piggy-backs the existing timer — no extra
  clock drift).
- **`AIActivityTimelineView`** (new SwiftUI view) renders the
  per-AI-tool timeline inside the existing AI Guard tab:
  - Session picker chips for each active AI tool with event counts +
    alert badges
  - Per-session card with kind-counted summary (LLM / spawn / read /
    write / net / alerts) and chronological event rows
  - Newest-first ordering, monospaced HH:mm:ss timestamps,
    kind-specific SF Symbols and colors
  - Inline cap of 80 events with "Show all N" disclosure for
    long-running sessions
  - Empty-state copy listing the 8 supported tools so the feature
    is discoverable on a quiet machine

### Stability hardening

A second-pass review specifically for endpoint stability before
shipping:

- **In-flight guard on `AgentLineageService.writeSnapshot`.** The
  heartbeat fires every 30 s and dispatches a `Task { await
  writeSnapshot }`. Without a guard, a slow disk (full /Library,
  VACUUM contention, IO pressure) would queue successive Tasks
  indefinitely — each holding actor refs and the encoded payload —
  until the actor mailbox blew up. Now drops the snapshot if the
  previous write hasn't returned.
- **Atomic-swap race fixed.** Both new `writeSnapshot` paths used
  `removeItem(path) → moveItem(tmp, path)` which left a brief
  destination-missing window. Switched to `moveItem` first with
  `removeItem + retry` fallback only when the rename refuses,
  matching the existing heartbeat-write pattern.
- **mtime gate on `AppState.refreshAgentLineage`.** Dashboard polls
  every 10 s; daemon writes every 30 s. Without the gate, 2 of every
  3 polls re-decoded an unchanged file. Now `stat()` first, skip the
  decode unless mtime advanced. Cuts steady-state work by 2/3.
- **Orphan `.tmp` cleanup.** Both writers `removeItem` the tmp file
  on encoder error.

### Tests

**799 tests pass (up from 783)** — 16 new across four suites and one
stability-invariant test:

- `LLMServiceFactoryTests` (3) — disabled / Claude-without-key /
  OpenAI-without-key all return nil.
- `ThreatIntelFeedCachedStatsTests` (3) — missing file, present
  file, garbage JSON.
- `SecurityToolIntegrationsSnapshotTests` (4) — missing,
  round-trip, garbage, live-scan write.
- `AgentLineageSnapshotTests` (4) — missing, garbage, full
  round-trip across all 6 event kinds (with optional bytes on
  `llmCall` and associated severity on `alert`), `kindCounts`
  accuracy.
- `LLMCacheTests.evictsBulkOverflow` (1) — bulk-overflow path
  exercised end-to-end.
- `AgentLineageSnapshotTests.concurrentWritesDropSecond` (1) —
  stability invariant: a second concurrent `writeSnapshot` must
  drop, not pile up.

Existing 9 `AgentLineageServiceTests` and 11 `AlertClusterServiceTests`
continue to pass — `AgentSessionSnapshot.lastActivity` was added with
a default that derives from events so every existing call site
compiles unchanged.

### Upgrading

Install v1.6.15. Existing alerts, suppression state, rules, and
configuration are preserved unchanged. No data migration. The Agent
Activity Timeline view appears under AI Guard the next time you open
the dashboard — populates within 30 s once an AI tool is running.
The "Get AI Triage Recommendation" button appears under each alert
in the detail panel when an LLM backend is configured (Settings → AI
Backend).

### Meta-lesson

Three of the four Bundle A items (TriageService, threatIntelStats,
alertClusterService) had the same shape as the v1.6.9 NoiseFilter
bug, the v1.6.12 size-cap bug, and the v1.6.14 decoder bug: a
service or setting was *visible* to the user but *no daemon code
consumed it*. Each shipped with passing tests for the component
itself but no test asserting "given X surfaced, daemon does Y." The
wiki article [[maxdatabasesizemb-wired-bug]] has been updated; the
discipline going forward is to write a "given setting X, daemon does
Y" test for every new published-state field, not just the underlying
component.
