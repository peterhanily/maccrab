## MacCrab v1.4.6 — field-FP hotfix for v1.4.5

Shipped same-day after the v1.4.5 upgrade surfaced six remaining
false positives on a real MacBook Pro. None of these add detection;
each closes one specific FP class.

### Fixed

- **Credential-exfil sequence on `networkserviceproxy`.** The
  `credential_theft_exfil` sequence rule (keychain / cookie / SSH
  key read → outbound network from same PID) was firing Critical
  on Apple's `/usr/libexec/networkserviceproxy` — an Apple-signed
  system daemon whose job legitimately spans both events. Added
  step-level `filter_apple` (SignerType=apple) and `filter_system_path`
  (`/System/`, `/usr/libexec/`, `/usr/sbin/`, `/Library/Apple/`)
  to the `cred_read` step. PID-same correlation means the filter
  applies to the whole chain — Apple daemons can't trigger the
  rule regardless of what the network connection looks like.

- **Cross-process correlation on `/private/var/log/wifi.log.0`.**
  `newsyslog`'s rotation workflow emits
  `setmode → setowner → close_modified → unlink` across
  `airportd` / `bzip2` / `newsyslog` on rotated log files. The
  correlator converged on this as "3 processes, 4 events" at
  Medium severity. Two fixes:
    - New `hasRotatedLogSuffix` helper treats `*.log.N`,
      `*.log.N.gz`, and `*.log.N.bz2` as ignored paths.
    - `/private/var/log/` and `/var/log/` added to
      `ignoredPathSubstrings`. `.log.bz2` added to the suffix
      list alongside `.log.gz`.

- **Apple built-in USB churn.** Apple vendor id `0x05AC` covers
  the built-in keyboard, trackpad, camera, touchbar, and T2
  chip. Every sleep/wake cycle produces connect/disconnect
  events for these devices. The v1.4.5 rate limiter would still
  surface the first event of each tuple per 24h, which on a
  reboot-heavy day gave the user four informational alerts with
  no useful signal. Apple VID is now skipped entirely for
  non-mass-storage events. Mass-storage still surfaces
  regardless of vendor — exfil class.

- **`csrutil_status_check` fires on user-shell csrutil.** The
  rule's `filter_apple` only excluded parents under `/System/`.
  A user running `csrutil status` in Terminal has a shell
  (zsh/bash) as the ES parent — below the `/System/` filter —
  so the rule fired. Added `filter_terminal` covering bash /
  zsh / sh / fish / dash plus the common terminal bundles.

- **`system_enumeration_burst` fires on `whoami` / `uname` in
  Terminal.** Same ancestor-chain issue as csrutil — the filter
  required both `SignerType: apple` AND `ParentImage|startswith /System/`,
  but shell-launched discovery commands don't have a `/System/`
  parent. Added parallel `filter_terminal` step.

- **`process_listing_by_unsigned` shell-parent gap.** The
  existing filter listed Terminal.app, iTerm2, Alacritty, etc.
  but missed `/bin/zsh`, `/bin/bash` — when a user runs `ps`
  from a zsh shell, the ES parent is `zsh`, not Terminal.
  Shell basenames added to the filter list.

---

No schema changes. 600 tests pass.
