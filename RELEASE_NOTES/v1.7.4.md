## MacCrab v1.7.4 — Heartbeat-silent regression fix

Follow-up hotfix to v1.7.3. The v1.7.3 memory fix combined two
changes that, together, produced a new failure mode: when any
single snapshot writer blocked, the heartbeat thread held its
overlap-guard lock indefinitely, every subsequent 30 s tick was
dropped, and the dashboard showed "Detection engine appears silent"
after 120 s. User-reported on a v1.7.3 host.

### Root cause

The v1.7.3 design layered an outer `HeartbeatInFlight` guard around
a body that serially `await`-ed four snapshot writers. Three of the
four writers (MCPBaselineService, RuleEngine, TCCMonitor) lacked
their own per-instance in-flight guards (only AgentLineageService
had one — added in v1.6.6). When fire-and-forget Tasks pre-v1.7.3
queued on those guard-less writers under load, captures piled up
(the v1.7.0–v1.7.2 leak). v1.7.3 cured the queue by serialising at
the heartbeat level, but introduced the deadlock-shaped failure:
any blocked writer = blocked heartbeat = silent dashboard.

### Right fix

Per-resource guards, not per-caller guards. Each snapshot writer
gets its own `snapshotWriteInFlight` bool matching the
v1.6.6 AgentLineageService pattern. Concurrent writeSnapshot calls
no-op gracefully — the queue can't form because the second call
returns early. Heartbeat goes back to fire-and-forget Tasks, the
critical-path heartbeat write is no longer gated on snapshot
work, and the dashboard never sees a stale heartbeat unless the
sysext itself is dead.

### Changes

- **`MCPBaselineService.writeSnapshot`** — added
  `private var snapshotWriteInFlight = false` + the guard pattern.
- **`RuleEngine.writeTelemetrySnapshot`** — same.
- **`TCCMonitor.writeSnapshot`** — same.
- **`DaemonTimers.swift`** — removed `HeartbeatInFlight` class and
  the outer overlap guard. Restored the four
  `Task { await ... .writeSnapshot(...) }` calls to fire-and-
  forget. Heartbeat write returns to the critical fast path.

### Tests

926 in 188 suites pass (was 922). +4 net (`V174GuardTests.swift`):
each writer round-trips a real snapshot file; 50 concurrent
`writeSnapshot` calls don't crash and at least one succeeds (the
file exists at the end).

### Notes for operators

- Existing v1.7.3 dashboards will start receiving fresh heartbeats
  immediately after upgrade. The "Detection engine appears silent"
  banner will clear within 30 s of the first new heartbeat tick.
- v1.7.3's memory cap is preserved — no return to the v1.7.2
  2 GB regression. The leak was at the actor-queue level
  (per-writer); v1.7.4 just moves the guard to the right scope.
- If the silent-heartbeat banner reappears in the future, suspect
  a NEW snapshot writer being added without a per-instance guard.
  Pass 8 catches uncapped collections; we may need a parallel
  Pass 9 for snapshot writers in a future release.

### Design lesson captured in wiki

Don't add overlap guards at multiple layers. v1.7.3 added a layer
at the heartbeat scope on top of the existing per-writer guard at
AgentLineageService. If even one inner writer lacks its own guard
and blocks, the outer guard becomes a deadlock. Per-resource
guards are the right level — each writer protects its own work,
each caller doesn't need to know about the protection.
