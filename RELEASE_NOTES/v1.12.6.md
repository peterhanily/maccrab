# MacCrab v1.12.6 — Internal-quality refinement + 2026 supply-chain rule wave

v1.12.6 is a refinement release on top of v1.12.5. Seven internal
waves harden storage discipline, attribution schema, AI-tool guard
coverage, and the audit pipeline, plus 12 new detection rules for
the 2026 supply-chain incident wave. No user-facing behavior changes
from v1.12.5; this release also lights up 21 previously-silent Sigma
rules that referenced enrichment fields with no resolver case.

If you're on **v1.12.5**, Sparkle in-place upgrade works directly:
open MacCrab.app, choose **Check for Updates…**, click **Install
Update**, relaunch when prompted.

## What's new

### Storage discipline (Wave 1)

- **`events.db` cap-timer cadence is now user-configurable.** New
  `eventsSizeCapIntervalMinutes` field (default 60 min). Pre-v1.12.6
  the timer fired every 6 hours, so on heavy hosts the file gained
  ~17 GB between sweeps. A non-configurable 60s watchdog also fires
  early when the DB exceeds 1.5× cap. Configure in
  `daemon_config.json` or `user_overrides.json`:
  ```json
  { "storage": { "eventsSizeCapIntervalMinutes": 30 } }
  ```
- **`SelfDefense.directoryHash()` migrated to in-process CryptoKit.**
  Removes ~1,704 `/usr/bin/shasum` subprocess spawns/minute on every
  host (shasum is a perl script). Combined-hash output is byte-equal
  to the previous path — no false `.rulesModified` alert on first
  upgrade.
- **EventStore enforces 64 KB cap on per-event raw_json** with
  per-arg → commandLine → final-fallback truncation. Truncated
  events get `payload.truncated="true"` and `payload.original_bytes=N`
  enrichments.

### Attribution schema (Wave 2)

30 new indexed columns across events.db / alerts.db / campaigns.db,
additive only. Old rows have NULL in new columns; rule matching is
backward-compatible (engine works against the in-memory Event struct
rehydrated from raw_json).

- **events.db v5 → v6**: user_id, user_name, group_id,
  working_directory, responsible_pid, architecture,
  is_platform_binary, is_notarized, process_sha256, parent_name,
  parent_executable, parent_signer_type, ai_tool, ai_tool_child,
  session_launch_source, tcc_decision. RuleEngine gains 16 new Sigma
  aliases. **6 dead Sigma rules unblocked** (rosetta detection × 3,
  notarization detection × 3).
- **alerts.db v4 → v5**: user_id, user_name, working_directory,
  ai_tool, parent_executable, process_sha256, host_name.
- **campaigns.db v1 → v2**: affected_users, affected_executables,
  first_seen, last_seen, process_tree_depth, techniques, ai_tools.

### LLM intent tie-breaker wired (Wave 3)

`state.intentClassifier` (LLM-aware) was constructed in DaemonState
but never called in production. Now fires as a tie-breaker for
AI-triggered installs when heuristic confidence < 0.7. Runs in
`Task.detached(.utility)` so the hot path never blocks. Per-tree-key
cooldown via `IntentRefinementCache` (LRU 256 entries, TTL 10 min)
prevents burst fan-out.

- `events.db`: v5 → v6 (16 columns added, additive)
- `alerts.db`: v4 → v5 (7 columns added, additive)

All migrations use `ALTER TABLE ADD COLUMN` with the v1.6.12

matching falls through to raw_json on legacy rows automatically.

## Performance

Insert throughput on events.db is ~10% slower with the 16-column
schema; per-event storage is ~43% larger (1791 B → 2563 B). Live
load (47 ev/s on a dev host) is ~5% of insert capacity — no
headroom risk. SelfDefense.directoryHash latency drops ~220× (6 s
→ 27 ms per call) — substantial steady-state CPU saving that more
than offsets the schema cost.

internal hosts, no signing material, no leaked PII in the public
tree or git history.

## RC1 → RC2 fix wave (Wave 9)

RC1 (`463272560b4c…`) was field-tested by the maintainer and surfaced
three compounding bugs that together broke the entire storage-discipline
story Wave 1 was meant to land. RC2 fixes the root causes — none of
this is cosmetic, the file actually wouldn't shrink.

- **9A — Dashboard-side stores held the DB open RW, blocking the
  daemon's VACUUM.** `V2LiveDataProvider` was opening EventStore,
  AlertStore, CampaignStore, and SQLiteCausalGraphStore in
  read/write mode. The daemon's "Reduce events.db now" handler
  ran `VACUUM` against the same file — SQLite needs an exclusive
  lock, the dashboard's RW connection refused to drop, the vacuum
  silently no-op'd and the file kept growing. Added
  `forceReadOnly: true` plumbing through all four stores;
  `V2LiveDataProvider` and `AppState` (which also opens stores from
  the dashboard side) both now pass it.
- **9B — Size-cap enforcer had no fallback when full VACUUM
  couldn't run.** A full `VACUUM` needs ~1.3× the DB-size in
  free disk. On a host where events.db has grown past the cap,
  the user is almost by definition disk-pressured. The size-cap
  timer now tries `PRAGMA incremental_vacuum` first (releases
  freelist pages, needs ~0× extra disk) and only falls back to
  full VACUUM if mode allows it.
- **9B.1 — PRAGMA order bug meant `incremental_vacuum` was a no-op
  on every DB ever shipped.** `PRAGMA auto_vacuum = INCREMENTAL`
  must be applied BEFORE `PRAGMA journal_mode = WAL`. SQLite
  silently refuses to flip auto_vacuum once the WAL setup has
  dirtied the DB header — no error, just stays in mode 0. Every
  store across every release has been running in mode 0, which
  makes Wave 9B's incremental_vacuum a no-op. Field-confirmed via
  runtime PRAGMA inspection on the RC1 user machine: tracegraph.db
  at 11 GB in mode 0. RC2 reorders the PRAGMA sequence across
  `StoragePragmas.swift`, `CampaignStore.swift`,
  `TraceStore.swift`, and `SQLiteCausalGraphStore.swift`. New DBs
  ship in mode 2 (INCREMENTAL); pre-existing mode-0 DBs need a
  one-shot manual `VACUUM` (or rm + recreate) to convert — the
  runtime short-circuits cleanly on mode 0 and logs a clear notice.
- **9C — `ai_tool` column landed NULL despite raw_json being
  populated.** Wave 2A's EventStore writer was reading the
  `agent_tool` enrichment key, but the production writer uses
  `ai_tool`. RC2 reads `ai_tool` first and falls back to
  `agent_tool` so both code paths populate the column.
- **9D — Silent insert failures.** During the disk-full incident
  the dashboard showed "0 ev/s" because every insert into events.db
  was failing — but the failure path logged nothing. RC2 adds a
  3-tier escalation ladder (`StorageErrorTracker`): first failure
  logs, then logarithmic ladder (10, 100, 1000, …), plus heartbeat
  fields `event_insert_errors_total / _rate / _last_kind` exposed
  in `maccrabctl status` so operators can see the gauge.
- **9E — Threat intel feeds didn't auto-load on dashboard mount.**
  `V2IntelligenceWorkspace`'s `.task(id:)` body deferred to the
  data provider, which on first fire returned the mock fixture
  data because `V2MockDataProvider` always wins the initial race.
  RC2's new `V2LiveDataProvider.loadFeedsFromCache(preferring:)`
  static helper probes a prioritized list of candidate cache dirs
  directly, sidestepping the mock-provider race.
- **9F — Bundled IOCs weren't persisted to disk on cold start.**
  Field-confirmed second-order gap in 9E: the dashboard read the
  on-disk cache correctly, but on a fresh install that file
  didn't exist until the daemon's first `updateAllFeeds()`
  completed (~14 min across all three abuse.ch feeds on the RC2
  user machine). `BundledThreatIntel.loadInto` adds IOCs to the
  in-memory actor state via `addCustomIOCs` but never wrote the
  cache file. RC2 (post-9F): `ThreatIntelFeed.start()` is now
  async-await — it awaits `loadCachedFeeds()` and
  `loadCustomIOCFiles()` inline before returning, with the
  periodic network refresh split into a background Task. New
  public `persistCacheNow()` flushes the in-memory IOC set to
  disk; `DaemonSetup` calls it immediately after
  `BundledThreatIntel.loadInto(...)`. Cold-boot result: the
  dashboard's Intelligence tab sees bundled IOCs the moment it
  opens. Warm-boot result: prior network IOCs are preserved
  because `start()`'s inline hydration runs before the bundled
  load, so `persistCacheNow()` writes the union (not a
  bundled-only overwrite that would lose feed history).
