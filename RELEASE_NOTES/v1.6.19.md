## MacCrab v1.6.19 — Safety hardening + AlertSink chokepoint + audit codification

The biggest release of the v1.6.x line. Three thrusts.

### 1. Four safety guards on paths that could damage the user's machine

A pre-release safety audit found four code paths where MacCrab could,
under a hallucinated alert or a poisoned threat-intel feed, act on
the wrong target. All four are now hardened.

- **`SafePIDValidator`** — refuses to kill PID ≤ 1 (would panic the
  kernel), MacCrab itself, the critical-system-process list
  (`launchd`, `kernel_task`, `WindowServer`, `loginwindow`, `securityd`,
  `opendirectoryd`, `coreaudiod`, `bluetoothd`, `mDNSResponder`, etc.),
  or anything running from `/System/`, `/usr/libexec/`, `/sbin/`,
  `/usr/sbin/`. Wired into `ResponseEngine.killProcess` and
  `SupplyChainGate.gate`. Even a malformed rule that names PID 1
  is now a no-op with a logged refusal.

- **`SupplyChainGate` ancestry + recycled-PID protection** — the
  gate that kills installers of suspiciously fresh packages now
  requires the PID to descend from a known package manager (npm,
  pnpm, yarn, pip, brew, cargo, mas, ...) within 5 hops, AND
  re-verifies the path before the 2-second-delayed SIGKILL so a
  recycled PID can't redirect the kill to a different process.

- **`DNSSinkhole` protected-domain allowlist** — refuses to sinkhole
  any domain matching `apple.com`, `*.icloud.com`, `ocsp.apple.com`,
  `github.com`, `googleapis.com`, `microsoft.com`, `aws.amazon.com`,
  `stripe.com`, `jetbrains.com`, `slack.com`, `digicert.com`,
  `sectigo.com`, `letsencrypt.org`, `maccrab.com` itself, etc.
  (75+ patterns) plus IP literals. A poisoned threat-intel feed
  entry that named `ocsp.apple.com` would previously have bricked
  code-signing system-wide; now it gets logged and dropped.

- **`PanicButton.activate(disableBluetoothInPanic: Bool = false)`** —
  the BT-disable step is now opt-in. Pre-v1.6.19 panic mode would
  power off Bluetooth as part of containment, which would strand a
  user with a Magic Keyboard or Trackpad. Default is now off.

### 2. `AlertSink` — single chokepoint for all alert emissions

The v1.6.9 NoiseFilter-layering bug class is now closed
architecturally, not by a string filter.

Pre-v1.6.19, 39 different code paths called `alertStore.insert`
directly. AI-Guard, threat-intel, supply-chain, monitor tasks, and
self-defense all bypassed the `AlertDeduplicator` entirely. A FP
flood from any of those 39 paths couldn't be stopped by tuning
NoiseFilter, because NoiseFilter only ran on rule-engine matches.

Post-v1.6.19, `AlertSink` is the single point through which every
alert reaches storage. It applies dedup before insert. The
architectural invariant ("zero direct `alertStore.insert` outside
AlertSink") is enforced at release time by the new audit script.

A TOCTOU race in the original AlertDeduplicator (between
`shouldSuppress` and `recordAlert` as separate actor hops) was
caught and fixed during the pre-push review: 50 concurrent submits
with the same key now insert exactly one (test pins the contract).

### 3. Audit codification — `scripts/pre-release-audit.sh`

Three architectural invariants are now enforced at release time as
hard fails:

- **Orphan audit** — every Settings `@AppStorage` that affects daemon
  behavior must declare a sync function. Catches the
  wire-the-orphans pattern (the bug class that produced four
  releases of structurally similar bugs: NoiseFilter-layering v1.6.9,
  `maxDatabaseSizeMB` v1.6.12, the 5-orphan bundle v1.6.15, the
  self-DoS rule v1.6.18).
- **Single-sink** — zero direct `alertStore.insert` outside AlertSink
  plus the two audited `DaemonSetup` exceptions.
- **Duplicate-source** — cross-file constants must agree.

Wired into `release.sh` as Step 0b. Fails the release on regression.

### Also in this release

- **Webhook config wiring (Settings → daemon).** Pre-v1.6.19 the
  Slack/Teams/Discord/PagerDuty TextFields wrote to UserDefaults but
  no daemon code consumed them. Now `SettingsView.syncWebhookConfig`
  writes `~/Library/Application Support/MacCrab/notifications.json`
  with a 500 ms debounce, SIGHUP triggers
  `NotificationIntegrations.reloadConfig()`, and configured webhooks
  actually fire. Existing operators with pre-v1.6.19 webhook URLs
  in UserDefaults: re-type the URL once (or copy-paste) to trigger
  the sync, or restart the app and any change will activate it.
- **`autoQuarantine` / `autoKill` / `autoBlock` Settings toggles
  removed.** Same wire-the-orphans shape — toggled UserDefaults but
  daemon never consumed them. Per-rule auto-actions live in the
  Response Actions tab, which IS wired into `ResponseEngine`.
- **`prerelease-check.sh` extended** with cask-version equality,
  Sparkle appcast-URL equality, Sparkle EdDSA public-key equality,
  app + sysext bundle-ID equality, Apple Team-ID consistency. Five
  new drift checks on top of the v1.6.18 short-version catch.
- **Webhook URL log privacy.** Pre-v1.6.19 a failed webhook POST
  would log the full URL (which contains the secret token) at
  warning level into Unified Logging. Now `privacy: .private`.
- **`PanicButton` removed from `DaemonState`** — was instantiated
  but `activate()` had zero callers (no UI surface). Type itself
  remains in `MacCrabCore` for re-introduction when a Panic surface
  ships.

### Tests

860 in 177 suites, all green (was 807). +53 net.

### Upgrading

Upgrade in place via Sparkle or `brew upgrade --cask maccrab`. No
config-file migration needed. Existing operators who configured
webhooks pre-v1.6.19 should re-type any one webhook field to
activate the new sync (one-time, then onChange handles it). Existing
auto-* Settings toggles are retired — use the Response Actions tab
for per-rule auto-quarantine/auto-kill/auto-block.
