# MacCrab v1.19.0

A detection-quality and trust release. Far fewer false alerts, a plugin catalog that verifies what it installs end to end, and storage that stays within its limits.

## Detection quality

- **Trusted-signer noise no longer floods as "critical."** A critical-severity rule firing on an Apple platform binary or a notarized Developer-ID app is now subject to the same trust filtering as everything else, instead of bypassing it. Genuine must-fire detections — ransomware, SIP/Gatekeeper/AMFI tampering, revoked certificates, command-and-control, credential theft — still always fire.
- **Attack-campaign correlation stops flagging routine development.** Both correlators — kill-chain and coordinated-attack — now apply the same severity floor and known-benign-process filter, so build helpers, test runners, AI coding tools, and local dev runtimes (e.g. JS/worker runtimes) that trip a couple of low- or medium-severity tactic rules no longer accumulate into a false critical "multi-stage attack." Genuine medium-and-above, multi-rule activity still correlates — including against trusted or agent tooling — so a real compromise is still caught.
- **Apple's own security tools are attributed correctly.** macOS background remediators (XProtect) are no longer mislabeled as AI-coding-tool activity, so they no longer produce spurious "AI tool accessed credentials" alerts.

## Plugin catalog trust

The rave plugin catalog now verifies trust end to end before anything installs:

- **Publisher pinning** — a plugin's signing key is bound to the signed catalog entry; a mismatch refuses the install.
- **Signed revocation list** — revoked plugins are refused, and an already-installed plugin that becomes revoked is quarantined (never silently deleted or run).
- **Version floors** — a plugin that requires a newer MacCrab than you're running won't install.
- **Anti-rollback** — a stale, replayed catalog or revocation list is rejected.
- **Signed install receipts** — every install records an offline-verifiable receipt of exactly what was installed and the trust state at install time.
- **Forensic case integrity** — collected evidence now carries a signed, append-only chain of custody verifiable offline.

- **Plugin provenance at a glance** — the dashboard, CLI, and agent surfaces label every plugin as **built-in** (first-party, shipped in the app), **third-party** (an operator-trusted publisher key), or **store** (installed from the verified catalog, with a signed install receipt), so it's always clear where a scanner came from.

The in-app catalog browser is live behind this verified install path, showing each plugin's trust state and provenance. Installs of first-party signed plugins require operator-signed binaries.

## Reliability & polish

- Trace-graph and trace databases now stay within their configured size caps (previously the trace-graph file could grow past its limit).
- The events database size cap is now honest about everything the file holds (events plus the full-text search index and captured evidence), so it stays within its configured limit instead of overshooting.
- The built-in rule severity control now shows the active override instead of always reading "Default."
- Dashboard tables no longer flash empty for a frame when opened.

## Build & supply chain

- Continuous integration with SHA-pinned actions and SLSA build provenance; the release pipeline is decomposed into reproducible stages.
- Key-rotation runbooks for the update-signing and catalog-signing keys.
