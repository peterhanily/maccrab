# v1.15.0-rc.2 — v1.13b back-port: Dashboard Forensics tab + ConsentManager

**Status: internal release candidate.** Not published to Sparkle appcast or Homebrew cask. Source tag only.

Back-port RC closing the v1.13b operator-surface items that the v1.13.0-rc.6 ship deferred (per plan §7 v1.13b kill criterion). Version stays on the v1.15 line because SemVer ordering doesn't allow going backwards; release notes capture the sub-slice provenance.

## What landed

**Dashboard Forensics tab — four sub-tabs under Investigation workspace.**

- **Forensics · Cases** — read-only list of every case from `CaseManager.listCases()`. Click a case → detail panel with name / created / encryption state / AI grant / scheduled-trusted flag / artifact totals + breakdown by content type. Encrypted-case detail load prompts Touch ID via `KeychainDEKVault` (the v1.13b-1 Touch ID upgrade from rc.6 carries through).
- **Forensics · Plugins** — read-only registry of every registered plugin. Click → manifest detail (TCC requirements, outputs with privacy-class chips, MCP tools).
- **Forensics · Artifacts** — sortable artifact table per selected case. Sort by observed / content type / summary. Filter by content type. Privacy class colored per row.
- **Forensics · Findings** — surfaces every `posture.*` artifact (the v1.15 PostureAnalyzer's output). Severity-colored chips: critical / high / medium / low / informational.

Case creation continues via the CLI per plan §3.7. The dashboard is strictly read-only at v1.13b — no edit affordance, no run buttons. The Run-this-plugin button lands when the operator-action audit log + per-run consent UX (below) hardens.

**ConsentManager policy layer.** Plan §10.5 + §10.8 codified into a single-method protocol. Implementations decide how to gather the verdict; the runtime only sees `.granted` / `.autoApproved(reason:)` / `.denied(reason:)`.

`LoggingConsentManager` (production default) applies the policy rules without blocking on a UI prompt:

| Mode | Condition | Verdict |
|---|---|---|
| interactive | always | `.granted` (operator initiated) |
| scheduled | `scheduled_trusted=1` | `.autoApproved` |
| scheduled | `scheduled_trusted=0` | `.denied` with pointer to `maccrabctl case mark-trusted-scheduled` |
| mcpFromAgent | metadata-class plugin | `.granted` |
| mcpFromAgent | non-metadata + `ai_content_allowed=1` | `.granted` |
| mcpFromAgent | non-metadata + `ai_content_allowed=0` | `.denied` — § 10.8 structured-error path with `maccrabctl case allow-ai --content` command in reason |

`AlwaysAcceptConsentManager` is the test path; unit tests inject it so consent flows don't block.

## What's deferred to a follow-up sub-slice

- **UNUserNotificationConsentManager** — production-mode visible toast backed by `UNUserNotificationCenter`. Needs the host process to be a registered `.app` with notification entitlements (MacCrabApp or the future `maccrab-forensicsd.app`; `maccrabctl` CLI can't post visible toasts). The protocol surface is wired in this RC — the UN upgrade is purely additive.

- **PluginRunner integration of ConsentManager** — the runner currently invokes plugins directly. Threading ConsentManager into runCollector / runEnricher / runAnalyzer is a one-line surface change + per-mode decision call; deferring it lets this RC ship the policy layer as a verifiable unit before the runner change adds the call sites.
