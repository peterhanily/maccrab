## MacCrab v1.6.12 ‚Äî The DB size cap now actually caps the DB size

Field report: Settings showed `Max database size: 500 MB`,
`Current size: 18.95 GB`. 38√ó over the cap. No pruning was
happening.

### Root cause

The `maxDatabaseSizeMB` field existed in `DaemonConfig` and the
Settings UI surfaced it with a "when the database exceeds this size,
the oldest events will be pruned automatically" caption ‚Äî but
**nothing in the daemon ever read that value**. The retention timer
only enforced time-based pruning (`retentionDays`, default 30). On a
high-event-rate machine 30 days of events easily exceeds any
reasonable size cap.

### Fix

1. **`EventStore.pruneOldest(count:)`** ‚Äî new API that deletes the N
   oldest events (by timestamp) and their FTS index entries, in 100K
   batches with `Task.yield()` between batches so concurrent inserts
   aren't starved.

2. **`EventStore.vacuum()`** ‚Äî runs `VACUUM` so pages marked free by
   the pruning are actually reclaimed into the on-disk file size.
   Without this, DELETE just leaves the file the same size with
   internal free space; the enforcement would be moot.

3. **New hourly `sizeCapTimer` in `DaemonTimers`** ‚Äî checks the
   `events.db` file size every hour. If it exceeds
   `maxDatabaseSizeMB`, iteratively prunes the oldest events + runs
   VACUUM until the file drops below 80% of the cap (the headroom
   prevents the next insert from immediately re-triggering).

4. **`DaemonState.maxDatabaseSizeMB`** plumbed through from
   `DaemonConfig`. Clamped at a 50 MB minimum so a misconfigured
   tiny value can't make the enforcer delete everything on first
   run.

### What operators will see

On a machine currently way over the cap, within the first hour
after upgrade the daemon will log:

```
DB size 19406 MB exceeds cap 500 MB. Pruning to 400 MB.
Size-cap enforcer: 19406 MB ‚Üí 398 MB (cap 500 MB)
```

‚Ä¶and the `events.db` file will shrink accordingly. Alerts, rules,
and every other piece of state are unaffected ‚Äî only the event-
history DB is pruned.

### Tests

**769 tests pass (up from 764).** +5 new tests cover:

- `pruneOldest` deletes by oldest-timestamp ordering
- `pruneOldest` with count > total returns actual-deleted (doesn't
  crash on over-request)
- `pruneOldest` with count=0 is a no-op
- `vacuum()` runs without error and doesn't corrupt the DB
- prune + vacuum together shrinks cumulative disk usage (db +
  wal + shm)

### Note on first run after upgrade

If you're currently over cap, the first size-cap sweep happens 10
minutes after daemon start, then hourly. VACUUM is costly on a
large DB ‚Äî expect a brief IO spike the first time the enforcer
runs against a multi-GB file. Subsequent runs are cheap (they
only fire when the cap is already exceeded).
