## MacCrab v1.4.3 — Fail loud, not silent

Wave A of the ongoing 1.4.x quality pass. Every change in this release
addresses a *silent failure of protection* class: states where MacCrab
appeared to be working but wasn't. Before v1.4.3 the audit found five
such paths where the dashboard looked normal while detection was
effectively off.

### Fixed

- **Zero detection rules → visible banner.** When the detection engine
  loads zero rules (rules directory missing, every JSON malformed,
  bundle tampered), the Overview now shows a red banner and the
  statusbar crab switches to a warning variant. Previously this was
  only logged at `.warning` level — silent to anyone who wasn't
  running `sudo log show`.
- **Sysext watchdog.** The dashboard polls the heartbeat every 10s;
  if it goes stale for more than 120s the app automatically resubmits
  an `OSSystemExtensionRequest.activationRequest`. Sysextd dedups so
  an already-healthy extension is a no-op; a crashed or hung one gets
  respawned without user intervention. Respects a 5-minute cooldown
  so a persistently-failing install doesn't hammer sysextd.
- **Storage failures surface in the UI.** `StorageErrorTracker` now
  persists a JSON snapshot after every failed alert or event insert.
  The Overview raises a red banner with the most recent error message
  whenever any storage write has failed in the last 10 minutes
  (disk full, DB locked, permissions denied). Before v1.4.3 these
  failures were swallowed via `StorageErrorTracker.shared` with a
  `logger.error` — real but invisible.
- **Detection engine heartbeat.** The sysext now writes a heartbeat
  JSON file to `/Library/Application Support/MacCrab/heartbeat.json`
  every 30 seconds. The dashboard polls it and raises a banner when
  the last heartbeat is older than 120s. Detects the
  "sysext silently replaced by a no-op" attacker scenario that was
  explicitly called out in the v1.4.2 audit as unprotected.
- **Rule-manifest integrity verification.** Every release now ships
  a signed `manifest.json` listing SHA-256 hashes of every compiled
  rule JSON file. `RuleBundleInstaller` verifies the bundled manifest
  before propagating rules to the system directory — if the bundle
  itself is tampered, we refuse to sync. It also verifies the
  installed tree on every launch; a mismatch (attacker modified
  `/Library/Application Support/MacCrab/compiled_rules/` post-sync)
  triggers auto-resync from the bundled copy plus a dashboard banner
  so the user knows tampering was detected and remediated.

### How to verify it's working

- **Simulate zero-rules:** `sudo rm -rf "/Library/Application Support/MacCrab/compiled_rules/*"` then relaunch MacCrab. Overview should show the red banner; statusbar crab should be `⚠️🦀`.
- **Simulate stale heartbeat:** `sudo pkill -STOP com.maccrab.agent` for 2 minutes. Dashboard should show the "Detection engine appears silent" banner within 30s; watchdog kicks in within the next poll cycle and reactivates. `sudo pkill -CONT com.maccrab.agent` to release.
- **Simulate storage failure:** set `compiled_rules/` mode to `0000`; insert activity that would normally store; banner appears within 10s.
- **Simulate rule tamper:** `sudo sh -c 'echo "x" >> /Library/Application Support/MacCrab/compiled_rules/execution/ps_listing.json'` — next launch surfaces the tamper banner and auto-resyncs from the bundled copy.

Backwards-compatible. No schema changes. Three new snapshot files
(`heartbeat.json`, `storage_errors.json`, `rule_tamper.json`) land in
`/Library/Application Support/MacCrab/` alongside the DB.
