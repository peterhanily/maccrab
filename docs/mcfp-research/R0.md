# MCFP R0 — dyld + SIP + library-validation paradox

**Status:** decided 2026-05-20. Tags first MCFP shipping artifact: v1.14.0-rc.1.

## The paradox

`DYLD_PRINT_LIBRARIES` and library injection both require disabling library validation. Plan §6.4 enumerates five options for resolving it without compromising the rest of the security model:

- **(a)** `_dyld_register_func_for_add_image` via launcher wrapper. Works for binaries we launch as children; useless for already-running processes.
- **(b)** Endpoint Security `NOTIFY_MMAP` events post-exec. Less granular than dyld telemetry; legal under SIP.
- **(c)** DTrace `pid` provider. Limited under SIP; unsigned dev binaries only.
- **(d)** Drop the dyld component entirely. Rely on static + (later) runtime ES-shape only.
- **(e)** Cooperative fingerprinting via `DYLD_INSERT_LIBRARIES` on operator-initiated launches. Constrained to operator-controlled binaries.

## Decision for v1.14

**Option (d) — drop the dyld component for v1.14.**

Rationale:

- The static components (`arch`, `lc`, `cs`, `ent`) already produce a stable per-binary identifier without touching the dyld layer. R1's MCFP corpus shows static-only stability ≥95% across reboots on the developer's daily-driver (the v1.14 ship corpus job).
- Adding ES `NOTIFY_MMAP` (option b) is the cleanest dynamic complement — it's legal under SIP and works for already-running processes — but landing it brings sysext changes into v1.14 scope. That couples MCFP to Track 1's detection-engine surface, which the plan explicitly avoids ("Track 2 work that breaks anything in Track 1 is reverted immediately" — plan §14.6).
- The DTrace path (c) doesn't survive SIP for the targets we actually care about.
- The launcher wrapper path (a) is useless for the fleet-scale fingerprinting model — operators need to fingerprint processes that already exist.
- The cooperative DYLD_INSERT path (e) is research/heuristic only per the plan; it survives as a future `maccrabctl fingerprint --cooperative <pid>` subcommand for operator-controlled binaries, gated behind explicit per-invocation opt-in. **Not in v1.14 scope.**

## What ships at v1.14.0-rc.1

`mcfp1/static/<arch>/<lc>/<cs>/<ent>` — every component computed without touching the dyld telemetry surface. The runtime + dyld components are reserved in the scheme (a future MCFP component can be added at the path `mcfp1/runtime/...` and `mcfp1/dyld/...` without breaking consumers of the static-only form).

## Re-evaluation triggers

R0 gets re-opened when any of these holds:

- **R1 stability fails on the static components alone.** If the corpus shows <95% static-component stability across reboots on representative binaries, MCFP v1's static-only form is unsound and the dyld component becomes necessary.
- **A specific imposter class evades static fingerprinting.** If R2's imposter experiment (v1.15) finds an imposter with byte-identical static components but distinct behavior, the dyld component is the next-tier defense.
- **The sysext gains a `NOTIFY_MMAP` subscription for another reason.** If Track 1 wires `NOTIFY_MMAP` for some other detection (anti-injection rules, supply-chain analysis), MCFP can ride along for free.

Until one of those fires, option (d) stands.
